# 第 2 章 大模型的核心能力与边界

## 本章导读

移动端开发者学习大模型，不能只停留在“能聊天、能总结、能写代码”的表层能力上。进入工程实践后，真正的问题会变成：为什么同一个问题有时回答不一样？为什么模型会编造不存在的接口？为什么上下文太长会变慢、变贵，甚至变差？为什么流式输出能改善体验，却不能改变模型逐步生成的本质？

本章用工程语言解释大模型的运行机制：文本先被切成 Token，再变成 Embedding；Transformer 通过 Attention 在上下文中建立关联；模型每一步预测下一个 Token 的概率分布；Temperature 和 Top-p 会影响采样；上下文窗口决定模型一次能“看到”的信息范围。

## 学习目标

- 理解大模型从输入文本到输出文本的基本链路。
- 理解 Token、Embedding、上下文窗口、Attention 和概率生成的工程含义。
- 知道 Temperature 和 Top-p 为什么会影响输出稳定性。
- 能把大模型能力映射到移动端业务场景。
- 识别幻觉、知识时效性、不确定输出和权限边界带来的风险。
- 能运行配套实验，观察 temperature 对采样结果的影响。

## 2.1 从移动端功能看模型能力

假设要在 App 中实现“崩溃日志分析助手”。用户上传崩溃堆栈、设备型号、系统版本和 App 版本后，系统返回崩溃摘要、可能原因、排查步骤、相关文档引用和是否建议创建工单。

不用大模型时，这类功能通常依赖规则：匹配异常类型、解析堆栈、维护关键词表、关联文档、生成说明文案。规则方案适合高频、稳定、格式明确的问题；但日志格式复杂、上下文不足或用户描述口语化时，规则会迅速膨胀。

大模型的价值在于处理非结构化输入，理解自然语言、日志和代码片段之间的关系，并生成接近人工分析的说明。但它不能天然知道团队最新规则，不能保证每次输出完全一致，也不能直接决定是否关闭线上告警。

合理设计不是“让模型接管 App”，而是让模型承担语言理解、归纳和候选答案生成，再由服务端、检索系统、权限系统和客户端交互共同约束它。

## 2.2 一次生成请求发生了什么

从工程角度看，一次大模型调用可以简化为 6 步：

1. 服务端把系统指令、用户输入、历史消息和检索资料拼成上下文。
2. 模型服务把文本切成 Token。
3. Token 被转换成向量表示，也就是 Embedding。
4. Transformer 根据上下文计算各位置之间的关系。
5. 模型输出下一个 Token 的概率分布。
6. 采样器根据概率和参数选择一个 Token，再继续预测下一个 Token。

这个过程会重复很多次，直到生成结束符、达到最大输出长度，或被服务端/客户端取消。

这带来两个结论。第一，大模型不是一次性“想好整段答案”再返回，而是一步一步生成 Token。流式输出正是利用这一点，把已生成片段尽早发给移动端。第二，模型输出不是数据库查询结果。数据库查找明确记录；大模型是在上下文和参数共同影响下生成概率上合理的文本。

真实服务里，还要关注请求生命周期：

| 阶段 | 工程含义 | 移动端影响 |
| --- | --- | --- |
| 上下文构造 | 选择系统指令、历史、检索资料和输出格式 | 决定质量、成本和隐私边界 |
| 分词 | 把文本切成 Token | 日志、JSON 和多语言内容可能很耗 Token |
| 预填充 | 模型处理已有上下文 | 影响首 Token 延迟 |
| 解码 | 逐个生成输出 Token | 影响流式输出速度 |
| 缓存 | 复用已有上下文计算 | 影响同轮生成效率 |
| 停止 | 结束符、超时、取消、安全策略或工具等待 | 决定停止按钮是否可控 |

移动端不应只看“总耗时”。更有价值的指标包括首 Token 延迟、生成速度、总耗时、取消响应时间和错误恢复时间。基本优化顺序是：先控制输入，再优化首 Token，再用流式输出改善长回答感知，最后用取消、重试和降级保证可控。

## 2.3 Token 与上下文窗口

Token 是模型处理文本的基本单位。它不等同于中文汉字、英文单词或代码符号；中文句子、英文单词、标点、空格、驼峰命名都可能被切成不同粒度的 Token。

Token 直接影响 3 个指标：

- 成本：大多数模型服务按输入 Token 和输出 Token 计费。
- 延迟：输入越长，预填充越慢；输出越长，生成越久。
- 稳定性：无关上下文越多，模型越可能关注错误片段。

上下文窗口表示模型一次请求最多能处理多少 Token，包含系统指令、用户问题、历史消息、检索资料和输出。它不是长期记忆，而是本次请求的工作区。请求结束后，模型不会自动记住对话；下一轮需要历史信息时，应用必须重新组织必要内容。

移动端会话常见有 3 种设计：

| 方式 | 适合场景 | 风险 |
| --- | --- | --- |
| 最近 N 轮 | 简单问答、低风险聊天 | 早期约束可能丢失 |
| 历史摘要 | 长对话、客服助手 | 摘要错误会被放大 |
| 检索召回 | 文档助手、知识问答 | 依赖切分、索引和权限过滤 |

不要把会话历史长期保存在客户端并原样回传。更合理的做法是：客户端保存会话 ID 和必要 UI 状态，服务端负责历史裁剪、摘要和检索。

## 2.4 Embedding 与 Attention

Embedding 是把文本转换为向量的方法。向量可以理解为一组数字，用来表示文本的语义特征。语义相近的文本，在向量空间中的距离通常更近。它的工程意义是支持知识库检索、相似问题推荐和上下文召回。

例如“忘记密码怎么办”“如何重置登录密码”“登录不上，提示密码错误”字面不同，但语义接近。关键词搜索可能只命中包含“密码”的文档，而向量检索更容易找到账号恢复流程。

Attention 是 Transformer 的关键机制。它不是人的注意力，而是一种计算方法：当模型处理某个 Token 时，会计算它和上下文中其他 Token 的相关程度，然后综合这些信息生成新的表示。它解释了为什么模型能把“iOS 17”“扫码页”“闪退”“AVCaptureSession”“startRunning”等相隔较远的片段关联起来。

这也解释了 Prompt 组织的重要性。同样的信息，如果结构混乱、边界不清，模型更容易关注错误内容。工程化 Prompt 会把角色、任务、上下文、约束和输出格式分开，就是为了帮助模型稳定使用上下文。

## 2.5 概率生成与采样参数

大模型生成文本时，每一步都会给出候选 Token 的概率分布。假设模型正在生成：

```text
建议先____
```

候选概率可能是：

| 候选 Token | 概率 |
| --- | ---: |
| 检查 | 0.62 |
| 查看 | 0.18 |
| 重启 | 0.10 |
| 删除 | 0.06 |
| 忽略 | 0.04 |

如果总选概率最高的 Token，输出更稳定但可能死板；如果按概率随机采样，输出更丰富但更不稳定。这就是同一问题可能得到不同回答的原因。

Temperature 控制概率分布的“尖锐程度”。温度低时，高概率 Token 更占优势；温度高时，低概率 Token 更容易被采到。Top-p 会选取累计概率达到阈值的一组候选，再在集合中采样，减少从低概率长尾里跑偏。

移动端工程场景可以这样选择：

| 场景 | 建议温度 | 原因 |
| --- | ---: | --- |
| 意图分类 | 0-0.2 | 需要稳定分类 |
| JSON 信息抽取 | 0-0.3 | 需要格式稳定 |
| 崩溃日志归因 | 0.2-0.5 | 需要谨慎推理 |
| 用户文案润色 | 0.5-0.8 | 需要表达多样性 |
| 营销创意生成 | 0.8 以上 | 需要更多变化 |

实际项目不建议同时大幅调高 temperature 和 top-p。知识问答、故障分析、权限说明和用户隐私场景应优先选择低温度、明确格式和可追溯引用。

## 2.6 幻觉：机制风险，不是偶发 Bug

幻觉指模型生成看似合理但实际错误的内容。移动端开发中常见幻觉包括：

- 编造不存在的 iOS、Android、Flutter 或 RN API。
- 把旧系统版本的行为当作新版本行为。
- 把通用网络错误归因到具体业务接口。
- 在资料不足时给出确定结论。
- 把截图误解为另一个页面状态。
- 生成看似合法但不可编译的 Swift、Kotlin、Dart 或 TypeScript 代码。

幻觉的根本原因是模型在生成概率上合理的文本，而不是查询事实数据库。工程上不能只要求模型“不要幻觉”，而要降低幻觉空间。

| 风险类型 | 控制方法 |
| --- | --- |
| 编造 API | 代码进入编译、静态检查或测试 |
| 事实过期 | RAG 召回最新文档并展示更新时间 |
| 错误归因 | 输出置信度和证据片段，标注“可能原因” |
| 越权建议 | 权限过滤在检索前执行 |
| 工具误调用 | 工具白名单、参数校验、审计日志和二次确认 |
| 隐私泄露 | 输入脱敏、输出检查和日志脱敏 |

校验失败时，服务端不应“凑合展示”，而应返回明确错误码，例如 `OUTPUT_INVALID`，或降级为人工复核、重新生成、只展示引用资料。

## 2.7 知识时效性

模型参数来自训练阶段。训练完成后，模型本身不会自动知道最新系统版本、新接口、新业务规则或团队内部文档。即使某些模型可以联网或使用工具，那也是应用层提供的外部能力，不是模型参数天然具备的能力。

移动端开发中，时效性尤其重要：

- iOS 和 Android 系统能力持续变化。
- 应用商店审核规则可能更新。
- 设备型号和系统版本碎片化严重。
- 团队内部 SDK、埋点规范和接口协议经常调整。
- 线上事故复盘和故障手册不断更新。

涉及最新事实和团队内部知识时，应优先通过检索、数据库查询、配置中心或工具调用提供上下文。模型负责理解和表达，事实来源应来自可追溯系统。

## 2.8 适合与不适合的任务

大模型适合处理输入复杂、规则难以穷举、输出需要自然语言表达的任务，例如用户反馈分类、崩溃日志初步归因、接口文档摘要、App 内知识库问答、多语言文案初稿、截图问题分析、测试用例生成、需求说明整理和代码 Review 辅助。

这些任务的共同点是：模型输出可以作为建议、草稿、候选结果或辅助解释，再由程序校验、用户确认或人工复核。

| 任务 | 模型角色 | 确定性系统角色 |
| --- | --- | --- |
| 用户反馈分类 | 生成分类候选和理由 | 校验枚举值、写入工单系统 |
| 崩溃日志分析 | 总结可能原因和排查步骤 | 脱敏、检索、关联版本信息 |
| 文档问答 | 基于引用资料生成答案 | 权限过滤、检索和引用来源 |
| 多语言文案 | 生成翻译或润色草稿 | 敏感词和长度校验 |
| 测试用例生成 | 生成用例草稿 | 测试平台执行与结果记录 |

以下任务不应直接由模型独立决定：支付、退款、删除数据、修改权限、自动封禁、生成并直接执行 SQL/Shell、修改线上配置、无人工确认地发送正式通知或法律文本。模型可以整理信息、生成草稿、解释原因和准备参数，但最终执行必须由确定性业务逻辑、权限系统和人工确认控制。

## 2.9 从模型边界反推产品设计

理解模型机制后，移动端产品设计会更清楚。

第一，客户端不直接调用模型供应商。API Key、Prompt 模板、检索逻辑、审计日志和限流策略都放在服务端。客户端只调用自有 API。

第二，长任务必须有状态反馈。因为模型按 Token 生成，移动端应支持“生成中”“停止生成”“重新生成”“复制答案”“查看引用”等交互。

第三，答案要可追溯。知识库问答、故障分析和权限说明都应该展示引用来源，避免用户把模型输出当作不可质疑的事实。

第四，失败路径要明确。模型超时、限流、无资料、格式错误、权限不足，都应该有清晰客户端提示，而不是统一显示“AI 出错了”。

第五，模型输出要进入校验流程。结构化结果做 Schema 校验；代码建议编译或静态检查；工具调用做权限校验；高风险动作让用户确认。

以崩溃日志分析助手为例，请求可以设计为：

```json
{
  "request_id": "req_20260621_001",
  "session_id": "s_123",
  "app_version": "6.8.0",
  "platform": "ios",
  "os_version": "17.5",
  "device_model": "iPhone15,3",
  "question": "请分析这段扫码页闪退日志",
  "attachments": [
    {
      "type": "crash_log",
      "object_key": "uploads/crash/req_20260621_001.log"
    }
  ]
}
```

响应可以包含 `summary`、`possible_causes`、`next_steps`、`confidence` 和 `citations`。移动端展示时应标注“初步分析”，并提供引用、复制、反馈和创建工单入口。

## 2.10 运行采样实验

配套工程提供了 temperature 采样实验：

```bash
cd examples/01-mobile-knowledge-assistant
python3 scripts/sampling_temperature_experiment.py
```

这个脚本不会调用真实模型，而是用固定候选概率模拟不同 temperature 下的采样差异。读者可以观察：低温度更稳定，高温度更多样，但也更容易偏离主路径。

## 本章小结

大模型的核心能力来自对上下文的表示、关联和概率生成。它擅长理解非结构化输入、生成自然语言和辅助复杂任务，但它不是事实数据库，也不是权限系统，更不是自动执行系统。移动端大模型应用的关键，是把模型放在合适的位置：用服务端保护密钥和上下文，用 RAG 提供事实来源，用校验和人工确认约束风险，用流式输出、取消和错误状态保证体验可控。

## 实践练习

1. 运行 temperature 采样实验，记录低温度和高温度输出差异。
2. 为一个崩溃日志助手设计请求和响应 JSON。
3. 找出你所在 App 中 3 个适合模型辅助、但不适合模型独立决策的功能。
4. 为“编造不存在 API”设计一个校验流程。
5. 为知识问答页面设计引用来源展示样式和资料不足状态。
