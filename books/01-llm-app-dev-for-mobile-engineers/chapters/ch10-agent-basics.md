# 第 10 章 Agent 基础

## 本章导读

Agent 不是“让模型自动乱跑”，而是让模型在受控边界内完成观察、计划、调用工具、读取结果和停止的循环。对移动端工程师来说，Agent 的价值在于处理多步骤问题，例如分析文件、排查日志、整理资料、提出修改建议；风险在于工具权限、循环失控、错误参数和未经确认的高风险动作。

配套脚本：`examples/01-mobile-knowledge-assistant/scripts/file_triage_agent.py`

## 学习目标

- 理解 Agent 与普通问答、RAG、固定 Workflow 的区别。
- 掌握 Agent 执行循环：观察、计划、行动、读取结果、停止。
- 设计工具白名单、参数校验、最大步数和停止条件。
- 理解记忆、上下文和审计日志的边界。
- 运行只读文件分析 Agent，读懂执行轨迹。

## 10.1 Agent 的工程定义

在本书语境中，Agent 是一种受控循环：

```text
目标 -> 观察上下文 -> 选择工具 -> 执行工具 -> 读取结果 -> 判断是否继续
```

它和普通问答的差别在于：普通问答一次生成答案；Agent 可以多次调用工具并根据结果继续决策。它和 RAG 的差别在于：RAG 通常只检索资料再生成答案；Agent 可以按目标选择不同工具。它和 Workflow 的差别在于：Workflow 的步骤固定，Agent 的下一步可以由模型或规划器动态决定。

Agent 不能脱离边界。它能调用哪些工具、每个工具接收什么参数、最多执行几步、什么时候必须停止，都应由服务端控制。

## 10.2 执行循环

一个最小 Agent 循环包含 5 类信息。

| 信息 | 作用 |
| --- | --- |
| `goal` | 用户目标或系统任务 |
| `state` | 当前已知事实、已读文件、已执行动作 |
| `tools` | 可用工具及其参数 Schema |
| `trace` | 每一步观察、动作和结果 |
| `stop_condition` | 成功、失败、最大步数、人工确认或错误 |

配套脚本采用确定性规划器，而不是直接调用真实模型。这样读者无需 API Key，也能看清 Agent 边界。把规划器替换为模型后，工具白名单、参数校验、最大步数和审计轨迹仍然必须保留。

## 10.3 工具是能力边界

Agent 的能力来自工具，风险也来自工具。只读文件分析 Agent 只提供有限工具，例如列出文档、读取文档片段、按关键词搜索和输出报告。它不提供删除文件、执行 Shell、修改配置、上传资料等能力。

工具设计要遵守 4 个原则：

- 白名单：模型只能选择服务端注册过的工具。
- Schema：参数必须符合类型、枚举、长度和路径限制。
- 最小权限：只给当前任务需要的能力。
- 可审计：每次工具调用都写入执行轨迹。

不要用 Prompt 代替工具权限。Prompt 中写“不要删除文件”不能保证安全；不提供删除工具，才是可靠边界。

## 10.4 参数校验与路径安全

模型输出的参数不能直接使用。文件路径、关键词、数量限制、用户 ID、订单 ID 都要确定性校验。配套脚本只允许读取示例文档目录中的文件，拒绝绝对路径和目录逃逸。

常见校验包括：

| 参数 | 校验 |
| --- | --- |
| 文件路径 | 必须位于允许目录，不能包含 `..` 逃逸 |
| 关键词 | 长度限制、空值拒绝、必要时脱敏 |
| top_k | 数值范围限制 |
| 工具名 | 必须在白名单中 |
| 输出报告 | 只能包含已读资料中的证据 |

参数校验失败时，Agent 应记录失败并停止或改用其他安全路径，而不是猜测用户意图继续执行。

## 10.5 最大步数和停止条件

Agent 必须有最大步数。没有最大步数，模型可能在“还需要更多信息”的判断中反复调用工具，造成成本、延迟和资源问题。

常见停止条件：

- 已找到足够证据，可以生成报告。
- 资料不足，无法继续判断。
- 工具调用失败且无法恢复。
- 达到最大步数。
- 需要人工确认。
- 命中安全策略。

移动端页面要把停止原因展示清楚。`done`、`no_evidence`、`tool_error`、`max_steps_reached`、`confirmation_required` 对用户和工程师意味着不同处理方式。

## 10.6 记忆不是随便保存上下文

Agent 的“记忆”可以是会话摘要、已读资料、工具结果、用户偏好或任务状态，但不能把所有上下文无差别保存。移动端场景尤其要注意隐私、Token 成本和过期信息。

建议区分：

| 类型 | 是否长期保存 | 说明 |
| --- | --- | --- |
| 执行轨迹 | 可短期保存 | 用于调试、审计和复盘 |
| 用户隐私原文 | 默认不保存 | 必须脱敏或引用对象 ID |
| 会话摘要 | 可保存 | 标明来源和更新时间 |
| 工具结果 | 按业务保存 | 遵守权限、租户和过期策略 |

Agent 的上下文应由服务端管理，移动端只保存任务 ID、页面状态和必要展示内容。

## 10.7 Agent、Workflow 与 RAG

三者可以组合，但边界不同。

| 形态 | 特点 | 适合场景 |
| --- | --- | --- |
| RAG | 检索资料后生成答案 | 文档问答、知识助手 |
| Workflow | 固定步骤、确定性门禁 | 周报、审批、发布流程 |
| Agent | 动态选择工具和下一步 | 文件分析、问题排查、多步骤探索 |

能用 Workflow 解决的，不必强行做 Agent。固定流程更容易测试、审计和解释。Agent 适合任务路径不完全确定，但工具边界可以清楚限定的场景。

## 10.8 移动端 Agent 的适用边界

适合 Agent 的移动端场景包括：分析崩溃日志和相关文档、整理多个工单的共性、检查知识库覆盖情况、生成排查报告、为测试人员整理复现步骤。

不适合直接交给 Agent 的场景包括：自动删除数据、自动退款、自动发布配置、自动关闭告警、无确认执行命令。Agent 可以整理证据和建议，但执行必须由确定性业务系统和人工确认控制。

## 10.9 运行文件分析 Agent

运行：

```bash
cd examples/01-mobile-knowledge-assistant
python3 scripts/file_triage_agent.py \
  --goal '检查移动端知识库是否覆盖密钥、流式输出、权限和脱敏要求' \
  --keyword 'API Key' \
  --keyword '流式输出' \
  --keyword '权限' \
  --keyword '脱敏'
```

输出会包含目标、步骤、读取的资料、命中的关键词、结论和停止原因。读者应重点看两件事：每一步调用了哪个工具，以及结论是否能追溯到已读资料。

## 10.10 人工确认

当 Agent 想执行高风险动作时，应返回确认请求，而不是继续执行。确认请求至少包含动作、对象、影响范围、风险等级和过期时间。移动端负责展示确认卡，服务端负责二次校验和执行。

确认流程不能只依赖前端按钮状态。服务端应生成 `confirmation_id`，绑定原始 `request_id`、工具名、参数摘要和过期时间；用户确认后，服务端再次检查权限和资源状态。

## 本章小结

Agent 的核心不是“模型更聪明”，而是受控循环：目标、状态、工具、轨迹和停止条件。移动端大模型应用引入 Agent 时，要先设计工具白名单、参数校验、最大步数、审计轨迹和人工确认。只有这些边界成立，Agent 才能从演示变成可上线能力。

## 实践练习

1. 运行 `file_triage_agent.py`，记录每一步工具调用和停止原因。
2. 新增一个只读工具，用于统计文档章节数量。
3. 为路径参数增加目录逃逸测试。
4. 设计 `max_steps_reached` 的移动端提示。
5. 说明哪些动作必须进入人工确认，而不能由 Agent 自动执行。
