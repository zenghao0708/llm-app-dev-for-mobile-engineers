# 第 11 章 多工具调用与工作流编排

## 本章导读

多工具调用不等于 Agent。很多业务流程的步骤是固定的：收集输入、生成草稿、校验字段、人工确认、发布结果、记录日志。这样的任务更适合 Workflow。Workflow 用确定性步骤组织模型和工具，让模型只处理适合生成的部分，把权限、校验、确认和发布交给系统。

配套脚本：`examples/01-mobile-knowledge-assistant/scripts/weekly_report_workflow.py`

## 学习目标

- 区分 Workflow、Agent 和 RAG 的边界。
- 设计工作流节点：输入、模型、工具、校验、确认、发布。
- 理解为什么模型草稿不能直接进入发布系统。
- 设计移动端工作流状态、失败重试和幂等策略。
- 运行周报工作流，读懂步骤列表和人工确认门禁。

## 11.1 Workflow 与 Agent 的边界

Workflow 是固定流程，Agent 是动态循环。Workflow 的下一步通常由流程定义决定；Agent 的下一步可能由模型或规划器决定。

| 形态 | 下一步如何决定 | 适合场景 |
| --- | --- | --- |
| RAG | 检索后生成 | 知识问答、引用答案 |
| Workflow | 流程固定 | 周报、审批、发布、工单创建 |
| Agent | 根据状态动态选择 | 文件分析、多步骤排查 |

能用 Workflow 解决的问题，不必强行做 Agent。固定流程更容易测试、审计、回滚，也更容易向移动端解释当前状态。

## 11.2 多工具调用不等于 Agent

一个流程可以调用多个工具，但仍然不是 Agent。例如周报生成流程可能读取输入、生成摘要、检查敏感词、等待确认、写出 Markdown 文件。这些步骤都固定，模型只在“生成摘要”节点参与。

多工具流程要解决的是编排问题，而不是让模型自由选择一切。确定性节点越多，线上行为越稳定。

## 11.3 工作流节点设计

一个常见工作流可以拆成：

| 节点 | 职责 | 是否适合模型 |
| --- | --- | --- |
| 输入收集 | 收集日期、项目、数据源、用户身份 | 不适合 |
| 字段校验 | 校验必填、类型、枚举和权限 | 不适合 |
| 模型生成 | 生成摘要、解释、草稿、建议 | 适合 |
| 工具调用 | 查询数据、创建草稿、写文件 | 由服务端执行 |
| 结果校验 | 检查格式、敏感词、链接、长度 | 不适合只靠模型 |
| 人工确认 | 高风险动作或正式发布前确认 | 必须由系统控制 |
| 发布记录 | 写入文件、工单、CMS 或消息系统 | 由服务端执行 |

模型节点只负责适合生成的部分。发布、权限、校验和审计必须由确定性代码负责。

## 11.4 输入收集与字段校验

移动端发起工作流时，不应把所有内容塞进自然语言。应传递结构化字段，例如：

```json
{
  "request_id": "req_weekly_001",
  "week": "2026-W25",
  "project": "mobile-ai-assistant",
  "audience": "mobile-team",
  "input_file": "data/workflow/weekly_report_inputs.json"
}
```

服务端校验字段后再进入模型节点。缺少必填字段、日期格式错误、用户无权访问数据源时，应返回稳定错误码，而不是让模型猜。

## 11.5 模型节点与校验节点

模型节点适合生成摘要、风险说明、下一步建议和面向读者的表达。但模型草稿不能直接进入发布系统。校验节点至少检查：

- 输出是否包含必填区块。
- 是否包含敏感词、密钥、手机号、内部链接等不应发布内容。
- 链接和文件引用是否存在。
- 文案长度是否符合渠道要求。
- 是否包含未经证实的确定性结论。

校验失败时，工作流可以重新生成一次、返回给用户编辑，或进入人工复核。不要为了流程顺利而忽略校验失败。

## 11.6 人工确认门禁

高风险动作必须有人工确认。确认不仅是 UI 按钮，也是一条服务端门禁。

确认卡应包含：

- 将要执行的动作。
- 发布对象或影响范围。
- 草稿预览。
- 风险提示。
- 确认 ID 和过期时间。
- 取消与确认按钮。

用户确认后，服务端应再次校验权限、草稿版本和确认 ID。不要让移动端仅凭本地状态直接执行发布。

## 11.7 工作流状态与移动端页面

移动端应展示清晰状态，而不是只显示“AI 处理中”。

| 状态 | 含义 | 页面表现 |
| --- | --- | --- |
| `collecting` | 等待输入 | 表单或上传入口 |
| `validating` | 校验字段和权限 | 加载态 |
| `generating` | 模型生成草稿 | 流式或进度提示 |
| `checking` | 规则校验草稿 | 检查中 |
| `waiting_confirmation` | 等待用户确认 | 确认卡 |
| `publishing` | 写入目标系统 | 禁止重复提交 |
| `done` | 完成 | 展示结果和入口 |
| `failed` | 失败 | 展示错误和重试策略 |

旧请求的事件不能覆盖新请求。所有事件都应带 `request_id`，移动端根据当前活动任务决定是否渲染。

## 11.8 失败重试与幂等

Workflow 很容易遇到部分成功：模型生成成功但发布失败，文件已写入但回调失败，用户重复点击确认。必须设计幂等。

建议：

- 所有写操作绑定 `request_id` 或业务幂等键。
- 发布节点记录目标路径、版本号和写入结果。
- 确认卡设置过期时间。
- 重试只重试可重入节点，不能重复执行高风险动作。
- 失败日志记录节点名、错误码和可恢复建议。

移动端的“重试”按钮不应简单重新发起整个流程，而应让服务端判断从哪个节点恢复。

## 11.9 运行周报工作流

运行：

```bash
cd examples/01-mobile-knowledge-assistant
python3 scripts/weekly_report_workflow.py
```

脚本读取 `data/workflow/weekly_report_inputs.json`，生成周报草稿，执行校验，再进入确认或发布步骤。它展示了一个典型模式：模型负责生成草稿，系统负责字段校验、敏感值检查、确认门禁和输出文件。

读输出时重点看：

- `steps`：每个节点的状态和耗时。
- `draft`：模型或模板生成的草稿。
- `checks`：校验是否通过。
- `requires_confirmation`：是否等待人工确认。
- `output_path`：发布或写出的目标文件。

这个脚本也可以改造成工单工作流：输入变成用户反馈和日志，模型节点生成工单摘要，校验节点检查字段，确认节点让用户确认后再创建工单。

## 11.10 从示例到生产

生产工作流需要补齐：

| 方向 | 要求 |
| --- | --- |
| 身份权限 | 登录态、租户、角色、数据范围 |
| 状态存储 | 持久化步骤、草稿、确认 ID 和结果 |
| 队列执行 | 长任务进入任务队列，支持取消和重试 |
| 版本管理 | Prompt、流程定义和校验规则可回滚 |
| 审计日志 | 记录输入摘要、节点结果、确认人和发布时间 |
| 监控告警 | 失败率、耗时、重试次数和人工确认耗时 |

不要把所有逻辑都写进 Prompt。Prompt 只能影响模型输出；Workflow 的可靠性来自流程定义、状态机、校验和审计。

## 本章小结

Workflow 适合固定步骤的业务流程，Agent 适合路径动态的问题探索。移动端大模型应用应优先判断任务能否用 Workflow 表达：输入、模型生成、校验、确认、发布、审计。模型负责草稿和解释，服务端负责边界和执行，移动端负责状态展示和用户确认。
