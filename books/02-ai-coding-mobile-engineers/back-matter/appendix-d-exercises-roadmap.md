# 附录 D 读者练习与能力路线图

本附录给出一套从入门到团队落地的练习路线。读者不需要一次完成所有练习，可以按自己的项目成熟度选择。

## 练习 1：只读理解仓库

目标：让 agent 在不修改代码的情况下理解一个移动端模块。

输入：

- 目标模块路径。
- README 或项目地图。
- 相关测试目录。

Prompt：

```text
请只读分析，不要修改代码。
目标是理解 [模块名]。
请输出：
- 模块职责。
- 主要入口文件。
- 依赖方向。
- 现有测试。
- 修改该模块时应运行的命令。
- 可能的高风险点。
```

验收：

- 输出能帮助新人理解模块。
- 没有编造不存在的文件。
- 没有修改代码。

## 练习 2：补一个低风险测试

目标：让 agent 为一个纯逻辑函数或 ViewModel 补测试。

适合对象：

- 错误码 mapper。
- 表单校验。
- ViewModel 状态转换。
- 本地化 key 检查。

要求：

- 不修改生产逻辑，除非先说明原因。
- 不使用真实网络。
- 不使用脆弱 sleep。
- 运行对应测试。

复盘问题：

- 测试是否覆盖真实风险？
- 是否只断言实现细节？
- 是否容易偶发失败？
- agent 是否遵守了不改生产逻辑的约束？

## 练习 3：小范围 bug 修复

目标：用 agent 修复一个可复现 bug。

输入：

- 复现步骤。
- 错误日志。
- 相关文件。
- 测试命令。

流程：

1. 先让 agent 解释问题，不修改。
2. 让 agent 写失败测试。
3. 人工确认测试。
4. 让 agent 做最小修复。
5. 运行测试。
6. 输出 PR 摘要。

停止条件：

- 需要修改高风险文件。
- 需要新增依赖。
- 连续两轮测试仍失败。

## 练习 4：受控重构

目标：练习把大任务拆成小循环。

示例任务：

- 抽取统一错误模型。
- 拆分一个大型 UI 组件。
- 统一网络状态枚举。
- 清理重复 formatter。

规则：

- 每轮最多修改 3-5 个文件。
- 每轮必须有测试或检查。
- 每轮输出下一步，但不自动执行下一步。
- 高风险页面只读分析。

复盘：

- diff 是否容易审查？
- 是否有回滚方式？
- 是否出现无关格式化？
- 是否越过模块边界？

## 练习 5：跨端桥接审查

目标：训练 Flutter/RN 项目中的双平台思维。

任务：

- 选择一个 MethodChannel 或 Native Module。
- 让 agent 对比跨端层、iOS 和 Android 实现。
- 输出协议表和风险清单。

验收：

- iOS/Android 参数一致。
- 错误码一致。
- 用户取消、权限拒绝、系统不可用都有处理。
- 未验证平台被明确列出。

## 练习 6：建立团队 AI Coding 规则

目标：为团队写第一版 agent 规则。

产物：

```text
docs/ai-coding/
  project-map.md
  commands.md
  high-risk-files.md
  prompt-templates.md
  review-checklist.md
```

最低要求：

- 常用测试命令写清。
- 高风险文件写清。
- 禁止命令写清。
- PR 输出格式写清。
- 敏感信息处理写清。

## 能力路线图

可以把个人和团队能力分成四个阶段。

| 阶段 | 目标 | 典型练习 |
| --- | --- | --- |
| 入门 | 会让 agent 做只读理解和小测试 | 练习 1、2 |
| 进阶 | 会做小 bug 和小重构 | 练习 3、4 |
| 熟练 | 会设计上下文、harness 和 loop | 练习 4、5 |
| 团队化 | 会沉淀规则、评测和治理 | 练习 6 |

每个阶段都不要跳过验证。AI coding 的能力不是看生成了多少代码，而是看结果是否可运行、可审查、可复盘。

## 30 天学习计划

第 1 周：理解工具和工作区。

- 阅读第 1-3 章。
- 在自己的项目中运行只读分析。
- 写一份项目地图。
- 整理常用测试命令。

第 2 周：练习四类工程能力。

- 阅读第 4-7 章。
- 为一个 bug 写任务 prompt。
- 准备上下文包。
- 运行一次小循环。
- 记录失败和改进。

第 3 周：进入移动端真实场景。

- 阅读第 8-11 章。
- 选择 iOS/Android/Flutter/RN 中一个低风险任务。
- 让 agent 补测试或修小 bug。
- 审查 diff 和测试结果。

第 4 周：团队化和评测。

- 阅读第 12-14 章。
- 写 high-risk-files。
- 设计 3 个评测任务。
- 产出一份 AI coding PR 模板。
- 复盘哪些任务适合继续使用 agent。

## 自评清单

完成本书后，读者应能回答：

- 如何判断一个任务是否适合 agent？
- 如何写出有边界的 prompt？
- 如何组织移动端上下文？
- 如何设计命令白名单和高风险文件清单？
- 如何让 agent 在失败后停止而不是乱试？
- 如何评测 AI coding 的收益？
- 如何把个人经验沉淀成团队 playbook？

如果这些问题仍然模糊，建议不要急着扩大使用范围。先回到低风险任务中练习，把流程跑稳。
