# 第 10 章 调试、重构与测试自动化

## 本章导读

AI coding 最容易产生稳定收益的三类任务是调试、重构和测试补充。它们都有明确输入、明确目标和可验证结果。对于移动端工程师来说，真正值得沉淀的不是“让 agent 写一段代码”，而是把调试现场、重构边界和测试反馈组织成可重复的工作流。

本章讨论三件事：如何让 agent 参与 bug 定位，如何控制重构范围，如何让测试成为 agent 的反馈信号。它们分别对应“理解问题”“改变代码”和“证明结果”。

## 学习目标

- 掌握用 agent 处理 bug、重构和测试的任务拆分方式。
- 知道如何控制重构范围。
- 能够让测试成为 agent 的反馈信号。
- 能够识别 agent 在调试和重构中的常见误判。

## 调试任务：先还原现场，再修改代码

调试任务要给 agent 足够现场：

- 崩溃堆栈。
- 设备型号和系统版本。
- App 版本、构建号、渠道或 flavor。
- 最近变更。
- 复现步骤。
- 相关日志片段。
- 期望行为和实际行为。
- 已尝试过的排查动作。

很多失败的 AI 调试都来自输入太少。只给一句“页面偶现崩溃，帮我修一下”，agent 只能猜。更好的做法是要求 agent 先做三件事：解释堆栈、列出假设、说明还需要哪些证据。只有当假设和证据对齐后，才进入修改阶段。

### 崩溃分析 Prompt

```text
你是移动端崩溃分析助手。先不要修改代码。

输入：
- 崩溃堆栈如下：...
- 设备：iPhone 15，iOS 18.5
- App 版本：5.8.0 build 2301
- 最近变更：Profile 页面网络错误处理重构
- 复现步骤：打开设置页 -> 点击头像 -> 断网后重试

任务：
1. 用 5-8 句话解释堆栈指向的调用链。
2. 列出不超过 3 个修复假设。
3. 标出每个假设需要验证的文件、日志或测试。
4. 在我确认前不要修改代码。
```

这个 prompt 的核心是“先不要修改代码”。调试的第一轮如果直接改代码，容易把猜测变成新 bug。

## 调试中的假设表

建议让 agent 输出一张假设表：

| 假设 | 支持证据 | 反证方式 | 下一步 |
| --- | --- | --- | --- |
| 异步回调返回后页面已销毁 | 堆栈指向 UI 更新，日志有页面退出 | 增加生命周期日志或测试取消路径 | 检查 task cancel |
| 错误对象为空导致强制解包 | 堆栈指向 unwrap | 构造 nil error 测试 | 修复 mapper |
| 多次点击导致重复提交 | 日志中同一 requestId 出现两次 | 增加重复点击测试 | 加提交中状态 |

这张表能把 agent 的推理变成可审查对象。人类工程师不需要完全相信模型，只需要检查它的假设是否被证据支持。

## 重构任务：边界比方案更重要

重构任务的关键是边界：

- 一次只重构一个模块或一种重复模式。
- 禁止同时改业务行为和架构。
- 先加测试或快照，再改实现。
- 每轮输出修改文件列表。
- 每轮说明回滚方式。
- 对高风险文件要求人工确认。

移动端项目中，重构经常牵涉 UI、状态、网络、埋点、本地化和平台能力。如果 prompt 只说“帮我优化一下代码结构”，agent 很可能做出风格化但不可控的大 diff。正确做法是把重构拆成多轮，每轮只有一个目标。

### 可控重构 Prompt

```text
你是移动端重构助手。本轮只做第一步：抽取网络错误到统一错误模型。

允许：
- 新增 AppError 类型。
- 修改 NetworkClient 的错误映射。
- 为 AppError 增加单元测试。

禁止：
- 迁移页面调用方。
- 修改 UI 文案。
- 修改埋点。
- 修改登录、支付、风控逻辑。
- 修改构建配置和依赖。

验收：
- 新增 AppErrorTests。
- 运行指定测试命令。
- 输出下一轮建议，但不要执行下一轮。
```

这个 prompt 把“想要完成的重构”和“本轮允许执行的重构”区分开。大任务可以有路线图，但每轮执行必须小。

## 测试自动化：让测试成为反馈信号

AI coding agent 可以快速补测试，但测试质量需要人类审查：

- 是否覆盖真实风险。
- 是否只断言实现细节。
- 是否引入脆弱 mock。
- 是否能在 CI 中稳定运行。
- 是否会因为时间、线程、网络、系统语言而偶发失败。

移动端测试有几个常见层次：

| 测试层次 | iOS 示例 | Android 示例 | 适合 agent 做什么 |
| --- | --- | --- | --- |
| 纯逻辑单测 | XCTest | JUnit | 补边界案例 |
| ViewModel 测试 | XCTest + async | Coroutine test | 验证状态流 |
| UI/快照测试 | Snapshot、XCUITest | Robolectric、Compose test | 补关键状态 |
| 集成测试 | 真机、模拟器 | emulator、device farm | 生成脚本和清单 |

Agent 最适合补前两层测试。UI 和集成测试也能参与，但需要更强 harness，否则容易写出不稳定测试。

## 测试先行的 AI 流程

对 bug 修复和重构，推荐使用“测试先行”的轻量流程：

1. 让 agent 解释问题，不改代码。
2. 让 agent 写一个失败测试。
3. 人工确认测试确实表达需求。
4. 让 agent 做最小实现。
5. 运行测试。
6. 如果失败，让 agent 根据失败日志修复。
7. 输出最终 diff 和风险说明。

这个流程并不要求严格 TDD，但它能防止 agent 只改到“看起来合理”。测试是 agent 的反馈回路，也是人工审查的支点。

## 日志和隐私

调试经常需要日志，但移动端日志可能包含用户数据、设备标识、地理位置、账号、订单、token 或内部接口。给 agent 提供日志前应先脱敏：

- 替换手机号、邮箱、用户 ID。
- 删除 token、cookie、Authorization header。
- 删除精确地理位置。
- 删除生产接口真实域名中的敏感路径。
- 保留 requestId、错误码、时间顺序和调用链。

脱敏不能把日志变得无用。调试需要的是结构和关系，而不是用户真实身份。

## 常见失败模式

调试、重构和测试自动化中的常见失败包括：

- 没有复现条件，直接猜修复。
- 为了让测试通过而修改测试期望。
- 把业务行为改成了测试容易断言的行为。
- 重构同时改变 API、UI、埋点和文案。
- 引入不稳定异步测试。
- 忽略真机和系统版本差异。
- 没有输出未验证项。

解决思路不是“写更长 prompt”，而是让流程分阶段：先理解，再计划，再小改，再验证。

## 移动端调试案例：取消后的状态回写

下面用一个常见问题串起调试流程。某 iOS 页面在弱网下点击重试，退出页面后偶现崩溃。堆栈显示崩溃发生在 `ProfileViewModel` 更新 UI 状态的位置。

不成熟的 agent 可能直接给出结论：“页面释放后仍更新状态，给闭包加 weak self。”这不是错误答案，但它可能过早。成熟流程应该先收集证据：

1. 查看页面生命周期，确认退出页面时是否取消请求。
2. 查看网络层是否支持取消。
3. 查看 ViewModel 是否在回调中检查任务状态。
4. 查找已有取消测试。
5. 增加一个可复现的失败测试。

适合给 agent 的任务是：

```text
先不要修复。请检查 ProfileViewModel、ProfileRepository 和 NetworkSession 的取消路径。
输出：
- 请求从发起到 UI 状态更新的调用链。
- 取消发生在哪一层。
- 是否已有测试覆盖 cancel 后回调。
- 一个最小失败测试建议。
```

确认后再进入修改：

```text
请只修复 cancel 后仍回写 UI 状态的问题。
约束：
- 不修改网络协议。
- 不迁移其他页面。
- 不改变错误文案。
验收：
- 新增 cancel 后不更新状态的测试。
- 运行 ProfileViewModelTests。
```

这个案例说明，AI 调试不是“让模型猜根因”，而是让它帮助构造证据链。

## 重构案例：从重复错误文案到统一错误模型

重构比 bug 修复更容易失控，因为它通常没有单一失败现象。以网络错误文案为例，多个页面都写了自己的错误处理：

- 首页把 timeout 显示为“网络异常”。
- 设置页把 timeout 显示为“请求超时”。
- 订单页把 5xx 显示为“系统繁忙”。
- 埋点字段有的叫 `error_code`，有的叫 `code`。

如果 prompt 只说“统一网络错误处理”，agent 可能一次迁移所有页面。更稳妥的重构路线是：

1. 只建立统一错误模型。
2. 只给错误模型补测试。
3. 迁移一个低风险页面。
4. 人工确认文案和埋点。
5. 再迁移下一批页面。

每一步都可以形成一个独立 PR。这样即使中途发现产品文案需要调整，也不会影响已经验证过的基础模型。

## 测试生成的审查清单

Agent 生成测试后，工程师要审查：

| 检查项 | 好测试 | 坏测试 |
| --- | --- | --- |
| 断言目标 | 断言用户可见行为或公开状态 | 断言私有实现细节 |
| 数据构造 | 使用最小必要输入 | 构造大量无关 mock |
| 异步控制 | 使用测试调度器或显式等待 | 使用脆弱 sleep |
| 失败可读性 | 失败信息指向业务语义 | 失败只能看到对象不相等 |
| 稳定性 | 与时间、语言、设备解耦 | 依赖本地时区或系统语言 |

AI 很擅长批量生成测试，但也容易生成“看起来覆盖了很多，实际只验证实现细节”的测试。测试的质量不在数量，而在它是否守住真实风险。

## 调试输出模板

建议要求 agent 在调试结束时输出：

```text
根因：
- ...

证据：
- 堆栈：...
- 测试：...
- 代码路径：...

修改：
- 文件 A：...
- 文件 B：...

验证：
- 命令：...
- 结果：...

未验证：
- 真机弱网尚未验证。
- iOS 低版本尚未验证。

风险：
- ...
```

这种输出让调试过程可审查，也让 PR 评审者不用重新翻整段对话。

## 本章小结

让 agent 做调试、重构和测试，必须把反馈信号变成第一等公民。没有测试的自动重构只是更快地产生风险。成熟的移动端 AI coding 工作流，会要求 agent 先建立假设，用测试验证假设，再在明确边界内修改代码。
