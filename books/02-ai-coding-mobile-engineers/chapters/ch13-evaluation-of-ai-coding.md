# 第 13 章 AI Coding 效果评测

## 本章导读

AI coding 是否有效，不能只靠主观感觉。团队需要评估任务成功率、修改质量、测试覆盖、审查成本和回归风险。对于移动端团队，还要关注崩溃率、启动时间、包体、权限、兼容性和发版风险。

评测的目标不是证明某个 agent “很强”，而是回答三个工程问题：

1. 它适合做哪些任务？
2. 它需要什么上下文和工具才能稳定完成？
3. 它在哪些边界上必须由人类接管？

## 学习目标

- 建立 AI coding 评测指标。
- 知道如何设计移动端任务集。
- 能够区分效率提升和质量下降。
- 能够用评测结果改进 prompt、context、harness 和 loop。

## 评测维度

| 维度 | 指标 |
| --- | --- |
| 任务完成 | 完成率、一次通过率、平均轮数 |
| 代码质量 | diff 大小、架构一致性、重复代码、复杂度 |
| 测试验证 | 测试是否新增、测试是否通过、覆盖风险点 |
| 审查成本 | 人工修改次数、审查意见数量 |
| 安全合规 | 是否触碰敏感文件、是否泄露秘密 |
| 移动端质量 | 崩溃率、启动时间、包体、权限、兼容性 |

单一指标会误导团队。例如“代码生成速度提升 50%”没有意义，如果它同时让审查时间翻倍、线上回归增加。AI coding 的评测必须同时看效率和质量。

## 任务集设计

移动端 AI coding 评测任务可以包括：

- 修复一个有堆栈的崩溃。
- 给 ViewModel 补测试。
- 拆分一个复杂 UI 组件。
- 迁移一个网络错误模型。
- 修复一个跨端平台差异。
- 生成一份发版检查文档。
- 修复一个构建失败。
- 补齐一个本地化遗漏。
- 检查一次权限变更风险。

任务集要覆盖不同难度，而不是只挑 agent 擅长的任务。建议分为三档：

| 难度 | 特征 | 示例 |
| --- | --- | --- |
| L1 | 单文件或少量文件，可单测验证 | 表单校验、mapper 测试 |
| L2 | 跨 3-8 个文件，需要理解模块边界 | ViewModel 状态重构、组件拆分 |
| L3 | 跨平台或高风险，需要人工确认 | 权限、桥接、依赖升级、发版检查 |

如果一个 agent 只在 L1 表现好，仍然有价值，但不能宣传成能自动完成复杂工程任务。

## 评测记录模板

建议每个评测任务记录：

```yaml
task_id: mobile-ai-coding-001
title: 修复 ProfileViewModel 弱网错误提示
level: L2
platform: iOS
allowed_paths:
  - Sources/Profile/**
  - Tests/ProfileTests/**
forbidden_paths:
  - ios/Signing/**
  - Info.plist
commands:
  - xcodebuild test ...
expected:
  - timeout 显示弱网提示
  - offline 显示离线提示
  - serverError 显示服务端错误
result:
  completed: true
  test_passed: true
  human_edits: 1
  review_findings: 2
  touched_risky_files: false
```

这种结构化记录能帮助团队比较不同工具、不同 prompt、不同上下文策略的效果。

## 质量分级

可以把 agent 输出分成五级：

| 级别 | 描述 | 处理方式 |
| --- | --- | --- |
| A | 直接可合并，测试和说明完整 | 常规审查 |
| B | 小问题，人工少量修改即可 | 记录改进点 |
| C | 方向正确，但需要明显返工 | 优化上下文和 prompt |
| D | 方案错误或破坏边界 | 降级任务类型 |
| E | 触碰安全红线 | 禁止该类自动化 |

这个分级比“成功/失败”更有信息量。C 级任务并不等于没有价值，它说明 agent 可能需要更好的上下文或更小的任务切分。

## 评测中的对照组

如果团队想知道 AI coding 是否真的提升效率，需要有对照：

- 人工完成同类任务的平均时间。
- 人工审查同类 PR 的平均意见数。
- 过去三个月同类 bug 的回归率。
- 现有测试覆盖和 CI 时长。

没有对照时，团队很容易被“生成速度”影响判断。真正要看的，是从任务开始到可合并 PR 的总时间，以及合并后的质量。

## 移动端特有指标

移动端项目还要关注：

- 启动时间是否变化。
- 包体是否变化。
- 崩溃率是否变化。
- ANR 或卡顿是否变化。
- 权限声明是否变化。
- 离线、弱网、低电量场景是否受影响。
- 多语言和暗黑模式是否受影响。

这些指标不一定每个任务都要测，但如果任务涉及相关模块，就必须纳入验收。

## 用评测改进四类工程能力

评测结果要反哺前面四类能力：

- Prompt Engineering：任务描述是否清楚，禁止事项是否足够。
- Context Engineering：是否提供了正确文件、架构说明和错误现场。
- Harness Engineering：工具、命令、权限和测试是否足够。
- Loop Engineering：是否有清晰的停止条件和人工接管点。

例如某类任务经常失败，不一定是模型能力问题。可能是 prompt 没有限定范围，可能是上下文缺少项目地图，也可能是测试命令不稳定。

## 评测任务样例

下面是一组适合移动端团队起步的评测任务。

| 编号 | 任务 | 难度 | 关注点 |
| --- | --- | --- | --- |
| E01 | 为错误码 mapper 补充边界测试 | L1 | 测试质量 |
| E02 | 修复 ViewModel timeout 状态 | L1 | 最小修改 |
| E03 | 拆分一个 300 行 UI 组件 | L2 | 重构边界 |
| E04 | 迁移两个页面到统一错误模型 | L2 | 上下文和 loop |
| E05 | 分析 Gradle 构建失败 | L2 | 日志理解 |
| E06 | 审查相册权限变更 | L3 | 安全治理 |
| E07 | 修复 Flutter 平台通道错误 | L3 | 双平台验证 |
| E08 | 生成发版风险清单 | L2 | 文档和审计 |

每个任务都应准备标准答案或人工评审标准。否则评测会变成主观印象。

## 评测报告模板

评测结束后，可以输出：

```markdown
# AI Coding 评测报告

## 概览

- 工具：
- 模型/版本：
- 任务集：
- 日期：

## 结果

| 任务 | 完成 | 测试 | 人工修改 | 风险 |
| --- | --- | --- | --- | --- |

## 主要发现

- 适合任务：
- 不适合任务：
- 常见失败：
- 需要补充的上下文：
- 需要补充的 harness：

## 后续动作

- 更新 AGENTS.md。
- 增加高风险文件清单。
- 新增模块级测试命令。
- 调整 prompt 模板。
```

评测报告要服务于改进流程，而不是只给工具打分。

## 线上指标与离线评测的关系

离线评测只能说明 agent 在任务集上的表现，不能直接代表线上质量。移动端团队还要观察合并后的实际影响：

- agent 参与的 PR 是否引入回归。
- 是否增加线上崩溃或 ANR。
- 是否增加审查时间。
- 是否降低测试缺失率。
- 是否让新成员更快完成低风险任务。

建议给 agent 参与的 PR 打标签，例如 `ai-assisted`。这样团队可以在一段时间后比较：

- AI 参与 PR 的平均审查轮数。
- AI 参与 PR 的回滚率。
- AI 参与 PR 的测试覆盖变化。
- AI 参与 PR 的线上问题率。

没有这些数据，团队很难判断 AI coding 到底是提升效率，还是把成本转移到审查和线上风险。

## 评测伦理与团队沟通

AI coding 评测不应该用来简单考核个人“会不会用 AI”。更合理的目标是识别工程系统短板。某个任务失败，可能是因为：

- 仓库文档缺失。
- 测试命令不稳定。
- 任务本身不可验证。
- prompt 模板不成熟。
- agent 工具权限过大或过小。

如果团队把失败都归因于个人，工程能力不会提升。评测应该推动团队补文档、补测试、补规则，而不是制造压力。

## 小规模试点评测方案

团队可以用两周做一个小规模试点：

第一周，准备任务集和规则。

- 选 8-10 个低中风险任务。
- 写清每个任务的输入、边界和验收。
- 准备高风险文件清单。
- 确定记录模板。
- 确定参与工程师和审查人。

第二周，执行和复盘。

- 每个任务由 agent 辅助完成。
- 记录耗时、轮数、测试、人工修改和审查意见。
- 合并前做常规人工审查。
- 周末集中复盘：哪些任务适合继续使用，哪些要降级。

试点不要一开始追求“节省多少人天”。更有价值的指标是：

- 哪些任务能稳定产出可审查 diff。
- 哪些任务经常越界。
- 哪些文档缺失影响 agent。
- 哪些测试命令不稳定。
- 哪些 prompt 模板需要沉淀。

AI coding 的早期收益通常来自流程清理，而不只是生成速度。

## 评测数据的最小字段

如果团队要持续度量，建议至少记录：

| 字段 | 说明 |
| --- | --- |
| task_id | 任务编号 |
| task_type | bug、test、refactor、doc、review |
| platform | iOS、Android、Flutter、RN |
| risk_level | green、yellow、red |
| tool | 使用的 agent 工具 |
| rounds | 交互轮数 |
| files_changed | 修改文件数量 |
| tests_run | 执行的验证命令 |
| tests_passed | 是否通过 |
| human_edits | 人工后续修改次数 |
| review_findings | 审查意见数量 |
| merged | 是否合并 |
| rollback | 是否回滚 |

字段不要一开始设计太复杂。能稳定记录并用于复盘，比设计完美指标更重要。

## 评测结果如何决策

评测结束后，可以把任务分成四类：

| 结果 | 决策 |
| --- | --- |
| 高成功、低风险 | 纳入常规 AI coding playbook |
| 高成功、高风险 | 保留人工确认点 |
| 低成功、低风险 | 优化 prompt、context 或测试 |
| 低成功、高风险 | 暂不自动化，只做只读分析 |

例如“给 ViewModel 补测试”可能是高成功、低风险；“修改权限配置”可能是高成功但高风险；“跨端桥接迁移”可能需要更强上下文和双平台验证；“支付流程重构”则应保持人工主导。

## 把评测接入日常研发

一次集中评测只能回答“在这组任务上表现如何”，还不能回答“长期使用是否稳定”。如果团队希望把 AI coding 纳入日常研发，应把评测数据接入现有流程。

第一步是在 Issue 或需求卡片中标记任务类型。比如 `ai-candidate:test`、`ai-candidate:refactor`、`ai-candidate:doc`、`ai-candidate:bugfix`。这能帮助团队识别哪些任务适合 agent 辅助，而不是每次靠工程师临时判断。

第二步是在 PR 模板中增加 AI 辅助字段：

```markdown
## AI 辅助说明

- 是否使用 AI coding agent：
- 使用在哪些阶段：分析 / 编码 / 测试 / 文档 / 审查
- Agent 运行过的命令：
- 人工修改过哪些输出：
- 需要审查人重点关注：
```

这个模板的目的不是给 AI 打标签，而是让审查人知道哪些内容需要重点看。比如 agent 修改了测试，审查人就要确认测试不是为了错误实现而改；agent 修改了权限文件，审查人就要确认产品文案、平台审核和隐私说明都被覆盖。

第三步是把 agent 参与的 PR 打标签。建议至少使用三个标签：

| 标签 | 含义 |
| --- | --- |
| `ai-assisted` | 有 agent 参与，但人类主导 |
| `ai-generated-tests` | 测试主要由 agent 补充 |
| `ai-risk-review` | 涉及权限、支付、登录、发布等高风险区域 |

这些标签能让团队在一个版本周期后回看数据：AI 参与的 PR 是否更容易被回滚，审查意见是否集中在某类问题，测试新增是否真的覆盖风险。

第四步是把评测结果写回 playbook。比如发现 agent 经常遗漏 Android 返回键，就更新 React Native 或 Android 任务模板；发现 agent 经常误改 `Info.plist`，就把它加入高风险清单；发现某个测试命令经常失败，就修复命令或把它从默认验证命令中移除。

只有完成这个闭环，AI coding 才会从个人技巧变成团队工程能力。

## 建立失败样本库

成功案例容易传播，失败样本更能提升工程质量。建议团队维护一个轻量失败样本库：

```text
docs/ai-coding/failures/
  2026-06-login-timeout-wrong-state.md
  2026-06-ios-permission-plist-risk.md
  2026-06-flutter-platform-channel-missing-android.md
```

每个失败样本记录：

- 任务背景。
- Agent 的错误做法。
- 为什么测试没有发现。
- 人类如何修正。
- 应更新的 prompt、context、harness 或 loop。

例如，一个失败样本可以这样写：

```markdown
# Flutter 平台通道修复遗漏 Android 侧

## 背景

Agent 根据 iOS 崩溃日志修复了 MethodChannel 参数解析。

## 错误

只修改了 Dart 和 iOS Swift 代码，没有同步 Android Kotlin 实现。

## 漏检原因

任务上下文只提供了 iOS 日志，验证命令只跑了 Dart 单测。

## 修正

补充 Android Kotlin 分支，增加双平台 contract test。

## Playbook 更新

Flutter platform channel 任务必须同时检查 Dart、iOS、Android 三侧文件，并运行平台接口 contract test。
```

失败样本库的价值在于把经验从“某个审查人记得”变成“整个团队和 agent 都能读取”。当新任务开始时，agent 可以先读取相关失败样本，避免重复犯错。

## 评测成熟度模型

团队可以用四个等级判断自己处在哪个阶段：

| 等级 | 状态 | 特征 |
| --- | --- | --- |
| M0 | 临时使用 | 个人凭经验使用 agent，没有记录 |
| M1 | 有任务模板 | 低风险任务有 prompt 和命令模板 |
| M2 | 有评测数据 | 记录成功率、测试、人工修改和风险 |
| M3 | 有闭环改进 | 评测结果持续更新 playbook、测试和权限边界 |

大多数团队不需要一开始追求 M3。更务实的路径是先把一个低风险任务做到 M1，例如“为 ViewModel 补测试”；再用 10-20 个真实任务积累数据，进入 M2；最后把复盘结果写回仓库，进入 M3。

这个成熟度模型也提醒团队：AI coding 的提升不只来自工具升级。即使模型能力不变，只要任务模板更清晰、上下文更准确、验证命令更稳定、失败样本更完整，团队使用 agent 的成功率也会明显提升。

## 本章小结

评测的目标不是证明 agent 很强，而是找出它适合哪些任务、不适合哪些任务，以及需要什么 harness 才能稳定发挥。移动端团队要用任务集、指标、质量分级和对照组，把 AI coding 从“感觉有效”推进到“可管理的工程能力”。
