# 第 13 章 AI Coding 效果评测

## 本章导读

AI coding 是否有效，不能只靠主观感觉。团队需要评估任务成功率、修改质量、测试覆盖、审查成本和回归风险。对于移动端团队，还要关注崩溃率、启动时间、包体、权限、兼容性和发版风险。

评测的目标不是证明某个 agent “很强”，而是回答三个工程问题：

1. 它适合做哪些任务？
2. 它需要什么上下文和工具才能稳定完成？
3. 它在哪些边界上必须由人类接管？

## 学习目标

- 建立 AI coding 评测指标。
- 知道如何设计移动端任务集。
- 能够区分效率提升和质量下降。
- 能够用评测结果改进 prompt、context、harness 和 loop。

## 评测维度

| 维度 | 指标 |
| --- | --- |
| 任务完成 | 完成率、一次通过率、平均轮数 |
| 代码质量 | diff 大小、架构一致性、重复代码、复杂度 |
| 测试验证 | 测试是否新增、测试是否通过、覆盖风险点 |
| 审查成本 | 人工修改次数、审查意见数量 |
| 安全合规 | 是否触碰敏感文件、是否泄露秘密 |
| 移动端质量 | 崩溃率、启动时间、包体、权限、兼容性 |

单一指标会误导团队。例如“代码生成速度提升 50%”没有意义，如果它同时让审查时间翻倍、线上回归增加。AI coding 的评测必须同时看效率和质量。

## 任务集设计

移动端 AI coding 评测任务可以包括：

- 修复一个有堆栈的崩溃。
- 给 ViewModel 补测试。
- 拆分一个复杂 UI 组件。
- 迁移一个网络错误模型。
- 修复一个跨端平台差异。
- 生成一份发版检查文档。
- 修复一个构建失败。
- 补齐一个本地化遗漏。
- 检查一次权限变更风险。

任务集要覆盖不同难度，而不是只挑 agent 擅长的任务。建议分为三档：

| 难度 | 特征 | 示例 |
| --- | --- | --- |
| L1 | 单文件或少量文件，可单测验证 | 表单校验、mapper 测试 |
| L2 | 跨 3-8 个文件，需要理解模块边界 | ViewModel 状态重构、组件拆分 |
| L3 | 跨平台或高风险，需要人工确认 | 权限、桥接、依赖升级、发版检查 |

如果一个 agent 只在 L1 表现好，仍然有价值，但不能宣传成能自动完成复杂工程任务。

## 评测记录模板

建议每个评测任务记录：

```yaml
task_id: mobile-ai-coding-001
title: 修复 ProfileViewModel 弱网错误提示
level: L2
platform: iOS
allowed_paths:
  - Sources/Profile/**
  - Tests/ProfileTests/**
forbidden_paths:
  - ios/Signing/**
  - Info.plist
commands:
  - xcodebuild test ...
expected:
  - timeout 显示弱网提示
  - offline 显示离线提示
  - serverError 显示服务端错误
result:
  completed: true
  test_passed: true
  human_edits: 1
  review_findings: 2
  touched_risky_files: false
```

这种结构化记录能帮助团队比较不同工具、不同 prompt、不同上下文策略的效果。

## 质量分级

可以把 agent 输出分成五级：

| 级别 | 描述 | 处理方式 |
| --- | --- | --- |
| A | 直接可合并，测试和说明完整 | 常规审查 |
| B | 小问题，人工少量修改即可 | 记录改进点 |
| C | 方向正确，但需要明显返工 | 优化上下文和 prompt |
| D | 方案错误或破坏边界 | 降级任务类型 |
| E | 触碰安全红线 | 禁止该类自动化 |

这个分级比“成功/失败”更有信息量。C 级任务并不等于没有价值，它说明 agent 可能需要更好的上下文或更小的任务切分。

## 评测中的对照组

如果团队想知道 AI coding 是否真的提升效率，需要有对照：

- 人工完成同类任务的平均时间。
- 人工审查同类 PR 的平均意见数。
- 过去三个月同类 bug 的回归率。
- 现有测试覆盖和 CI 时长。

没有对照时，团队很容易被“生成速度”影响判断。真正要看的，是从任务开始到可合并 PR 的总时间，以及合并后的质量。

## 移动端特有指标

移动端项目还要关注：

- 启动时间是否变化。
- 包体是否变化。
- 崩溃率是否变化。
- ANR 或卡顿是否变化。
- 权限声明是否变化。
- 离线、弱网、低电量场景是否受影响。
- 多语言和暗黑模式是否受影响。

这些指标不一定每个任务都要测，但如果任务涉及相关模块，就必须纳入验收。

## 用评测改进四类工程能力

评测结果要反哺前面四类能力：

- Prompt Engineering：任务描述是否清楚，禁止事项是否足够。
- Context Engineering：是否提供了正确文件、架构说明和错误现场。
- Harness Engineering：工具、命令、权限和测试是否足够。
- Loop Engineering：是否有清晰的停止条件和人工接管点。

例如某类任务经常失败，不一定是模型能力问题。可能是 prompt 没有限定范围，可能是上下文缺少项目地图，也可能是测试命令不稳定。

## 本章小结

评测的目标不是证明 agent 很强，而是找出它适合哪些任务、不适合哪些任务，以及需要什么 harness 才能稳定发挥。移动端团队要用任务集、指标、质量分级和对照组，把 AI coding 从“感觉有效”推进到“可管理的工程能力”。
