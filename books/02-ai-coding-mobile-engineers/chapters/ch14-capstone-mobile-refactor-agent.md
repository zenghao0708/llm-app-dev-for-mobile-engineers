# 第 14 章 综合项目：移动端重构 Agent 工作流

## 本章导读

本章把前面的四类工程能力串起来，设计一个可复用的移动端重构 agent 工作流。目标不是展示某个工具的演示效果，而是形成一套团队可执行的流程。读者可以把本章当作一次小型项目演练：从任务定义、上下文准备、prompt 设计、harness 配置、循环验证到 PR 交付。

本章选择的综合项目是“统一网络错误处理”。这是移动端项目中非常常见的问题：错误码散落在多个页面，文案不一致，埋点不完整，测试覆盖不足。它足够真实，也足够可控，适合作为 AI coding 工作流训练题。

## 项目目标

选择一个移动端项目中的真实问题：网络错误处理分散在多个页面，导致 UI 文案不一致、埋点不完整、测试覆盖不足。我们希望 agent 帮助完成一次受控重构。

目标不是“一次改完所有页面”，而是建立统一错误模型，并迁移少量低风险页面，证明流程可行。

## 初始问题

假设项目中有三类重复代码：

- 每个页面都把 HTTP status 映射成自己的文案。
- 弱网、超时、服务端错误的 UI 表达不一致。
- 埋点字段在不同页面命名不同。

这会导致几个问题：

- 产品体验不一致。
- 测试难以覆盖。
- 新错误码接入成本高。
- 线上问题定位困难。
- 多语言文案容易遗漏。

这是典型的适合 agent 协助但不适合 agent 一次性全自动完成的任务。

## 工作流阶段

1. 任务定义：说明重构目标和禁止事项。
2. 上下文收集：读取网络层、错误模型、页面调用方和测试。
3. 计划生成：列出最小修改路径。
4. 第一轮修改：只抽取统一错误模型，不改 UI。
5. 验证：运行单元测试和静态检查。
6. 第二轮修改：迁移两个低风险页面。
7. 人工审查：确认文案、埋点和平台行为。
8. 扩展迁移：按页面批次推进。
9. 交付：生成 PR 描述、测试结果和风险说明。

每个阶段都应该有明确输入和输出。没有输出的阶段，等于没有审计。

## 项目上下文包

在开始前，准备一个上下文包：

```text
docs/ai-coding/network-error-refactor.md

目标：
- 建立统一 AppError。
- 统一 timeout/offline/serverError/businessError。
- 先迁移 Profile 和 Settings 两个页面。

禁止：
- 不修改登录、支付、风控。
- 不修改接口协议。
- 不修改埋点平台 SDK。
- 不修改发布配置。

验证命令：
- iOS: xcodebuild test ...
- Android: ./gradlew testDebugUnitTest
- Flutter: flutter test
- RN: yarn test

人工确认：
- 文案。
- 埋点字段。
- 弱网真实体验。
```

这个文件就是 Context Engineering 的产物。它让 agent 不必从零猜项目规则。

## Prompt 模板

```text
你是移动端重构助手。任务是把网络错误处理迁移到统一错误模型。

约束：
- 不修改签名、发布、证书和生产开关。
- 不改变登录、支付和风控逻辑。
- 每轮最多迁移两个页面。
- 每轮必须运行指定测试。
- 如果需要改高风险文件，先停止并说明原因。

本轮目标：
- 只新增统一 AppError。
- 只修改网络层错误映射。
- 为 AppError 增加单元测试。
- 不迁移任何页面。

输出：
- 本轮计划。
- 修改文件列表。
- 测试命令和结果。
- 仍需人工审查的点。
```

注意本轮目标非常小。综合项目不是让 agent 一次完成所有目标，而是让它在可收敛循环中推进。

## 第一轮：统一错误模型

第一轮只做统一错误模型。合理输出应包括：

- 新增 `AppError` 或等价类型。
- 定义 timeout、offline、serverError、businessError、unknown。
- 建立从底层网络错误到 `AppError` 的 mapper。
- 增加 mapper 单元测试。
- 不修改页面 UI。

验收清单：

- timeout 有测试。
- offline 有测试。
- 5xx 有测试。
- 业务错误码有测试。
- unknown 有兜底。
- 旧调用方仍编译通过。

如果 agent 在第一轮主动迁移了多个页面，就说明 prompt 边界或 loop 控制不够严格。

## 第二轮：迁移低风险页面

第二轮选择两个低风险页面，例如 Profile 和 Settings。选择标准：

- 页面逻辑相对独立。
- 没有支付、登录、风控。
- 测试已有基础。
- 产品文案容易确认。
- 回滚简单。

本轮 prompt 应写明：

```text
本轮只迁移 Profile 和 Settings 两个页面。
不要迁移其他页面。
如果发现其他页面有重复逻辑，只在最终输出中列为后续任务。
```

这个限制非常重要。Agent 很容易“顺手”迁移更多调用方，但移动端重构最怕无边界扩散。

## 第三轮：评审与扩展

前两轮通过后，进入人工评审：

- 产品确认错误文案。
- 数据确认埋点字段。
- 测试确认弱网和超时体验。
- 平台负责人确认没有触碰权限和发布配置。
- 研发确认下一批迁移范围。

评审通过后，才按批次迁移更多页面。每批仍然最多 2-3 个页面，并保留独立测试结果。

## PR 描述模板

```markdown
## 背景

网络错误处理散落在多个页面，本 PR 建立统一 AppError，并迁移 Profile/Settings 页面。

## 修改范围

- 新增 AppError。
- 新增网络错误 mapper。
- 迁移 Profile 错误展示。
- 迁移 Settings 错误展示。
- 新增对应单元测试。

## 验证

- [x] 单元测试通过：...
- [x] lint 通过：...
- [ ] 真机弱网体验：待测试确认

## 风险

- 文案需产品确认。
- 埋点字段需数据同学确认。
- 其他页面尚未迁移。

## 回滚

可回滚本 PR；旧页面逻辑未批量删除。
```

PR 描述是 agent 交付的一部分，不是附加工作。它能显著降低审查成本。

## 验收清单

- 统一错误模型有单元测试。
- 页面迁移有回归测试或快照验证。
- 文案进入本地化资源。
- 埋点字段不丢失。
- 弱网和超时有兜底。
- PR 描述包含迁移范围和剩余页面。
- 高风险文件没有被自动修改。
- 未执行的验证项被明确列出。

## 复盘：四类工程能力如何协同

这个综合项目中，四类能力分别发挥作用：

- Prompt Engineering：定义本轮目标、禁止事项和输出格式。
- Context Engineering：准备错误模型、页面、测试、文案和埋点上下文。
- Harness Engineering：提供测试命令、风险文件检查和 PR 模板。
- Loop Engineering：按“计划-修改-验证-审查-下一轮”推进。

如果项目失败，可以按这四类能力定位原因。不是所有失败都应该归因于模型本身。

## 可运行示例工程建议

正式配套代码可以准备一个小型示例工程，而不是只给伪代码。示例工程应满足：

- 有一个最小移动端后端或共享逻辑模块。
- 有错误模型、repository、ViewModel 或等价层。
- 有 3-5 个页面调用方。
- 有单元测试。
- 有一个故意保留的重复错误处理问题。
- 有脚本可以运行测试和生成报告。

如果为了跨平台简化，可以先用 Python 或 TypeScript 模拟移动端业务层，再在章节中说明如何映射到 iOS/Android。关键是代码真实可运行，测试真实可执行，不能只是演示片段。

示例目录可以这样设计：

```text
examples/02-ai-coding-mobile-refactor/
  README.md
  pyproject.toml
  src/
    app_error.py
    network_client.py
    profile_view_model.py
    settings_view_model.py
  tests/
    test_error_mapper.py
    test_profile_view_model.py
    test_settings_view_model.py
  docs/
    ai-task.md
    high-risk-files.md
    ci-equivalent.md
```

这个示例的目标不是替代真实移动端工程，而是让读者能在本地体验 agent 工作流：先跑失败测试，再让 agent 修复，再观察 diff 和测试结果。

## 分批迁移表

综合项目中，建议维护一张迁移表：

| 页面 | 风险 | 测试现状 | 迁移批次 | 状态 |
| --- | --- | --- | --- | --- |
| Profile | 低 | 有 ViewModel 测试 | 第一批 | 已迁移 |
| Settings | 低 | 有基础测试 | 第一批 | 已迁移 |
| Order | 中 | 缺少超时测试 | 第二批 | 待补测试 |
| Payment | 高 | 涉及风控 | 人工主导 | 暂停 |
| Login | 高 | 涉及账号状态 | 人工主导 | 暂停 |

这张表能防止 agent 忘记边界，也能让团队看到整体进度。长任务中，迁移表比一段自然语言总结更可靠。

## 失败演练

综合项目还应该包含失败演练。让读者看到 agent 不一定一次成功，反而更接近真实工程。

例如：

1. Agent 第一轮迁移 Profile 页面。
2. 单元测试失败，提示 offline 文案不一致。
3. Agent 读取失败输出后发现本地化 key 缺失。
4. Agent 补充 key，但没有更新英文文案。
5. 人工审查指出多语言缺口。
6. Agent 更新文案并补测试。

这段演练能说明：Loop 的价值不是保证一次成功，而是让失败可见、可定位、可修复。

## 综合项目的交付标准

本章项目完成时，应满足：

- 读者能理解任务背景。
- 能看到 agent 任务模板。
- 能看到分批迁移策略。
- 能看到测试和验证命令。
- 能看到 PR 描述和风险说明。
- 能看到失败恢复方式。
- 能把流程迁移到自己的 iOS/Android/跨端项目。

如果后续继续完善配套代码，建议把本章和示例工程绑定：章节讲流程，代码给读者动手验证。

## 示例工程任务单

为了让读者真实练习，可以把综合项目拆成 6 个任务单。

### 任务 1：建立项目地图

目标：让 agent 只读分析示例工程，输出模块地图。

验收：

- 能说明网络层、ViewModel、测试目录。
- 能列出错误处理重复点。
- 不修改代码。

### 任务 2：新增统一错误模型

目标：新增 `AppError` 和 mapper。

验收：

- timeout、offline、serverError、businessError、unknown 都有测试。
- 旧调用方仍通过测试。
- 不迁移页面。

### 任务 3：迁移 Profile 页面

目标：只迁移 Profile。

验收：

- ProfileViewModel 测试通过。
- UI 文案 key 不硬编码。
- 不修改 Settings、Order、Payment。

### 任务 4：迁移 Settings 页面

目标：迁移第二个低风险页面。

验收：

- SettingsViewModel 测试通过。
- 与 Profile 使用同一错误模型。
- 记录剩余页面。

### 任务 5：审查高风险页面

目标：只读分析 Order、Payment、Login 是否适合迁移。

验收：

- 输出风险原因。
- 标出必须人工确认的字段。
- 不修改代码。

### 任务 6：生成 PR 材料

目标：生成可审查交付件。

验收：

- PR 描述。
- 测试结果。
- 风险清单。
- 回滚方式。
- 后续迁移表。

这些任务单可以放进示例工程 README，让读者按顺序练习。

## 读者练习：让 agent 失败一次

建议读者刻意设计一次失败练习。例如给 agent 一个不完整 prompt：

```text
帮我统一错误处理。
```

观察 agent 是否会：

- 一次修改太多文件。
- 忽略测试。
- 迁移高风险页面。
- 修改文案或埋点。
- 不输出未验证项。

然后再用本章模板重写 prompt，对比两次结果。这个练习能让读者直观看到 Prompt、Context、Harness 和 Loop 的价值。好的工程方法往往不是让成功看起来更漂亮，而是让失败更可控。

## 从示例项目迁移到真实项目

读者完成示例工程后，不应立刻在真实项目中做大重构。建议按下面路线迁移：

1. 选择一个低风险模块。
2. 只让 agent 做只读影响面分析。
3. 人工确认文件范围和测试命令。
4. 让 agent 补一个测试。
5. 让 agent 做最小修复。
6. 提交 PR，按团队流程审查。
7. 复盘 prompt、context、harness 和 loop。

真实项目的复杂度来自历史代码、业务规则、团队习惯和发版节奏。示例工程能训练方法，但不能替代审慎迁移。

## 本章小结

> **综合项目重点**：综合项目的关键不是让 agent 一次做完所有重构，而是把大任务拆成可验证的小循环。人类负责边界、优先级和审查，agent 负责执行、反馈和重复性工作。移动端团队真正需要沉淀的，是这种可复用工作流，而不是某一次提示词的偶然成功。
