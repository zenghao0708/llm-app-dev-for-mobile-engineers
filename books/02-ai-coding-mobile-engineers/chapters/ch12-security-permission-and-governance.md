# 第 12 章 权限、安全与团队治理

## 本章导读

AI coding agent 可以读文件、改代码、运行命令，意味着它必须被纳入安全治理。越强大的 agent，越需要明确权限、审计和回滚机制。移动端项目尤其敏感，因为仓库里可能包含签名配置、权限声明、隐私文案、埋点逻辑、支付登录代码和发版脚本。

安全不是阻止团队使用 AI coding，而是让自动化在可信边界内运行。本章把安全拆成三层：任务权限、数据边界和团队治理。

## 学习目标

- 知道 AI coding 中常见安全风险。
- 能够设计移动端仓库的权限分层。
- 掌握团队治理和代码审查要求。
- 能够建立 agent 任务的审计和回滚机制。

## 风险清单

移动端 AI coding 常见风险包括：

- 泄露 API Key、证书、token、日志和用户数据。
- 修改签名、发布、生产配置。
- 引入未审查依赖。
- 执行危险命令。
- 生成不符合隐私合规的权限说明。
- 在不了解业务规则的情况下修改支付、登录、风控代码。
- 把测试假数据写入生产路径。
- 把调试开关、mock server 或日志级别带入发版分支。

这些风险不是理论问题。Agent 的优势是快速读写和执行，风险也来自同一件事：它可以在短时间内修改许多文件。治理的目标是把速度限制在可审查范围内。

## 权限分层

| 权限级别 | 允许动作 | 适用任务 |
| --- | --- | --- |
| 只读 | 搜索、阅读、解释 | 代码理解、方案评审 |
| 受限写 | 修改指定目录 | 小 bug、测试、文档 |
| 受限执行 | 运行白名单命令 | 测试和构建 |
| 人工确认 | 依赖、权限、签名、发布 | 高风险变更 |
| 禁止 | 密钥、生产证书、删除历史 | 不应交给 agent |

默认策略应当是：新任务从只读开始，只有当任务边界清晰、验收命令明确、风险文件清单明确时，才进入受限写和受限执行。

## 移动端高风险路径

建议团队把高风险路径写进仓库：

```text
docs/ai-coding/high-risk-files.md

iOS:
- ios/*.xcodeproj/**
- ios/*.xcworkspace/**
- **/project.pbxproj
- **/*.entitlements
- **/Info.plist
- signing/**

Android:
- **/AndroidManifest.xml
- **/build.gradle
- **/gradle.properties
- **/proguard-rules.pro
- **/*.keystore

通用：
- .env
- secrets/**
- release/**
- payment/**
- auth/**
- risk-control/**
```

这份清单不是为了让 agent 永远不读这些文件，而是为了阻止它在没有确认时修改它们。

## 数据边界与脱敏

移动端调试常需要日志、崩溃、网络请求和用户反馈。这些数据进入 agent 前必须脱敏。最低要求包括：

- 删除真实 token、cookie、Authorization header。
- 删除手机号、邮箱、身份证号、精确地址。
- 删除真实用户 ID 或做不可逆映射。
- 删除生产证书、keystore、profile。
- 保留错误码、调用链、时间顺序、设备型号和系统版本。

如果团队使用 MCP 或内部工具提供日志，应在工具层默认脱敏，而不是要求每个工程师手动处理。手动复制日志最容易出错。

## 依赖与供应链风险

Agent 经常会建议“新增一个库解决问题”。移动端项目要格外谨慎，因为一个依赖可能影响包体、启动时间、隐私合规、构建速度和平台兼容性。

依赖变更应满足：

- 有明确业务必要性。
- 有维护状态和许可证检查。
- 有替代方案比较。
- 有包体和方法数影响说明。
- 有回滚方式。
- 必须人工审查。

Prompt 中可以直接写：

```text
不允许新增依赖。若你认为必须新增依赖，只能在最终输出中说明理由和备选方案，不要修改依赖文件。
```

这句话能避免很多不必要的变更。

## 审计与追踪

Agent 任务结束后，应保留最小审计信息：

- 任务目标。
- 输入上下文摘要。
- 修改文件列表。
- 执行命令和结果。
- 未执行的验证项。
- 高风险文件是否被触碰。
- 人工审查结论。

这些信息可以放在 PR 描述中，也可以由工具自动生成。它的作用不是追责，而是让团队知道“这个变更是如何产生和验证的”。

## 团队治理

团队需要定义：

- 哪些任务可以直接由 agent 做。
- 哪些任务必须开 PR。
- 哪些文件变更必须人工审查。
- 生成代码是否需要额外测试。
- agent 输出是否纳入审计。
- 是否允许 agent 访问外部系统。
- 是否允许 agent 在本地执行脚本。

建议把任务分为三类：

| 类型 | 示例 | 规则 |
| --- | --- | --- |
| 绿色任务 | 文档、测试、局部逻辑 | 可由 agent 修改，常规审查 |
| 黄色任务 | 小重构、依赖小升级、桥接改动 | 必须人工确认计划和结果 |
| 红色任务 | 签名、发布、支付、隐私、风控 | 只读分析或人工主导 |

这比简单地说“可以用 AI”或“不准用 AI”更可执行。

## 回滚策略

Agent 参与的改动必须容易回滚。实践上可以采用：

- 小提交。
- 小 PR。
- 每轮最多改一个模块。
- 功能开关保护。
- 保留旧路径一段时间。
- 分批迁移调用方。
- 在 PR 描述中写清楚回滚方式。

对于移动端发版，回滚不一定等于马上撤回 App。更现实的策略是服务端开关、灰度、热修配置、关闭入口或回退远程配置。因此 agent 修改代码时要说明是否涉及这些控制面。

## 安全检查清单

建议把下面的检查清单放进 PR 模板或 agent 任务结束输出中：

| 检查项 | 问题 | 处理 |
| --- | --- | --- |
| Secrets | 是否新增 `.env`、token、证书、keystore | 必须移除 |
| 权限 | 是否修改 Info.plist 或 Manifest 权限 | 人工确认 |
| 依赖 | 是否新增三方库 | 许可证、包体、维护状态审查 |
| 发布 | 是否修改 release、signing、upload 脚本 | 默认禁止 |
| 用户数据 | 日志、埋点是否包含敏感信息 | 脱敏或删除 |
| 高风险业务 | 是否触碰登录、支付、风控 | 业务负责人审查 |
| 测试 | 是否运行相关验证 | 写明结果和缺口 |

这个清单应该成为团队默认流程，而不是安全团队临时提醒。

## 权限变更案例

假设一个需求要求接入相册选择。Agent 可能会修改 iOS `Info.plist`、Android Manifest、Flutter/RN 插件配置和用户文案。这个任务表面是功能开发，实质是权限治理。

正确流程如下：

1. 先让 agent 只读分析需要哪些权限。
2. 人工确认权限必要性和产品文案。
3. agent 修改配置和代码。
4. 人工检查 iOS/Android 是否一致。
5. 真机验证首次授权、拒绝、再次请求、系统设置返回。
6. PR 中说明隐私影响和未验证平台。

Prompt 应写明：

```text
如果需要新增或修改系统权限，请先停止并说明：
- 权限名称。
- 使用场景。
- 用户可见文案。
- iOS/Android 配置位置。
- 是否有替代方案。
不要直接修改权限文件。
```

这样做不是降低效率，而是避免 agent 在不了解审核和隐私承诺的情况下自动改配置。

## 本地凭据与公开仓库

当书稿、示例代码或团队模板要放到公开仓库时，secret hygiene 必须成为硬门槛。公开仓库中允许出现：

- `.env.example`。
- 占位符 API key。
- mock provider。
- 本地配置说明。
- 脱敏日志样例。

不允许出现：

- 真实 API key。
- 真实 token。
- 生产接口私密路径。
- keystore、profile、证书。
- 用户日志和内部工单原文。

Agent 参与提交前，应强制执行 staged 文件检查，而不是只看工作区。很多泄露发生在“本地文件不显眼，但已经被暂存”的阶段。

## 治理落地路线

团队可以按三阶段落地治理：

第一阶段，规则可见。把高风险路径、禁止命令、敏感数据处理写进仓库。

第二阶段，检查自动化。提交前扫描 secrets、权限文件、依赖文件和发布脚本。

第三阶段，流程化。不同任务类型走不同权限：绿色任务自动化，黄色任务人工确认，红色任务只读分析。

治理不是一次写完规则，而是把每次事故、误改、审查意见回写到规则中。

## 本章小结

安全不是阻止使用 AI coding，而是给它建立边界。没有边界的自动化最终会变成不可控风险。移动端团队要把高风险文件、敏感数据、依赖变更、执行命令和审计信息纳入统一治理，让 agent 在明确权限内提高效率。
