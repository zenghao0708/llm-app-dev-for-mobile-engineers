# 第 7 章 Loop Engineering：设计可收敛的反馈循环

## 本章导读

AI coding agent 的强项不是一次生成完美答案，而是在反馈中修正。Loop Engineering 关注如何设计一个能收敛的循环：理解任务、采取行动、验证结果、读取反馈、更新计划，再继续。

OpenAI Codex 文档明确描述了这种循环：提交 Prompt 后，Codex 会调用模型并执行模型输出指示的动作，例如读取文件、编辑文件和调用工具，直到任务完成或被取消。Anthropic 的 agent SDK 文章也把常见循环概括为 gather context、take action、verify work、repeat。Loop Engineering 要解决的问题是：这个循环怎样才不会失控、不会空转、不会过早结束。

## 学习目标

- 理解 agent loop 的基本结构。
- 能够为 bug 修复、重构和测试补充设计反馈循环。
- 掌握停止条件、检查点和失败恢复。
- 学会在移动端任务中设计人类介入点。

## 7.1 基本循环

一个可控的 AI coding loop 可以写成：

```text
理解任务 -> 定位文件 -> 制定小计划 -> 修改代码 -> 运行验证 -> 读取失败 -> 修正 -> 总结交付
```

关键不是循环次数越多越好，而是每轮都要减少不确定性。一次好的循环应该回答：

- 我刚刚做了什么？
- 证据是什么？
- 失败在哪里？
- 下一步最小动作是什么？
- 是否需要人类确认？

如果 agent 每轮只是继续改代码，但没有新的证据，这不是循环，是漂移。

## 7.2 Bug 修复循环

移动端 bug 修复适合使用“假设-验证”循环。

```text
读取现场 -> 建立假设 -> 写测试或添加检查 -> 最小修复 -> 运行验证 -> 总结风险
```

以弱网重试崩溃为例：

1. 读取崩溃堆栈和复现步骤。
2. 定位 ViewModel、Repository 和网络层。
3. 做最小假设：取消后回调仍更新 UI，或重复回调导致状态冲突。
4. 写一个失败测试。
5. 修改代码。
6. 运行测试。
7. 如果失败，读取失败输出并缩小范围。
8. 通过后总结风险：真机弱网仍需人工验证。

这个循环避免 agent 直接“猜修复”。它要求每一步都有证据。

## 7.3 重构循环

重构循环的核心是保持可回滚。

```text
识别重复 -> 确认边界 -> 加保护测试 -> 小步重构 -> 运行验证 -> 提交检查点
```

移动端重构不要让 agent 一次性迁移全部页面。更稳的方式是：

- 第 1 轮：抽取统一模型和测试。
- 第 2 轮：迁移一个低风险页面。
- 第 3 轮：迁移第二个页面，并检查文案和埋点。
- 第 4 轮：总结迁移 Playbook，再批量推进。

每轮都应该能独立审查，独立回滚。

## 7.4 测试补充循环

测试补充不是让 agent “多写几个 test”。更好的循环是：

1. 阅读生产代码和现有测试风格。
2. 列出未覆盖风险。
3. 先写一个最有价值的测试。
4. 运行测试。
5. 如果测试很脆弱，调整测试设计，而不是改生产代码迎合测试。
6. 输出覆盖了什么、没覆盖什么。

移动端测试常见问题是 agent 写出过度 mock 的测试。这样的测试通过了，但没有覆盖真实风险。Loop 中应要求 agent 解释每个测试对应的用户场景或失败模式。

## 7.5 停止条件

Agent loop 必须有停止条件：

- 测试通过。
- 达到最大步数。
- 需要人类确认。
- 涉及高风险文件。
- 发现需求不明确。
- 多次修复后仍失败，需要升级为人工调试。

可以在 Prompt 中写明：

```text
如果连续两次运行 LoginTests 仍失败，请停止，不要继续猜修。
输出你已经验证过的事实、失败日志摘要和你建议人类检查的点。
```

停止不是失败。对 agent 来说，知道何时停止是可控性的表现。

## 7.6 人类介入点

> **审查重点**：Loop Engineering 不是追求全自动，而是设计正确的人类介入点。

移动端常见介入点：

- 修改权限配置。
- 修改登录、支付、风控。
- 引入新依赖。
- 修改签名或发布配置。
- UI 视觉和交互体验。
- 真机弱网、通知、相机、定位等设备能力。
- 性能和包体变化。

在这些位置，agent 应该停止并说明原因，而不是继续执行。

## 7.7 多 Agent 循环

随着 Codex app 等工具支持多 agent 并行，Loop Engineering 还要考虑多 agent 协作。并行不是让多个 agent 改同一个文件，而是拆分职责：

- Agent A：阅读 Issue 和仓库，生成计划。
- Agent B：实现低风险代码修改。
- Agent C：补测试。
- Agent D：做代码审查或文档更新。

移动端项目中，多 agent 并行必须避免冲突：

- 不让两个 agent 修改同一模块。
- 每个 agent 有明确文件范围。
- 每个 agent 输出验证结果。
- 最终由人类合并和审查。

如果没有文件范围和交付协议，多 agent 只会放大冲突。

## 7.8 失败恢复

Loop 失败后要能恢复。恢复机制包括：

- Git 检查点。
- 进度文件。
- 失败日志摘要。
- 回滚说明。
- 最小复现命令。

例如：

```text
本轮失败：
- 修改 LoginViewModel 后 LoginTests 仍失败。
- 失败断言：retryState expected idle but got loading。

已确认事实：
- retry callback 会在 cancel 后触发。
- ErrorMapper 没有参与本次失败。

建议下一步：
- 检查 NetworkSession cancel token 是否正确清理。
- 暂停迁移，不继续改 UI。
```

这种输出能让人类或下一轮 agent 快速接手。

## 7.9 循环质量检查

| 检查项 | 好循环 | 差循环 |
| --- | --- | --- |
| 每轮目标 | 小而明确 | 模糊扩大 |
| 证据 | 有测试、日志、diff | 只有解释 |
| 修改范围 | 可审查 | 大量文件混改 |
| 失败处理 | 读取失败并缩小范围 | 盲目重试 |
| 停止条件 | 明确 | 无限继续 |
| 人类介入 | 高风险点暂停 | 自动越权 |

## 7.10 设计可收敛循环

一个可收敛循环必须同时满足三点：

第一，每轮目标足够小。目标越小，反馈越明确。移动端重构中，“迁移两个页面”比“迁移所有页面”更容易收敛。

第二，每轮都有证据。证据可以是测试、构建、lint、截图、日志或人工确认。没有证据的循环只是在反复生成解释。

第三，每轮都有停止条件。比如最多尝试两次修复同一测试失败；如果涉及权限或支付，立即停止；如果测试环境缺失，输出阻塞原因而不是继续猜。

可以把一个循环写成模板：

```text
本轮目标：
- 迁移 Profile 页面错误处理。

允许：
- 修改 Profile 模块和测试。

验证：
- 运行 ProfileViewModelTests。

停止：
- 如果需要修改网络协议，停止。
- 如果测试失败两次仍无法定位，停止并输出事实。
```

这种模板让 agent 不会无限扩展任务，也让人类知道何时介入。

## 7.11 循环中的角色切换

Agent 在不同轮次中可以扮演不同角色：

- 分析者：只读仓库，输出影响面。
- 计划者：拆分任务，设计验证。
- 执行者：修改代码并运行命令。
- 审查者：检查 diff、风险和测试缺口。
- 记录者：更新进度文件和 PR 描述。

不要让同一轮同时承担所有角色。尤其是在高风险任务中，应先让 agent 只读分析，再由人类确认计划，然后进入受限执行。角色切换越清晰，循环越容易控制。

## 7.12 复盘循环质量

任务结束后，团队应该复盘循环质量，而不仅是看代码是否合并。

复盘问题包括：

- 初始 prompt 是否足够明确？
- 上下文是否缺失或污染？
- 测试命令是否准确？
- agent 是否触碰了禁止文件？
- 人工介入是否太晚？
- 是否有重复失败？
- 哪些规则应该沉淀到 skill 或 AGENTS.md？

这类复盘能把一次任务经验变成团队资产。AI coding 的能力提升不是靠记住某次对话，而是靠把失败模式写回流程。

## 7.13 移动端循环案例：从崩溃到补测试

下面看一个完整循环。任务是修复 Android 商品详情页在弱网下偶现 loading 不结束的问题。

第一轮，agent 只读分析：

```text
请先不要修改代码。阅读 ProductDetailViewModel、ProductRepository、NetworkErrorMapper 和现有测试。
输出：
- 加载状态从触发到结束的路径。
- 异常路径是否会设置 loading=false。
- 是否已有 timeout/offline 测试。
```

这一轮的目标不是修复，而是建立事实。Agent 可能输出：正常路径会结束 loading，HTTP 5xx 会结束 loading，但 timeout 抛出的异常没有被 mapper 捕获，导致状态停留在 loading。

第二轮，写失败测试：

```text
请只添加一个失败测试，模拟 repository timeout。
期望：UI state 进入 error，并且 loading=false。
不要修改生产代码。
```

第三轮，最小修复：

```text
请修复 timeout 未结束 loading 的问题。
约束：
- 只修改 ProductDetailViewModel 或错误 mapper。
- 不改 UI 文案。
- 不新增依赖。
验证：
- 运行 ProductDetailViewModelTest。
```

第四轮，复盘：

```text
请输出：
- 根因。
- 新增测试。
- 修改文件。
- 未覆盖场景。
- 是否建议把 timeout 处理迁移到统一错误模型。
```

> **重点提示**：这个案例体现了 Loop Engineering 的核心：每轮只做一件事，并且每轮都有证据。即使第三轮失败，也能回到第二轮的测试和第一轮的事实，不会在猜测中扩散。

## 7.14 循环反模式

移动端 AI coding 中常见循环反模式包括：

| 反模式 | 表现 | 结果 |
| --- | --- | --- |
| 解释循环 | agent 每轮都解释很多，但不产生测试或 diff | 任务看似推进，实际没有证据 |
| 大爆炸循环 | 一次修改几十个文件 | 难以审查和回滚 |
| 测试追逐 | 测试失败后不断改测试 | 真实问题被掩盖 |
| 越权循环 | 为了修 bug 修改权限、签名、依赖 | 风险被放大 |
| 忽略失败循环 | 命令失败但 agent 继续宣布完成 | 交付不可用 |
| 无交接循环 | 多轮后没有进度摘要 | 下一轮无法接手 |

解决这些反模式，要靠明确规则，而不是期待 agent 自觉。比如规定“测试失败后必须先解释失败原因，不得直接修改测试期望”；规定“单轮修改超过 8 个文件必须停止并重新拆分”；规定“命中高风险文件必须人工确认”。

## 7.15 循环与产品节奏

移动端开发有发版节奏。Loop Engineering 不能只看代码，还要看版本阶段。

| 阶段 | Agent 适合做什么 | 不适合做什么 |
| --- | --- | --- |
| 需求早期 | 方案比较、影响面分析、任务拆分 | 直接批量改代码 |
| 开发中 | 小任务实现、补测试、文档同步 | 无边界重构 |
| 提测前 | 测试补齐、风险清单、PR 描述 | 引入新依赖 |
| 灰度中 | 日志分析、崩溃归类、回归定位 | 修改核心逻辑后直接发版 |
| 发版后 | 事故复盘、监控摘要、补充测试 | 自动修改生产配置 |

同一个 agent 能力，在不同阶段风险不同。开发中可以让 agent 迁移低风险页面；发版前同样的迁移可能就不应该做。人类工程师需要把产品节奏写进任务约束。

## 本章小结

Loop Engineering 的目标是让 agent 在复杂任务中逐步收敛，而不是无限尝试。好的循环会保留证据、暴露风险，并让人类在关键点介入。到这里，本书的四类核心能力已经完整展开：Prompt 决定任务表达，Context 决定信息质量，Harness 决定执行边界，Loop 决定能否收敛。下一篇会把这些能力落到 iOS、Android、Flutter 和 React Native 的真实场景中。
