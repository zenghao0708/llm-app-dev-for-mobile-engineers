# 第 2 章 Agentic Coding 的能力地图

## 本章导读

Agentic coding 工具看起来都能“帮你写代码”，但真正需要比较的不是界面，而是工程能力边界。一个工具能否稳定处理真实任务，取决于它能看到什么、能做什么、能验证什么、能被怎样约束，以及能以什么形式交付结果。

本章建立一张能力地图。它不是为了给工具排名，而是帮助移动端工程师判断：某个任务适合 IDE 助手、终端 agent、云端 agent，还是团队自建 harness；某个任务需要只读分析、受限写入、测试闭环，还是必须人工执行。

## 学习目标

- 建立 AI coding agent 的核心能力维度。
- 区分 IDE 内助手、终端 agent、云端 coding agent 和自建 agent harness。
- 理解上下文、工具、权限、验证、交付形态之间的关系。
- 能够判断移动端项目中哪些任务适合 agent，哪些任务不适合。

## 2.1 能力地图的七个维度

一个 AI coding agent 至少需要从七个维度评估。

| 维度 | 要回答的问题 | 移动端例子 |
| --- | --- | --- |
| 上下文输入 | agent 能看到哪些信息？ | 当前文件、崩溃堆栈、设计稿、Gradle 输出、Issue |
| 代码修改 | agent 能改哪些范围？ | 单个 ViewModel、多模块网络层、测试文件、本地化资源 |
| 命令执行 | agent 能运行哪些命令？ | `xcodebuild`、`gradle test`、`flutter test`、`npm test` |
| 工具连接 | agent 能连接哪些外部系统？ | GitHub、CI、崩溃平台、日志平台、设计系统、MCP server |
| 权限控制 | 哪些动作需要确认或禁止？ | 签名文件、证书、`.env`、发布脚本、生产开关 |
| 反馈循环 | agent 能否读取失败并修正？ | 编译失败后定位类型错误，测试失败后修正 mock |
| 交付形态 | 结果如何给人类审查？ | diff、patch、commit、PR、测试报告、风险清单 |

这七个维度组合起来，才决定 agent 的真实能力。一个能运行命令但不能读测试输出的工具，无法形成闭环；一个能改文件但没有权限边界的工具，会放大风险；一个能搜索仓库但不能输出结构化变更摘要的工具，会增加审查成本。

## 2.2 四类工具形态

AI coding 工具可以按运行位置和工程集成方式分为四类。

| 工具形态 | 典型位置 | 优势 | 风险 |
| --- | --- | --- | --- |
| IDE 助手 | 编辑器内 | 贴近当前文件、补全体验好、适合小改动 | 容易局部优化，缺少完整验证 |
| 终端 agent | 本地项目目录 | 能运行命令、理解真实工程、适合调试和重构 | 权限和命令风险更高 |
| 云端 coding agent | 远端沙箱 | 可以并行处理任务、长时间运行、生成 PR | 本地设备、证书、私有依赖和模拟器较难复现 |
| 自建 agent harness | 团队平台 | 可接入内部工具、指标和审批流程 | 建设成本高，需要持续维护 |

这四类工具不是替代关系，而是任务分层关系。写一个小组件，IDE 助手就够；修一个编译错误，终端 agent 更合适；处理一个明确 Issue 并生成 PR，云端 agent 可能更高效；接入公司内部崩溃平台、设计系统和发布流程，则需要团队级 harness。

## 2.3 IDE 助手：局部效率工具

IDE 助手的优势在于低摩擦。开发者不需要离开当前文件，就可以获得补全、解释、重构建议和局部代码生成。

适合 IDE 助手的任务：

- 补全 Swift/Kotlin/Dart/TypeScript 代码。
- 生成小函数和局部测试。
- 改写 UI 文案。
- 解释当前文件中的复杂逻辑。
- 根据当前类型补齐 `switch` 或 `when` 分支。

不适合只靠 IDE 助手的任务：

- 跨模块架构重构。
- 构建系统升级。
- 涉及多平台配置的改动。
- 需要运行测试并根据失败修正的任务。

IDE 助手的正确定位是“局部加速器”。它可以减少手写代码成本，但不能替代工程验证。

## 2.4 终端 Agent：贴近真实工程

终端 agent 的关键优势是贴近真实工作区。Claude Code 和 Codex CLI 这类工具都强调在本地项目目录中工作：读取代码、修改文件、运行命令、处理 Git 流程。对移动端工程师来说，这意味着 agent 可以看到真实的构建脚本、测试命令、目录结构和错误输出。

终端 agent 适合：

- 修复编译错误。
- 根据测试失败定位问题。
- 批量修改 API 调用。
- 生成或更新测试。
- 整理 README 和脚本。
- 做小范围重构。

但终端 agent 也最需要权限边界。移动端项目里有很多不能随意触碰的内容：

- iOS 签名、证书、entitlements。
- Android keystore、签名配置、发布 flavor。
- `.env`、API key、内部域名。
- 发布脚本、上传脚本、生产开关。
- 本地缓存、构建产物和用户日志。

终端 agent 的安全设计不是“永远不让它执行命令”，而是把命令分级：哪些可以自动运行，哪些需要确认，哪些永远禁止。

## 2.5 云端 Coding Agent：并行和长任务

云端 coding agent 的优势是并行和隔离。OpenAI Codex 的产品演进中，一个重要方向是让多个 agent 在隔离环境中处理不同任务，并让开发者审查它们的变更。Codex app 进一步把这种模式提升为“管理多个 agent 的命令中心”：agent 可以在独立线程和 worktree 中工作，开发者可以查看 diff、评论、接管或继续让 agent 运行。

这种形态适合：

- 明确 Issue 的修复。
- 文档更新。
- 测试补充。
- 低风险重构。
- 多方案探索。
- 并行处理多个小任务。

移动端项目使用云端 agent 时要注意环境差异：

- 云端是否能安装 Xcode 或 Android SDK？
- 是否有私有依赖仓库权限？
- 是否需要访问内部 CocoaPods/Gradle 源？
- 是否能运行模拟器或只做静态检查？
- 是否能安全处理证书和密钥？

很多移动端任务最终仍需要本地设备或 CI 验证。因此云端 agent 更适合作为“生成可审查变更”的生产力工具，而不是完整替代本地验证。

## 2.6 自建 Harness：团队级工程能力

当团队希望把 AI coding 纳入研发流程，就不能只依赖个人工具配置。团队需要自建或配置一套 harness，把工具、权限、测试、日志、审批和知识库连接起来。

一个团队级 harness 可能包括：

- 任务入口：Issue 模板、PR 模板、任务分类。
- 上下文入口：架构文档、模块说明、代码地图、历史决策。
- 工具入口：GitHub、CI、崩溃平台、日志平台、设计系统、MCP server。
- 权限入口：只读、受限写、命令白名单、敏感文件黑名单。
- 验证入口：单元测试、lint、构建、截图对比、性能检查。
- 审查入口：自动生成 PR 摘要、风险说明、测试结果。

自建 harness 的价值不在于“做一个自己的 ChatGPT”，而在于把团队工程知识变成 agent 可以使用的环境。

## 2.7 移动端任务适配表

| 任务 | 推荐工具形态 | 必要验证 | 人工介入点 |
| --- | --- | --- | --- |
| 补一个小 UI 状态 | IDE 助手 | 局部测试或预览 | UI 视觉确认 |
| 修复单元测试失败 | 终端 agent | 单测通过 | 变更摘要审查 |
| 迁移网络错误模型 | 终端 agent + 分批 PR | 单测、集成测试、弱网检查 | 文案、埋点、兼容性 |
| 更新权限配置 | 终端 agent 只读建议 | 平台配置检查 | 必须人工确认 |
| 升级 Gradle/AGP | 终端 agent 或云端 agent | 构建、测试、回滚说明 | 依赖和 CI 审查 |
| 批量补文档 | 云端 agent | 链接检查、目录检查 | 内容准确性 |
| 多方案重构探索 | 云端多 agent | 每个方案独立构建 | 架构决策 |
| 内部崩溃自动分析 | 自建 harness | 任务集评测 | 隐私和权限审查 |

## 2.8 能力边界：Agent 不是资深同事

再强的 agent 也不是团队中的资深同事。它可以执行、搜索、总结和修复，但它没有真实责任边界，也不理解你公司内部的产品优先级、合规承诺和线上事故成本。人类仍然需要负责：

- 决定做什么、不做什么。
- 判断风险是否可接受。
- 审查架构方向。
- 确认用户体验。
- 对生产变更负责。

Anthropic 对 agentic coding 的研究也指出，人和 agent 之间存在分工：人决定构建什么，agent 处理如何构建的很大一部分。这一点对移动端团队尤其重要，因为移动端改动直接触达用户设备，回滚和修复成本往往高于服务端。

## 2.9 能力成熟度模型

团队可以用五级模型评估自己的 AI coding 能力。

| 等级 | 特征 | 移动端表现 |
| --- | --- | --- |
| L0 手工试用 | 个人临时复制粘贴 | 偶尔让模型解释报错 |
| L1 局部辅助 | IDE 补全和聊天稳定使用 | 补测试、写 mapper、解释 Gradle 错误 |
| L2 仓库感知 | agent 能读取仓库并小范围改动 | 修复 ViewModel、更新文档、运行单测 |
| L3 工程闭环 | 有命令、权限、测试和 PR 流程 | agent 可完成低风险 Issue 并提交可审查 diff |
| L4 团队平台 | 有 MCP、skills、hooks、评测和治理 | 团队按任务类型分配 agent，并持续度量收益 |

多数团队不需要一步到 L4。更现实的路径是先把 L1 和 L2 做扎实：让 agent 能看懂项目、跑对测试、遵守禁止事项。只有当这些基础稳定后，再考虑多 agent、云端并行和内部工具接入。

成熟度模型的作用不是贴标签，而是帮助团队避免错配。一个还没有稳定测试命令的仓库，直接上云端多 agent 并行，只会让多个任务同时失败。一个没有高风险文件清单的团队，让 agent 自动修改移动端项目，很容易碰到签名、权限和发布配置。

## 2.10 任务选择的三问

判断一个任务是否适合 agent，可以问三个问题：

第一，任务是否有明确输入？例如崩溃堆栈、失败测试、目标文件、Issue 描述、验收条件。如果输入只有“帮我优化一下”，就不适合直接执行。

第二，任务是否有明确边界？例如只修改某个模块，不新增依赖，不改权限，不碰支付。边界越清晰，agent 越容易成功。

第三，任务是否有明确验证？例如单元测试、构建命令、lint、快照测试、人工真机检查。没有验证的任务，即使 agent 生成了看似合理的代码，也很难判断质量。

可以把这三个问题写成表格：

| 问题 | 是 | 否 |
| --- | --- | --- |
| 输入明确 | 可以进入计划阶段 | 先补上下文 |
| 边界明确 | 可以受限写入 | 先拆任务 |
| 验证明确 | 可以进入闭环 | 先建设 harness |

这张表能帮助移动端团队避免把不成熟任务过早交给 agent。

## 2.11 多工具共存策略

真实团队不会只使用一种 AI coding 工具。有人习惯 IDE 补全，有人偏好终端 agent，有人用云端 agent 处理 Issue，还有人接入内部工具。多工具共存时，需要统一的是规则，而不是界面。

建议团队至少统一四件事：

- 统一任务分类：哪些任务是绿色、黄色、红色。
- 统一高风险文件：哪些路径必须人工审查。
- 统一验证命令：每个平台、每个模块怎样跑测试。
- 统一交付格式：PR 描述必须包含修改范围、验证结果、风险和未验证项。

这样即使工具不同，工程结果仍然可审查。AI coding 的治理重点不是限制大家只能用某个工具，而是让所有工具都进入同一套工程边界。

## 本章小结

AI coding agent 的价值来自“工程动作链”，不是来自单次代码生成。评估工具时，要看它能看到什么上下文，能调用什么工具，能执行什么命令，如何限制权限，如何验证结果，以及如何交付给人类审查。下一章会把这些能力落到移动端工作区：如何整理仓库，让 agent 更容易做对事。
