# 第 4 章 Prompt Engineering：把编程任务说清楚

## 本章导读

在 AI coding 中，Prompt 不只是“请帮我写代码”。它更像一份任务委托书：告诉 agent 要解决什么问题、在哪些边界内行动、可以使用哪些证据、应该如何验证，以及最终要交付什么。

OpenAI Codex 的官方提示建议强调两点：一是提供相关文件、图片、复现步骤和验证方法等上下文；二是复杂任务要拆成更小、更聚焦的步骤，因为小任务更容易测试，也更容易审查。这个原则对移动端工程尤其重要。移动端任务经常跨 UI、状态、网络、权限、本地化、构建和测试，如果 Prompt 只写“帮我修一下”，agent 很容易把猜测当成事实。

本章讨论 Prompt Engineering 在 AI coding 中的具体写法。它不是为了追求“咒语”，而是为了把工程任务变成可执行、可验证、可审查的委托。

## 学习目标

- 能够把模糊需求改写为可执行任务。
- 掌握移动端 bug 修复、重构、测试补充和文档生成的 Prompt 模板。
- 理解 Prompt、上下文、工具和验证之间的边界。
- 学会为 agent 写清楚禁止事项、停止条件和交付格式。

## 4.1 Prompt 是任务协议

人类同事接到任务时，会主动问澄清问题，会根据团队经验避开危险文件，也知道什么时候应该找负责人确认。Agent 不具备这些组织经验，除非你把它们写进 Prompt、规则文件或 harness。

一个编程任务 Prompt 至少要包含六类信息：

| 组成 | 要回答的问题 | 移动端例子 |
| --- | --- | --- |
| 背景 | 任务属于哪个模块、平台和业务？ | iOS 登录页弱网重试崩溃 |
| 目标 | 完成后用户或代码应该有什么变化？ | 重试时不崩溃，并显示可恢复错误 |
| 约束 | 哪些文件、依赖、行为不能改？ | 不改签名、登录协议、埋点字段 |
| 上下文 | 哪些文件、日志、Issue、设计说明相关？ | 崩溃堆栈、最近提交、ViewModel |
| 验收 | 用什么命令证明任务完成？ | 运行 `LoginTests` 和 lint |
| 输出 | 最终要交付什么？ | 修改摘要、测试结果、风险点 |

这六类信息越清楚，agent 越不需要猜。

## 4.2 差 Prompt 和好 Prompt

差 Prompt：

```text
帮我修一下登录页崩溃。
```

这个 Prompt 缺少平台、复现条件、相关文件、禁止事项和验收方式。Agent 可能直接搜索“login crash”，然后改第一个看起来相关的文件。

更好的 Prompt：

```text
任务：修复 iOS 登录页在弱网重试后偶发崩溃的问题。

背景：
- 崩溃发生在 5.8.0 版本，主要集中在 iOS 17。
- 用户点击“重试”后，页面偶发退出。
- 崩溃堆栈指向 LoginViewModel.handleRetry。

请先阅读：
- ios/App/Login/LoginViewModel.swift
- ios/App/Login/LoginRepository.swift
- ios/App/Network/NetworkSession.swift
- ios/AppTests/LoginTests.swift

约束：
- 不要修改签名、证书、登录协议和埋点字段。
- 不要新增第三方依赖。
- 优先做最小修复。

要求：
1. 先解释你对崩溃原因的判断。
2. 如果需要改代码，只改最小范围。
3. 补充一个覆盖弱网重试或取消状态的测试。
4. 运行 LoginTests。

输出：
- 修改文件列表。
- 修复思路。
- 测试命令和结果。
- 仍需人工真机验证的点。
```

这个 Prompt 没有要求 agent 一次性解决所有登录问题，而是把任务限定在一个可验证范围内。

## 4.3 Prompt 中的“禁止事项”

AI coding 中最容易忽略的是禁止事项。人类工程师知道不要随便改签名文件、发布脚本、生产配置和密钥，但 agent 不一定知道。

移动端 Prompt 常见禁止事项：

- 不修改 `.env`、证书、profile、keystore、签名配置。
- 不新增第三方依赖，除非任务明确要求。
- 不修改生产域名、灰度开关和发布脚本。
- 不改变登录、支付、风控、隐私权限等高风险业务逻辑。
- 不删除测试以换取通过。
- 不做大范围格式化。

更好的写法不是“注意安全”，而是写具体文件和行为：

```text
禁止修改：
- ios/Signing/
- android/keystore.properties
- .env*
- scripts/release_*.sh

如果你认为必须修改这些文件，请停止并说明原因，不要直接修改。
```

## 4.4 Prompt 要包含验证

没有验证的 Prompt，只是在请求生成代码。Agent 的输出可能语法正确，但仍然不满足工程要求。

验证可以分层：

| 层级 | 示例 |
| --- | --- |
| 快速验证 | 单个模块单元测试、类型检查 |
| 中等验证 | lint、相关测试集、Debug 构建 |
| 高成本验证 | UI 测试、真机验证、全量构建 |
| 人工验证 | 视觉、交互、隐私、权限、埋点 |

Prompt 中应写清楚最低验证要求：

```text
完成修改后运行：
- ./scripts/test_unit.sh LoginTests
- ./scripts/lint.sh ios

如果命令无法运行，请说明原因和你已经做过的替代检查。
```

“无法运行也要说明原因”很重要。它能防止 agent 把未验证的工作包装成已完成。

## 4.5 任务拆分 Prompt

复杂任务不要直接要求 agent 实现。先让它拆任务。

```text
请不要先改代码。

目标：把 Android 网络错误处理迁移到统一 ErrorModel。

请先输出：
1. 需要阅读的文件列表。
2. 可能受影响的模块。
3. 建议的迁移步骤。
4. 每一步的验证命令。
5. 哪些步骤需要人工确认。
```

这类 Prompt 的作用是让 agent 先做分析，把任务从“模糊大目标”变成“可执行小步骤”。之后再选择其中一步交给 agent 执行。

## 4.6 四类常用 Prompt 模板

### Bug 修复模板

```text
任务：修复 [平台/模块] 中的 [现象]。

输入：
- 复现步骤：
- 错误日志/崩溃堆栈：
- 相关文件：
- 最近改动：

约束：
- 不修改：
- 不新增依赖：

要求：
1. 先解释原因假设。
2. 做最小修复。
3. 补充或更新测试。
4. 运行验证命令。

输出：
- 根因判断。
- 修改摘要。
- 测试结果。
- 未覆盖风险。
```

### 重构模板

```text
任务：重构 [模块]，目标是 [降低重复/拆分组件/统一接口]。

约束：
- 不改变外部行为。
- 每次只改一个明确范围。
- 不做无关格式化。

要求：
1. 先列出当前重复点或结构问题。
2. 提出最小重构计划。
3. 等我确认后再改代码。
4. 每轮修改后运行相关测试。
```

### 测试补充模板

```text
任务：为 [模块/函数] 补充测试。

请覆盖：
- 正常路径。
- 错误路径。
- 边界输入。
- 移动端生命周期或取消场景。

约束：
- 不为了测试修改生产逻辑，除非先说明原因。
- 不引入脆弱 sleep。
- 优先使用现有测试工具和 mock 风格。
```

### 文档生成模板

```text
任务：为 [模块] 生成面向移动端团队的维护说明。

要求：
- 先阅读源码和测试。
- 说明模块职责、入口、依赖和常见错误。
- 列出修改该模块时必须运行的验证命令。
- 不要编造代码中不存在的能力。
```

## 4.7 Prompt 与其他工程能力的边界

Prompt 不是万能的。很多问题不能靠写得更礼貌、更详细解决。

- 如果 agent 看不到相关文件，这是 Context Engineering 问题。
- 如果 agent 不能运行测试，这是 Harness Engineering 问题。
- 如果 agent 一直重复失败，这是 Loop Engineering 问题。
- 如果 agent 能修改敏感文件，这是权限治理问题。

Prompt Engineering 的职责是清楚表达任务。它不能替代仓库整理、工具接入和验证机制。

## 4.8 常见失败模式

| 失败模式 | 表现 | 修正方式 |
| --- | --- | --- |
| 任务过大 | agent 一次改很多模块 | 先要求拆计划，再分批执行 |
| 验收缺失 | agent 只说完成，没有测试 | Prompt 中写明命令和输出格式 |
| 禁止事项缺失 | agent 修改签名或配置 | 列出具体禁止文件 |
| 目标模糊 | agent 做了相邻但不需要的优化 | 写清楚“只做什么” |
| 输出不明确 | agent 不知道要解释还是改代码 | 写清楚最终交付物 |
| 无停止条件 | agent 多轮尝试仍继续 | 设置最大步数和升级条件 |

## 4.9 面向移动端的 Prompt 评审

移动端 prompt 写完后，可以像代码一样评审。评审重点不是语言是否流畅，而是任务是否可执行。

检查项如下：

- 是否说明平台和技术栈。
- 是否说明目标模块和相关文件。
- 是否说明不允许修改的文件。
- 是否说明测试命令。
- 是否说明输出格式。
- 是否说明人工确认点。
- 是否限制了本轮范围。
- 是否要求列出未验证项。

例如“帮我修复登录页弱网问题”不是好 prompt，因为它缺少平台、文件、复现、边界和验证。更好的写法是：“在 Android 登录模块中，修复 LoginViewModel 断网重试后 loading 不结束的问题。请先阅读 LoginViewModel、LoginRepository、NetworkErrorMapper 和现有测试；只允许修改登录模块和测试；不要改 Manifest、Gradle、依赖和埋点字段；新增 timeout/offline 测试；运行 `./gradlew testDebugUnitTest`；输出修改、验证和风险。”

这段 prompt 变长了，但长不是目的。它把工程任务从“愿望”变成“可执行委托”。

## 4.10 任务拆分 Prompt

当任务较大时，不要直接让 agent 修改代码。先让它拆分任务。

```text
请只做计划，不要修改代码。

目标：把订单模块的网络错误处理迁移到统一 AppError。

请输出：
1. 可能受影响的文件和模块。
2. 推荐拆分的 3-5 个小任务。
3. 每个小任务的验证命令。
4. 高风险点。
5. 第一轮最小可执行任务。
```

这种 prompt 的价值在于把 agent 从“执行者”切换成“分析者”。人类确认计划后，再进入受限写入阶段。对移动端重构来说，这一步能显著减少大 diff 和越界修改。

## 4.11 Prompt 版本化

团队中常用 prompt 不应该散落在聊天记录里。建议把它们版本化：

```text
docs/ai-coding/prompts/
  bugfix.md
  refactor.md
  test-generation.md
  ios-permission-review.md
  android-gradle-failure.md
  flutter-widget-split.md
  react-native-bridge-review.md
```

每个 prompt 模板都应该经过真实任务验证。模板不是写完就固定，而是根据失败案例持续更新。例如某次 agent 修改了 `Info.plist` 引发审核风险，就应该把 Info.plist 加入相关模板的禁止清单。

Prompt 版本化还有一个好处：团队可以复盘 AI coding 质量。如果某类任务经常失败，可以对比 prompt、上下文和测试命令，而不是只说“模型不行”。

## 本章小结

Prompt Engineering 的目标不是让模型“更听话”，而是把工程任务变成可执行、可验证、可审查的委托。好的 Prompt 会定义目标、上下文、约束、验证和输出；更好的 Prompt 还会告诉 agent 什么时候停下来。下一章会讨论 Prompt 之外更大的问题：如何组织仓库上下文。
