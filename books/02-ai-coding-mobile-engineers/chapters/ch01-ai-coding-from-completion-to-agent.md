# 第 1 章 从代码补全到编程 Agent

## 本章导读

AI coding 的变化不是“模型更会写代码”这么简单。过去几年，开发者经历了三个明显阶段：先是在 IDE 中接受下一行代码补全，然后在聊天窗口里让模型解释错误、生成函数、整理文档，随后开始把模型放进真实工程环境，让它读取仓库、编辑文件、运行命令、观察结果并继续修正。第三个阶段才是本书讨论的重点：agentic coding。

对移动端工程师来说，这个变化尤其重要。移动端项目不是单个脚本，而是由 UI 状态、网络层、缓存、权限、平台配置、构建系统、签名、测试、发版和线上质量指标共同组成的工程系统。一个只会生成代码片段的模型，最多能帮你写一个 `ViewModel` 方法；一个能进入工程闭环的 agent，则可能帮助你定位崩溃、补测试、迁移网络层、更新文档、生成 PR 描述，并把失败的测试结果反馈给下一轮修改。

能力提升的同时，风险也放大了。agent 可以改多个文件，也可能改错多个文件；agent 可以运行测试，也可能运行危险命令；agent 可以总结仓库，也可能误解架构边界。AI coding 专家的核心能力，不是“会问 AI 问题”，而是能把 agent 放进正确的上下文、工具、权限和验证循环中。

## 学习目标

- 理解 AI coding 从代码补全、聊天辅助、仓库感知到 agentic coding 的阶段演进。
- 知道 Claude Code、Codex 等工具为什么代表了工程能力的变化，而不只是交互界面的变化。
- 区分模型能力、上下文能力、工具能力和工程流程能力。
- 建立移动端项目使用 AI coding 的收益边界和风险意识。
- 为后续 Prompt Engineering、Context Engineering、Harness Engineering、Loop Engineering 建立统一背景。

## 1.1 为什么代码补全不是终点

代码补全阶段的核心体验是“模型接着你写”。它通常依赖当前文件、光标附近代码、打开的相关文件和语言服务信息，生成下一行、一个函数或一个小片段。这类能力很适合重复性代码：DTO 映射、UI 样式、空状态处理、简单测试断言、平台 API 包装。

但代码补全有天然边界。它通常不知道完整业务目标，也不一定知道你为什么这样设计模块。它看到的是局部上下文，容易在局部最优上做得很好，却在工程全局上犯错。例如：

- 在 Android 项目中补了一个看似正确的网络错误处理，却绕过了团队统一的 `ErrorMapper`。
- 在 iOS 项目中生成了权限请求代码，却没有同步更新 `Info.plist` 文案。
- 在 Flutter 页面里补了状态变量，却没有考虑页面恢复、热重载和状态管理约束。
- 在 React Native 中生成了平台判断，却没有覆盖 Android 返回键或 iOS Safe Area。

代码补全解决的是“写得快”，但真实工程还需要“改得对、能验证、可审查、能回滚”。

## 1.2 聊天式编程带来了什么

聊天式编程把模型从“补下一行”推进到“解释和生成”。开发者可以贴一段错误日志，让模型解释堆栈；可以描述一个需求，让模型生成函数；可以让模型比较两个方案；也可以把 API 文档转成示例代码。

这一阶段对移动端开发很有价值，因为移动端项目有大量跨知识域问题：

- Xcode 构建错误可能来自签名、Swift Package、CocoaPods、编译缓存或架构设置。
- Gradle 错误可能来自插件版本、Kotlin 版本、AGP、JDK、仓库源或 flavor 配置。
- Flutter 问题可能同时涉及 Dart、iOS Pod、Android Gradle 和资源声明。
- React Native 问题可能同时涉及 TypeScript、Metro、Hermes、原生 bridge 和依赖版本。

聊天式工具能帮助工程师快速建立排查方向。但它仍然缺少一个关键能力：它不能直接进入工程现场。你贴给它什么，它就主要基于什么回答。你漏了构建脚本，它不知道构建脚本；你没贴测试，它不知道测试；你没说明不能改签名配置，它可能建议你改签名配置。

这也是为什么很多团队在早期使用 AI 编程时会经历一个落差：模型解释问题很强，但真正落地到仓库里，仍然需要工程师手工复制、修改、测试、回滚和整理 PR。

## 1.3 仓库感知：从片段到工程上下文

仓库感知阶段的变化，是工具开始理解“当前项目”而不只是“当前输入”。它可以搜索文件、读取目录、定位符号、理解调用关系、引用最近打开文件，甚至根据 Issue、PR、测试输出和文档补充上下文。

移动端项目尤其需要仓库感知。以“登录页弱网重试崩溃”为例，真正相关的上下文可能分散在多个位置：

- `LoginViewModel`：页面状态和重试入口。
- `LoginRepository`：业务请求封装。
- `NetworkSession` 或 `ApiClient`：底层网络回调。
- `ErrorMapper`：错误转换和用户文案。
- 本地化资源：错误文案是否覆盖多语言。
- 单元测试：是否已有重试和取消测试。
- 崩溃平台日志：是否只发生在特定系统版本。

如果 agent 只能看当前文件，它很可能做局部补丁；如果它能读取仓库并建立上下文，它就可能发现真正的问题是取消后的回调仍然更新 UI 状态，或者弱网重试触发了重复 completion。

仓库感知还带来一个新的工程问题：上下文不是越多越好。把整个仓库塞进上下文会浪费 token，也会引入噪声。优秀的 AI coding 工作流需要选择、压缩和更新上下文，这就是后续第 5 章讨论的 Context Engineering。

## 1.4 Agentic Coding：模型开始执行工程动作

Agentic coding 的关键变化是模型不再只是回答，而是可以执行一组工程动作：

1. 读取任务和约束。
2. 搜索仓库。
3. 打开相关文件。
4. 制定小计划。
5. 修改代码。
6. 运行测试或构建命令。
7. 读取失败输出。
8. 继续修复。
9. 输出变更摘要、测试结果和风险。

Claude Code 和 Codex 代表的正是这种方向。Claude Code 官方将其定位为生活在终端中的 agentic coding tool，能够理解代码库、执行常规任务、解释复杂代码并处理 Git 工作流。OpenAI 的 Codex 则从云端软件工程 agent、CLI、IDE 到桌面 app，不断强化“多 agent 并行、长任务、skills、MCP、代码审查和工作流编排”等能力。OpenAI 在 Codex app 介绍中明确指出，开发者的工作正在从和单个 agent 结对，转向监督多个 agent 跨项目并行工作。

这说明 AI coding 的焦点已经从“模型能不能写函数”转向“人如何委托、监督和协作”。对移动端工程师来说，真正的问题变成：

- 哪些任务适合委托给 agent？
- 哪些文件和命令必须禁止？
- 哪些测试必须运行？
- agent 修改失败后如何恢复？
- 多个 agent 并行时如何避免冲突？
- 人类应该在哪些检查点介入？

## 1.5 技术演进的四个阶段

为了后续讨论，本书把 AI coding 的演进分为四个阶段。

| 阶段 | 典型能力 | 主要收益 | 主要风险 |
| --- | --- | --- | --- |
| 代码补全 | 当前文件补全、函数生成 | 提升局部编码速度 | 容易忽略架构和业务边界 |
| 聊天辅助 | 解释错误、生成片段、方案比较 | 降低搜索和理解成本 | 依赖人工提供完整上下文 |
| 仓库感知 | 搜索文件、读取多文件、理解调用关系 | 能处理局部工程任务 | 上下文选择和噪声控制困难 |
| Agentic coding | 计划、编辑、执行命令、验证、循环修复 | 能处理可验证的真实任务 | 权限、安全、测试和审查要求更高 |

这四个阶段不是互相替代，而是叠加。今天的 AI coding 工具通常同时具备补全、聊天、仓库感知和 agent 能力。工程师要做的是根据任务选择合适的模式，而不是所有问题都交给 agent。

## 1.6 移动端案例：Android 网络层迁移

假设团队要把旧的网络错误处理迁移到统一错误模型。这个任务看起来简单，实际上涉及多个层次：

- Retrofit/OkHttp 或自研网络层。
- Repository 层错误映射。
- ViewModel 状态转换。
- UI 错误文案。
- 埋点字段。
- 单元测试和 UI 测试。
- 弱网、超时、取消、重试。

如果使用代码补全，模型可以帮你补一个 `when` 分支。如果使用聊天式 AI，它可以解释错误模型设计。如果使用仓库感知工具，它可以找到所有调用方。如果使用 agentic coding，它可以按批次迁移页面、补测试、运行测试、修复编译错误，并输出哪些页面还没迁移。

但这个任务不能完全无监督交给 agent。原因很简单：网络错误不仅是代码问题，也是产品体验、埋点分析和线上稳定性问题。正确做法是把任务拆成可验证的小循环：

1. 先让 agent 只读仓库，输出迁移影响面。
2. 人类确认迁移范围和禁止事项。
3. agent 只实现统一错误模型和测试。
4. 运行测试并审查 diff。
5. agent 迁移两个低风险页面。
6. 人类检查文案、埋点和弱网体验。
7. 再按页面批次扩展。

这就是本书反复强调的思路：AI coding 的核心不是放手，而是设计可控委托。

## 1.7 从工具崇拜到工程能力

AI coding 领域更新很快。Claude Code、Codex、Cursor、GitHub Copilot、Windsurf、Gemini CLI、各种 MCP server 和内部 agent 平台都会持续变化。工具会变，底层工程问题更稳定：

- 如何表达任务。
- 如何组织上下文。
- 如何限制权限。
- 如何运行验证。
- 如何保存中间状态。
- 如何审查和回滚。
- 如何评估收益。

这也是本书选择 Prompt Engineering、Context Engineering、Harness Engineering、Loop Engineering 作为主线的原因。它们不是某个工具的功能菜单，而是 agentic coding 能否在真实工程中稳定工作的四类基础能力。

## 本章小结

AI coding 的核心演进，是从局部生成走向工程闭环。代码补全解决写得快，聊天辅助解决理解快，仓库感知解决定位快，agentic coding 则开始解决“让模型参与真实工程动作”的问题。越靠近真实工程，越需要上下文、工具、权限、验证和人类审查。后续章节会围绕这四类能力展开，并把它们落到移动端开发的具体场景中。
