# 第 5 章 Context Engineering：组织仓库上下文

## 本章导读

AI coding 的质量往往由上下文决定。模型不知道你项目中的网络层封装、历史兼容逻辑、平台差异和团队约定，就很容易生成“语法正确但工程错误”的代码。

Anthropic 在 context engineering 文章中把上下文称为 agent 的关键但有限资源，并指出工程重点已经从“怎样写提示词”扩展到“怎样配置最可能产生期望行为的上下文”。Codex 文档也强调，agent 会从文件内容、工具输出以及自己已经做了什么/还要做什么的记录中持续收集上下文；长任务中还可能自动压缩上下文。换句话说，Context Engineering 不是一次性贴资料，而是贯穿 agent 工作全程的上下文管理。

## 学习目标

- 理解上下文不是越多越好，而是要相关、分层、可更新。
- 掌握仓库地图、错误现场、最近文件、架构摘要和任务上下文的组织方式。
- 知道如何避免上下文污染、上下文过载和上下文遗失。
- 能够为移动端项目设计适合 agent 的上下文入口。

## 5.1 Prompt 和 Context 的区别

Prompt 是指令，Context 是模型可以参考的状态。

| 项目 | Prompt | Context |
| --- | --- | --- |
| 作用 | 告诉 agent 要做什么 | 告诉 agent 当前世界是什么样 |
| 例子 | “修复登录页弱网崩溃” | 堆栈、源码、测试、架构文档 |
| 变化频率 | 任务开始时较稳定 | 随文件读取、命令输出和失败结果变化 |
| 风险 | 表达不清 | 过多、过期、不相关、不可信 |

在 AI coding 中，很多失败并不是 Prompt 写错，而是 Context 不完整。例如你告诉 agent “修复登录崩溃”，却没有给崩溃堆栈、最近改动和相关测试；agent 只能猜。

## 5.2 上下文类型

| 类型 | 内容 | 用途 |
| --- | --- | --- |
| 仓库地图 | 模块、目录、脚本、测试入口 | 让 agent 快速定位 |
| 架构摘要 | 网络层、状态管理、路由、依赖方向 | 避免违反架构 |
| 错误现场 | 堆栈、日志、复现步骤、设备信息 | 支持 bug 修复 |
| 最近文件 | 当前任务最相关的源文件和测试 | 支持局部修改 |
| 历史决策 | 为什么这样设计、为什么不能改 | 避免重复踩坑 |
| 验收上下文 | 构建、测试、lint、截图检查 | 支持闭环 |
| 运行记录 | agent 已做什么、还要做什么 | 支持长任务连续性 |

移动端项目中，最容易缺失的是历史决策和验证上下文。比如“为什么不能把错误文案写在网络层”，通常不在代码里；“修改权限后要检查隐私文案”，也不一定在测试里。这些内容需要写入文档、规则或 Playbook。

## 5.3 仓库地图：让 Agent 少搜索

仓库地图不是完整架构文档，而是给 agent 的导航图。

示例：

```text
核心目录：
- ios/App/Login/：登录页面和 ViewModel
- ios/App/Network/：网络层和错误映射
- ios/App/Resources/Localizable.strings：iOS 文案
- android/app/src/main/java/.../login/：Android 登录流程
- packages/shared/：跨端共享协议和模型

常用脚本：
- ./scripts/test_unit.sh LoginTests
- ./scripts/lint.sh ios
- ./scripts/check_localization.sh
```

仓库地图的价值是减少 agent 的盲目探索。它不需要一开始 `find` 全仓库，也不需要猜测试命令。

## 5.4 错误现场：Bug 修复的关键上下文

Bug 修复任务要给 agent “现场”，而不是只给现象。

弱网重试崩溃的上下文可以包括：

- 崩溃堆栈。
- 用户操作路径。
- 设备型号、系统版本、App 版本。
- 最近相关提交。
- 相关日志。
- 复现概率。
- 已排除的原因。
- 相关测试是否存在。

好的错误现场示例：

```text
现象：iOS 登录页弱网下点击重试，偶发崩溃。
版本：5.8.0，主要发生在 iOS 17。
堆栈：LoginViewModel.handleRetry -> LoginRepository.retry -> NetworkSession callback。
最近改动：NetworkSession 新增取消逻辑。
已知事实：未登录用户和已登录过期用户都可能触发。
需要检查：取消后回调是否仍更新 UI 状态。
```

这类上下文可以让 agent 建立合理假设，而不是泛泛搜索“crash”。

## 5.5 上下文预算：不是越多越好

移动端项目常见上下文很大：一个登录流程可能涉及 UI、ViewModel、Repository、网络层、缓存、风控、埋点、本地化和权限配置。把所有文件都塞给 agent 不现实，也不利于推理。

更合理的做法是分层：

1. 先给任务摘要和仓库地图。
2. 再让 agent 自己读取最相关文件。
3. 如果发现缺口，再补架构说明或日志。
4. 运行测试失败后，只补充失败输出和相关文件。
5. 长任务中定期压缩状态，保留决策、待办和风险。

Context Engineering 的关键是“选择”。信息过少会让 agent 猜；信息过多会稀释注意力，让重要约束被淹没。

## 5.6 上下文压缩与任务交接

长任务经常跨越多个上下文窗口。官方资料中，Codex 会在长任务中自动压缩上下文；Anthropic 的长任务 harness 研究则强调，压缩并不充分，agent 还需要明确的进度文件、feature list、git 历史和结构化交接。

移动端长任务可以使用类似策略：

```text
docs/ai-progress.md

## 目标
迁移网络错误模型。

## 已完成
- 新增 ErrorModel。
- 补充 ErrorModelTests。
- 迁移 Login 页面。

## 待完成
- 迁移 Payment 页面。
- 检查埋点字段。
- 更新本地化文案。

## 风险
- Payment 页面涉及风控错误码，需要人工确认。
```

这类文件的作用不是给用户看，而是给下一轮 agent 或人类接手时快速恢复上下文。

## 5.7 上下文污染

上下文污染指无关、过期或不可信内容进入 agent 推理链，导致错误修改。例如：

- 旧版本接口文档没有标注日期。
- 用户日志包含误导性猜测。
- 设计稿截图对应的是另一端平台。
- 外部网页内容试图覆盖系统规则。
- 另一个分支的实现被当成当前主干。

移动端常见污染来源是“历史兼容代码”。例如某个分支是为了旧系统版本临时保留的，agent 看到后可能尝试“清理掉”。如果没有注释或文档说明，agent 很难判断这是技术债还是业务兼容。

## 5.8 上下文入口设计

建议为移动端仓库设计四个上下文入口：

1. `README.md`：如何构建、测试和运行。
2. `AGENTS.md`：agent 规则、禁止事项、输出要求。
3. `docs/module-map.md`：核心模块和依赖方向。
4. `docs/ai-progress.md`：长任务进度和交接状态。

如果团队已经有内部知识库，可以通过 MCP 或脚本把 Issue、设计稿、崩溃日志和接口文档接入 agent。但接入之前要先处理权限和脱敏，不能把用户隐私和密钥直接暴露给模型。

## 5.9 移动端上下文检查清单

- 是否给出平台：iOS、Android、Flutter、React Native？
- 是否给出相关文件，而不是只给模块名？
- 是否给出复现步骤和错误日志？
- 是否说明最近改动？
- 是否说明哪些文件不能改？
- 是否给出测试命令？
- 是否标注文档日期和版本？
- 是否过滤用户隐私和密钥？
- 是否为长任务保留进度摘要？

## 本章小结

Context Engineering 的关键是选择和组织，而不是堆砌。优秀的上下文让 agent 少猜、少改错、少跑偏。移动端团队应该把仓库地图、架构摘要、错误现场、验证命令和长任务进度作为一等工程资产。下一章讨论 agent 能力的另一个基础：Harness Engineering。
