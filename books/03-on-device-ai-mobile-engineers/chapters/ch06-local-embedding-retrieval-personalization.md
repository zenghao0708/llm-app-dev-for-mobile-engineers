# 第 6 章 本地 Embedding、检索与个性化

## 本章导读

本地 embedding 和检索可以让 App 在不上传用户数据的情况下提供个性化能力。它适合本地笔记、浏览历史、收藏、聊天记录、相册描述、文档片段和个人知识库。相比直接调用云端问答，本地检索的优势是隐私、离线和低成本；难点是数据生命周期、索引更新、删除一致性、存储加密和结果解释。

对移动端工程师来说，本地检索不是“把向量库存到手机里”这么简单。它是一条从数据授权、文本抽取、分段、embedding、索引、查询、排序、展示、删除到同步的完整链路。任何一环设计不清楚，都会影响用户信任。

## 学习目标

- 理解本地 embedding 的价值和限制。
- 设计本地索引、更新和删除策略。
- 处理隐私、存储和性能问题。
- 能设计一个本地优先、云端可增强的个性化检索架构。

## 6.1 为什么需要本地检索

移动端有大量个人数据，但很多数据不适合直接上传：

- 本地笔记和草稿。
- 浏览历史和收藏。
- 聊天记录。
- 相册和截图。
- 离线文档。
- 录音转写。
- App 内行为偏好。

如果这些数据可以在端侧建立索引，App 就能提供更贴近个人的体验。例如用户搜索“上次会议提到的预算”，本地检索可以从笔记、录音摘要和文档片段中召回相关内容，而不必把全部原文上传云端。

本地检索也能作为端云协同的第一层。端侧先召回少量相关片段，再由用户决定是否交给云端模型生成完整答案。

## 6.2 本地检索链路

典型链路包括：

1. 本地文本抽取。
2. 分段和清洗。
3. embedding 生成。
4. 本地向量索引。
5. 查询召回。
6. 排序和结果展示。

每一步都要记录版本。分段规则变了，embedding 模型变了，索引结构变了，召回结果都会变化。没有版本记录，线上问题很难复现。

## 6.3 数据授权与抽取

本地检索的第一步不是技术，而是授权。用户应该知道哪些数据会被索引，用于什么功能，是否离开设备，如何删除。

数据抽取要遵守最小化原则。比如做“本地笔记搜索”，不一定要索引附件原文件；做“相册搜索”，可以先索引本地生成的低敏描述或标签，而不是保存完整图片特征到日志。

建议把数据源分级：

| 数据源 | 默认策略 |
| --- | --- |
| App 内公开内容 | 可索引 |
| 用户创建的笔记和草稿 | 授权后索引 |
| 相册、截图、录音 | 明确授权后索引 |
| 密钥、支付、证件、健康数据 | 默认不索引或单独确认 |

授权不是一次性弹窗。用户需要能查看、暂停、重建和删除本地索引。

## 6.4 分段策略

分段决定检索质量。分段太短，语义不完整；分段太长，召回不精准，也增加存储和计算成本。

常见策略：

- 按段落分段：适合笔记和文章。
- 按标题层级分段：适合文档。
- 按时间窗口分段：适合聊天和录音转写。
- 按页面或卡片分段：适合 App 内结构化内容。
- 滑动窗口分段：适合长文本，但要控制重复。

每个片段至少应记录：

```text
chunk_id
source_id
source_type
text
metadata
created_at
updated_at
embedding_model_version
deleted
```

不要只保存向量。没有原始片段和元数据，结果无法展示引用，也无法处理删除和重建。

## 6.5 本地索引与存储

移动端本地索引要考虑：

- 索引大小是否随数据量线性增长。
- 是否支持增量更新。
- 是否支持删除。
- 是否需要加密。
- 是否能在低端设备上加载。
- 是否能在 App 升级后迁移。

小规模数据可以使用简单本地存储和线性扫描。数据量变大后，需要更高效的向量索引或混合检索。工程上不要过早复杂化。对于很多移动端功能，几千到几万个片段已经足够提供明显价值。

本地检索通常还需要关键词检索配合向量检索。关键词检索适合精确词、编号、联系人、文件名；向量检索适合语义相近表达。混合策略比单一向量检索更稳。

## 6.6 查询与排序

查询流程可以这样设计：

1. 预处理用户查询。
2. 判断查询是否敏感。
3. 生成查询 embedding。
4. 从本地索引召回候选。
5. 结合关键词、时间、来源和权限排序。
6. 去重和过滤已删除内容。
7. 展示引用片段。
8. 可选：让云端基于用户确认的片段生成答案。

排序不能只看向量相似度。移动端个性化任务常需要时间和来源权重。例如最近笔记、收藏内容、用户常打开的文档，可能比相似度略高但很旧的片段更有用。

## 6.7 删除一致性

本地检索最容易被忽视的是删除。当用户删除笔记、撤销相册授权或清空聊天记录时，索引必须同步删除相关片段和向量。

建议采用软删除加后台清理：

- 业务数据删除时，立即把相关 chunk 标记为 deleted。
- 查询时过滤 deleted。
- 后台任务定期物理清理向量和文本。
- 清理完成后记录索引版本。

如果支持多设备同步，还要处理删除事件同步。不要让用户在 A 设备删除了内容，B 设备的本地索引仍然能搜索出来。

## 6.8 个性化与隐私边界

本地检索可以支撑个性化，但个性化不能变成不可见监控。建议遵守：

- 明确告诉用户哪些数据参与个性化。
- 提供关闭入口。
- 提供清空索引入口。
- 不把敏感片段写入崩溃日志或埋点。
- 上云增强前展示将使用的片段。
- 对企业或儿童场景使用更严格默认值。

个性化越强，越需要透明度。用户应该能理解“为什么推荐这个结果”，至少能看到引用来源。

## 6.9 个人知识助手案例

假设要做一个个人知识助手，支持搜索本地笔记并回答问题。可以采用：

```text
本地笔记 -> 分段 -> 本地 embedding -> 本地索引 -> 查询召回 -> 引用展示 -> 可选云端总结
```

关键设计：

| 模块 | 设计 |
| --- | --- |
| 授权 | 用户选择哪些笔记本参与索引 |
| 分段 | 按标题和段落分段，保留来源链接 |
| 索引 | 本地加密存储，记录模型版本 |
| 查询 | 混合关键词和向量召回 |
| 展示 | 先展示引用，再展示摘要 |
| 上云 | 用户确认后只上传选中的片段 |
| 删除 | 笔记删除后立即标记索引删除 |

这个产品不一定需要端侧大语言模型完整回答。第一阶段只做本地召回和引用展示，也能提供价值；第二阶段再加入本地短摘要；第三阶段再做云端增强。

## 6.10 验证清单

本地检索功能上线前建议检查：

| 项目 | 检查问题 |
| --- | --- |
| 授权 | 用户是否知道哪些数据被索引？ |
| 分段 | 召回结果是否有足够上下文？ |
| 删除 | 删除原文后是否无法再搜到？ |
| 存储 | 索引是否加密，大小是否可控？ |
| 性能 | 索引构建和查询是否影响前台体验？ |
| 离线 | 无网络时是否仍可用？ |
| 上云 | 云端增强前是否有用户确认？ |
| 观测 | 是否记录召回失败率和用户点击率？ |

## 本章小结

本地检索是端侧智能的重要组成部分。它的难点不只在算法，还在授权、分段、索引版本、删除一致性、加密存储、排序和用户透明度。一个好的本地检索架构，应先让用户数据在设备上产生价值，再谨慎决定哪些片段需要云端增强。
