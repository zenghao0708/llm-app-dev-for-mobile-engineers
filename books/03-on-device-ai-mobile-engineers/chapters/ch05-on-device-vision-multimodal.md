# 第 5 章 端侧视觉与多模态输入

## 本章导读

视觉和多模态能力让移动端 AI 更贴近真实用户场景。用户拿起手机拍照、截屏、扫描文档、浏览相册、识别商品、搜索图片，这些动作都天然发生在设备上。端侧视觉能力可以在上传前完成预处理、隐私遮挡、轻量识别和质量判断，让云端只处理真正需要复杂理解的部分。

视觉任务的难点是输入复杂。文本通常是字符串，视觉输入却包含尺寸、方向、色彩、EXIF、隐私区域、压缩质量、相册权限和内存峰值。一个模型在单张测试图片上表现好，不代表它能处理真实用户相册里的长图、截图、模糊照片、夜景、证件、二维码和多语言文档。

## 学习目标

- 理解端侧视觉任务的输入输出边界。
- 掌握图片权限、压缩、预处理和隐私处理要点。
- 知道多模态任务如何设计端云协同。
- 能为拍照、截图、相册和文档扫描场景设计稳定处理链路。

## 5.1 端侧视觉任务类型

移动端常见视觉任务包括：

- 图片质量判断：模糊、过暗、遮挡、反光。
- OCR 预处理：裁剪、矫正、增强、文字区域检测。
- 本地分类：票据、截图、证件、商品、文档、风景。
- 敏感区域检测：人脸、车牌、手机号、地址、二维码。
- 相册智能整理：按主题、时间、地点或场景聚类。
- 视觉搜索入口：先本地召回，再云端增强。
- 截图理解：识别页面类型、错误提示、可操作区域。
- 多模态助手：结合图片和文本做解释或建议。

这些任务不一定都要端侧完成。更常见的做法是本地做输入治理和轻量判断，云端做复杂推理。

## 5.2 图片输入链路

一个可发布的图片处理链路通常包括：

```text
选择/拍摄图片 -> 权限检查 -> 解码 -> 方向修正 -> 缩放/裁剪 -> 隐私处理 -> 本地模型 -> 结果展示/上云确认
```

每一步都可能影响结果。EXIF 中的方向信息如果处理不对，模型看到的图片可能是旋转的；图片过大如果直接解码，低端设备可能出现内存峰值；截图中如果包含手机号或聊天记录，上云前应做遮挡或用户确认。

移动端工程师要特别注意“解码尺寸”和“展示尺寸”不是一回事。用户看到的是屏幕上的缩略图，但程序如果按原图解码，内存可能突然增加。端侧视觉功能上线前，应在真实大图、长截图和低端设备上测试峰值内存。

## 5.3 权限与隐私边界

视觉输入通常涉及相册、相机和文件权限。产品设计上应区分：

- 单张图片选择：只处理用户明确选择的图片。
- 有限相册访问：只读取用户授权范围内的图片。
- 全量相册索引：需要更强理由和清晰说明。
- 相机实时分析：需要说明何时开始、何时停止。
- 截图分析：要避免把聊天、支付、地址等敏感内容直接上传。

端侧能力可以降低隐私风险，但不能自动解决合规问题。比如本地检测到图片中有人脸，并不代表可以无提示处理整本相册。移动端工程师应把权限说明、数据处理范围和日志策略一起设计。

## 5.4 多模态端云协同

多模态任务往往需要端云协同。一个截图解释功能可以这样拆分：

1. 本地读取用户选择的截图。
2. 本地 OCR 提取文字。
3. 本地检测手机号、邮箱、地址、头像等敏感区域。
4. 对敏感区域遮挡或替换成占位符。
5. 本地判断截图类型，例如错误页、订单页、聊天页。
6. 如果需要复杂解释，向用户说明将上传脱敏内容。
7. 云端生成解释或修复建议。
8. 本地展示引用和操作入口。

这个流程的重点不是“图片能不能理解”，而是“上传了什么、为什么上传、用户是否知道、失败时是否可用”。

## 5.5 文档扫描案例

文档扫描是端侧视觉的典型场景。它可以包含：

- 边缘检测。
- 透视矫正。
- 亮度和对比度增强。
- OCR。
- 本地分类。
- PDF 生成。
- 本地保存或云端同步。

工程上要注意：

| 问题 | 风险 | 处理 |
| --- | --- | --- |
| 图片过大 | 内存峰值过高 | 按目标尺寸解码 |
| 透视矫正失败 | 文档变形 | 允许用户手动调整角点 |
| OCR 错误 | 识别结果误导用户 | 标注可编辑、保留原图 |
| 证件类图片 | 隐私敏感 | 默认本地处理，上传需确认 |
| 多页文档 | 处理时间长 | 分页处理，可取消 |

文档扫描的用户体验非常依赖可控性。模型自动处理失败时，要允许用户调整，而不是强迫接受结果。

## 5.6 相册智能整理案例

相册整理适合端侧优先，因为图片属于高度个人化数据。一个本地相册整理功能可以做：

- 本地缩略图特征提取。
- 场景分类。
- 重复或相似图片检测。
- 模糊图片识别。
- 本地聚类。
- 用户手动确认删除或归档。

这里的高风险动作是“删除”。即使模型判断图片模糊，也不应自动删除。更合理的做法是生成候选集合，让用户确认。

还要考虑增量更新。用户每天新增图片，如果每次都全量扫描，相册越大性能越差。应记录已处理资源 id、模型版本、特征版本和处理时间，新增或变更时增量处理。

## 5.7 截图理解案例

截图理解在移动端 AI 产品中很常见，例如帮助用户解释错误、提取订单信息、生成待办或诊断页面问题。它的输入看似简单，实际上非常敏感。

建议采用三层策略：

| 层级 | 处理 | 目的 |
| --- | --- | --- |
| 本地基础处理 | OCR、页面类型、敏感区域检测 | 减少上传原始截图 |
| 用户确认 | 展示将使用的文字片段或遮挡结果 | 建立透明度 |
| 云端增强 | 复杂解释、知识补充、生成建议 | 提升能力 |

如果截图来自其他 App，尤其要谨慎。它可能包含聊天、支付、健康、位置或企业信息。端侧 AI 的优势是可以在上传前先判断和脱敏。

## 5.8 验证清单

端侧视觉与多模态能力上线前建议检查：

| 项目 | 检查问题 |
| --- | --- |
| 输入 | 拍照、相册、截图、文件来源是否都覆盖？ |
| 尺寸 | 大图、长截图、低端设备是否稳定？ |
| 方向 | EXIF 方向是否正确处理？ |
| 隐私 | 人脸、车牌、手机号、地址是否有策略？ |
| 权限 | 权限拒绝、有限相册授权是否可用？ |
| 取消 | 用户取消处理时是否释放资源？ |
| 端云 | 上传前是否有脱敏和用户确认？ |
| 结果 | 错误识别是否有人工修正入口？ |

## 5.9 图片处理状态机

视觉任务建议使用状态机管理：

```text
idle
permission_checking
image_selected
decoding
preprocessing
local_inference
privacy_review
need_user_confirmation
cloud_processing
showing_result
failed
cancelled
```

状态机能避免两个常见问题：一是用户取消后任务仍在后台处理，二是图片处理失败后 UI 不知道应该展示什么。对于多页文档和相册整理，还要支持 batch 状态：当前处理第几张、失败了几张、是否可继续。

## 5.10 输入预处理规范

视觉任务的预处理要写成明确规范：

| 项目 | 规范 |
| --- | --- |
| 解码 | 按目标尺寸解码，避免直接原图入内存 |
| 方向 | 统一处理 EXIF 方向 |
| 色彩 | 明确 RGB、灰度或其他输入格式 |
| 压缩 | 云端上传前控制大小和质量 |
| 裁剪 | 保留用户可调整入口 |
| 隐私 | 上云前遮挡敏感区域 |
| 临时文件 | 任务结束后清理 |

模型训练和客户端预处理必须一致。训练时使用中心裁剪，客户端却使用拉伸缩放，会让模型效果明显退化。

## 5.11 多模态评测样例

多模态评测集应覆盖真实移动端输入：

- 竖图和横图。
- 长截图。
- 模糊图片。
- 低光图片。
- 带 EXIF 旋转的图片。
- 包含手机号、地址、头像和二维码的截图。
- 多语言文档。
- 用户手动裁剪失败的文档。

每个样本应有期望结果：

```text
sample_id
input_type
expected_task
privacy_regions
expected_text
should_upload
expected_fallback
```

评测不只是准确率，还要看隐私漏检、误检、处理耗时、内存峰值和用户可修正性。

## 5.12 视觉结果的解释

视觉模型输出要尽量给用户依据。例如：

- OCR 结果显示原图高亮区域。
- 文档扫描显示自动识别的四个角点。
- 相册整理显示相似图片分组依据。
- 截图解释展示使用了哪些文字片段。

如果用户无法理解模型为什么给出结果，就很难信任它。端侧视觉功能尤其要避免“黑盒自动处理后直接提交”。提供预览和手动修正入口，是移动端体验的基本要求。

## 5.13 异步任务与资源释放

视觉任务通常比文本任务更重。工程上要保证：

- 页面销毁时取消任务。
- 取消后释放图片缓冲和模型输入。
- 大图处理使用后台线程。
- 多张图片分批处理。
- 低内存时停止非关键任务。
- 失败后删除临时文件。

这些细节不会提升模型准确率，但会决定功能是否能长期稳定运行。

## 5.14 实时相机能力

实时相机分析和单张图片处理不同。相机预览帧会持续到来，如果每帧都跑模型，设备很快会发热、掉帧或耗电。实时能力要先定义帧率和触发策略：

| 策略 | 说明 |
| --- | --- |
| 间隔采样 | 每隔固定时间处理一帧 |
| 事件触发 | 检测到画面稳定或用户按下按钮后处理 |
| 降分辨率 | 预览分析使用低分辨率，保存时使用原图 |
| 早停 | 结果稳定后停止重复推理 |
| 热降级 | 发热或低电量时降低频率 |

例如文档扫描中的边缘检测，可以在预览阶段低频运行，只给用户框线提示；真正生成高清扫描件时，再对拍摄图做完整处理。不要把预览链路和最终处理链路混在一起。

实时相机还要清楚提示用户何时正在分析画面。对于人脸、证件、二维码和屏幕内容，实时分析会带来更强隐私感知，UI 上应有可见状态和关闭入口。

## 5.15 多页文档与批处理

多页文档扫描、相册整理和批量截图处理都属于批处理任务。批处理的难点不是单张效果，而是进度、取消、失败恢复和资源控制。

建议批处理任务记录：

```text
batch_id
total_items
processed_items
failed_items
current_stage
can_pause
can_resume
created_at
model_version
```

批处理中某一页失败，不应让整个任务丢失。可以把失败项标记出来，让用户重拍、跳过或手动修正。对于多页文档，用户更关心最后文档是否完整，而不是每页都自动处理成功。

批处理还要控制内存。不要一次性解码全部图片，也不要把所有中间结果保存在内存中。更稳妥的方式是逐页处理、逐页落盘、最终合并，并在取消时清理临时文件。

## 5.16 隐私遮挡验收

视觉隐私遮挡需要专门验收。不能只看模型检测框是否大致命中，还要看遮挡后的内容是否真的不可识别。

验收项包括：

| 项目 | 检查 |
| --- | --- |
| 漏检 | 手机号、头像、地址、二维码是否遗漏 |
| 误检 | 普通文字或图标是否被过度遮挡 |
| 遮挡强度 | 模糊后是否仍能辨认 |
| 坐标准确 | 遮挡区域是否覆盖完整敏感内容 |
| 缩放一致 | 预览图和上传图遮挡位置是否一致 |
| 用户确认 | 上传前是否展示脱敏预览 |

视觉遮挡不要只在展示层完成。如果上传的是原图，而 UI 只显示遮挡预览，就会造成严重风险。工程上要确认上传 payload 使用的就是遮挡后的图像或脱敏后的文本。

## 5.17 多模态输入契约

多模态任务要定义输入契约。一个截图解释请求可以包含：

```json
{
  "image_ref": "local-file-handle",
  "image_type": "screenshot",
  "ocr_text_available": true,
  "privacy_review_status": "redacted",
  "user_question": "这个错误怎么处理？",
  "allow_cloud": true
}
```

契约要避免把大图片直接传给共享层或云端接口。移动端内部可以使用文件句柄、asset id 或临时路径；真正上云前再生成经过压缩和脱敏的 payload。

输出契约也要结构化：

```json
{
  "status": "success",
  "used_text_regions": [],
  "privacy_warnings": [],
  "answer": "...",
  "fallback_available": true
}
```

这样 UI 可以稳定展示引用区域、隐私提示和降级结果，而不是解析一段自然语言。

## 5.18 视觉任务的发布分层

视觉能力适合按风险分层发布：

| 阶段 | 能力 | 风险 |
| --- | --- | --- |
| 第一阶段 | 本地图片质量判断、模糊检测 | 低 |
| 第二阶段 | OCR、文档裁剪、截图类型识别 | 中 |
| 第三阶段 | 隐私遮挡和脱敏预览 | 中到高 |
| 第四阶段 | 用户确认后的云端解释 | 高 |
| 第五阶段 | 相册批量整理和删除建议 | 高 |

每一阶段都要有自己的指标和回滚方式。不要在第一版同时上线“批量相册整理 + 自动删除建议 + 云端解释”。视觉输入包含大量私人内容，发布节奏要比普通 UI 功能更谨慎。

## 本章小结

多模态能力不能只看模型效果。移动端工程师要把输入采集、解码尺寸、EXIF、权限、隐私、性能和失败处理放到同等重要的位置。端侧视觉的价值不仅是本地识别，更是为云端复杂理解建立安全、可控、可解释的输入边界。
