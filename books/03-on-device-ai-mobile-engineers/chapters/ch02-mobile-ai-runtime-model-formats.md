# 第 2 章 移动端 AI Runtime 与模型格式

## 本章导读

端侧 AI 需要 runtime、模型格式和硬件加速共同工作。工程师不一定要训练模型，但必须理解模型如何被加载、执行、量化、缓存和更新。否则，模型文件只是一个黑盒：能跑时看起来很神奇，不能跑时很难定位原因。

本章从移动端工程角度介绍 runtime 和模型格式。重点不是比较谁绝对更好，而是建立选型和排查框架。

## 学习目标

- 理解 Core ML、TensorFlow Lite、ONNX Runtime 等 runtime 的定位。
- 知道模型格式、量化和硬件后端对移动端体验的影响。
- 能为项目建立 runtime 选择清单。
- 理解模型加载、推理、缓存和更新的基本链路。

## 2.1 Runtime 是什么

Runtime 是模型在设备上执行的环境。它负责加载模型、准备输入、执行算子、调用硬件后端、返回输出，并处理内存和线程等底层细节。

移动端工程师可以把 runtime 类比成“模型的执行引擎”。同一个模型思路，落到不同 runtime 上，会遇到不同问题：

- 支持的算子不同。
- 模型格式不同。
- 量化方式不同。
- 硬件加速后端不同。
- 调试和 profiling 工具不同。
- iOS/Android 集成方式不同。

因此，端侧 AI 选型时不能只问“模型效果如何”，还要问“这个模型能否稳定进入目标 runtime”。

## 2.2 常见 Runtime 类型

| Runtime | 常见平台 | 适合场景 | 关注点 |
| --- | --- | --- | --- |
| Core ML | iOS、macOS | Apple 平台视觉、文本、小模型推理 | 模型转换、Neural Engine、系统版本 |
| TensorFlow Lite | Android、跨平台 | 移动端轻量模型、视觉、语音 | delegate、量化、设备兼容 |
| ONNX Runtime | 跨平台 | 跨框架模型部署 | 包体、算子支持、平台集成 |
| MediaPipe 等任务库 | 移动端多媒体任务 | 视觉、手势、姿态、音频 | 任务抽象好，但定制边界需确认 |
| 自研或第三方 LLM runtime | iOS/Android | 本地 LLM、embedding、tokenizer | 内存、KV cache、量化格式、性能 |

正式出版前，本书会按官方文档重新核验这些 runtime 的最新能力、命令和支持边界。读者在真实项目中也应以官方文档和自己的目标设备测试为准。

## 2.3 模型格式与转换

模型通常不会直接以训练框架原始格式进入移动端。常见流程是：

1. 训练或获取模型。
2. 导出中间格式。
3. 转换为目标 runtime 格式。
4. 量化或压缩。
5. 在目标设备验证效果和性能。
6. 集成到 App。

模型转换可能带来效果差异。某些算子不支持，需要替换；某些动态 shape 需要固定；某些量化方式会影响精度。移动端工程师至少要保留转换记录：

```text
模型名称：
来源版本：
转换脚本：
输入 shape：
量化方式：
目标 runtime：
验证数据集：
转换日期：
负责人：
```

没有这些记录，后续模型效果变化、线上问题和回滚都会很难追踪。

## 2.4 量化的工程意义

量化是端侧部署中的常见手段。它通过降低数值精度来减少模型体积和计算成本。工程上，量化通常带来：

- 包体下降。
- 内存占用下降。
- 推理速度提升。
- 能耗下降。

但量化也可能带来：

- 准确率下降。
- 边界样本表现变差。
- 不同设备后端表现不同。
- 调试更困难。

因此，量化不是“越低越好”。移动端项目需要建立量化前后评测表：

| 指标 | 原模型 | 量化模型 | 是否可接受 |
| --- | --- | --- | --- |
| 模型大小 | 待填 | 待填 | 待填 |
| 平均延迟 | 待填 | 待填 | 待填 |
| 峰值内存 | 待填 | 待填 | 待填 |
| 准确率 | 待填 | 待填 | 待填 |
| 关键样本失败数 | 待填 | 待填 | 待填 |

没有评测表的量化，只是压缩文件，不是工程优化。

## 2.5 模型加载链路

端侧模型通常要经过：

1. 检查模型是否存在。
2. 校验版本和完整性。
3. 加载 runtime。
4. 加载模型文件。
5. 初始化 tokenizer 或预处理器。
6. 预热或懒加载。
7. 执行推理。
8. 释放或缓存资源。

每一步都可能失败。App 应定义错误类型：

- 模型文件不存在。
- 校验失败。
- runtime 初始化失败。
- 设备不支持硬件加速。
- 内存不足。
- 推理超时。
- 输出解析失败。

这些错误不能只写进日志。用户体验、降级策略和线上观测都需要它们。

## 2.6 Runtime 选择清单

选型时建议填写：

| 维度 | 问题 |
| --- | --- |
| 平台 | 只支持 iOS，还是 iOS/Android 都要支持？ |
| 设备 | 最低系统版本和最低硬件是什么？ |
| 模型 | 文本、语音、视觉、embedding 还是 LLM？ |
| 性能 | 首次加载、单次推理、峰值内存预算是多少？ |
| 包体 | 模型和 runtime 允许增加多少 MB？ |
| 更新 | 模型随包发布还是远程下载？ |
| 调试 | 是否能定位加载失败和推理失败？ |
| 法务 | runtime 和模型许可证是否可用？ |

选型记录应进入仓库文档，避免半年后没人知道当初为什么选这个 runtime。

## 2.7 移动端集成建议

Runtime 不应直接散落在 UI 层。建议分层：

```text
UI -> ViewModel -> AiUseCase -> ModelRuntime -> ModelFileManager
```

其中：

- UI 只关心 loading、结果和错误。
- ViewModel 负责状态和取消。
- UseCase 负责业务任务。
- Runtime 封装推理细节。
- ModelFileManager 负责版本、校验、下载和缓存。

这种分层能让模型更容易替换，也能让测试更清晰。

## 2.8 Runtime 排障方法

端侧模型失败时，不要直接判断“模型不行”。更有效的排障顺序是从外到内：

1. 模型文件是否存在。
2. 文件完整性是否通过校验。
3. App 版本和模型版本是否兼容。
4. runtime 是否初始化成功。
5. 硬件加速后端是否可用。
6. 输入 shape 和类型是否正确。
7. tokenizer 或预处理是否和训练时一致。
8. 输出解析是否正确。
9. 推理结果是否在评测集上退化。

很多问题并不在模型本身。例如 tokenizer 版本不一致会让文本模型输出异常；图片预处理顺序变化会让视觉模型准确率下降；量化模型在某些设备后端上表现不同；模型文件下载中断但没有校验，会导致加载失败。

建议定义统一错误码：

| 错误码 | 含义 |
| --- | --- |
| model_missing | 模型文件不存在 |
| checksum_failed | 完整性校验失败 |
| runtime_init_failed | runtime 初始化失败 |
| unsupported_device | 当前设备不支持 |
| input_invalid | 输入格式、shape 或类型错误 |
| inference_timeout | 推理超时 |
| output_invalid | 输出解析失败 |

错误码要同时服务于用户提示、日志和线上观测。

## 2.9 模型转换流水线

模型转换不应靠个人手工操作。建议把转换过程脚本化，并保留输入输出记录。

一个基础流水线：

```text
source_model
  -> export
  -> convert
  -> quantize
  -> validate
  -> package
  -> publish_candidate
```

每一步都应产出可追踪信息：

- 源模型版本。
- 转换脚本版本。
- 转换工具版本。
- 量化配置。
- 评测集结果。
- 目标 runtime。
- 目标平台。
- 产物 checksum。

这样模型出问题时，团队能知道是模型变化、转换工具变化、量化配置变化，还是客户端加载逻辑变化。

## 2.10 输入输出契约

模型进入 App 前，需要定义输入输出契约。契约应该比“传一个字符串，返回一个结果”更具体。

例如本地意图识别：

```json
{
  "input": {
    "text": "打开收藏",
    "locale": "zh-CN",
    "max_length": 128
  },
  "output": {
    "intent": "open_favorites",
    "confidence": 0.92,
    "slots": {}
  }
}
```

契约要定义：

- 输入最大长度。
- 支持语言。
- 特殊字符处理。
- 输出枚举。
- 置信度范围。
- 低置信度策略。
- 错误类型。

没有契约，客户端、模型团队和服务端很容易各自理解不同。

## 2.11 Runtime 选择案例

假设团队要做本地截图敏感区域检测。选择 runtime 时可以这样评估：

| 维度 | 问题 |
| --- | --- |
| 平台 | iOS 和 Android 是否都支持？ |
| 模型类型 | 是否包含 OCR、视觉检测或分类？ |
| 输入 | 图片尺寸是否固定？是否支持批处理？ |
| 性能 | 低端设备是否能在预算内完成？ |
| 包体 | runtime 和模型总大小是否可接受？ |
| 调试 | 是否能输出每阶段耗时和失败原因？ |
| 更新 | 模型是否需要远程灰度？ |

如果 iOS 和 Android 选择不同 runtime，共享层就不能假设输出完全一致，而要定义统一结果格式和错误模型。

## 2.12 Runtime 生命周期管理

Runtime 不是一次初始化后永远可用的对象。移动端要考虑生命周期：

```text
not_initialized
  -> initializing
  -> ready
  -> running
  -> idle
  -> releasing
  -> failed
```

每个状态都需要明确规则：

| 状态 | 规则 |
| --- | --- |
| initializing | 不能重复初始化，UI 展示加载状态 |
| ready | 可接受任务，但要检查模型版本 |
| running | 支持取消、超时和资源监控 |
| idle | 可保留缓存，也可在内存压力下释放 |
| releasing | 停止新任务，释放模型和中间缓冲 |
| failed | 记录错误码，进入降级或重试 |

生命周期管理能避免常见问题：页面退出后推理还在跑、重复加载多个模型实例、内存压力下不释放、App 进入后台后继续重任务。端侧 AI 的稳定性，很大程度来自这些普通但严格的状态管理。

## 2.13 线程与任务队列

模型推理不应阻塞 UI 线程，但这只是最低要求。实际项目还要设计任务队列：

- 同一模型是否允许并发推理。
- 多个模型任务如何排队。
- 长任务能否被新任务取消。
- 页面销毁后任务如何停止。
- 后台任务如何受系统限制。
- 低电量和发热时是否暂停。

一个简单策略是按任务风险和交互优先级分队列：

| 队列 | 示例 | 策略 |
| --- | --- | --- |
| 交互队列 | 输入联想、语音命令 | 低延迟，旧任务可取消 |
| 前台队列 | 截图解释、文档扫描 | 用户可见，可展示进度 |
| 后台队列 | 相册索引、本地重建 | 分批执行，可暂停恢复 |
| 维护队列 | 模型下载、索引迁移 | 受网络、电量和存储约束 |

队列设计要和 runtime 生命周期打通。任务取消时，不只是 UI 不显示结果，还要尽可能停止推理、释放输入缓冲、清理临时文件，并避免结果回调写入已销毁页面。

## 2.14 模型包验收

模型文件进入 App 前，应通过模型包验收。验收对象不是单个权重文件，而是完整 bundle：

```text
model_bundle/
  manifest.json
  model.bin
  tokenizer.json
  preprocessor.json
  labels.json
  checksums.txt
  smoke_tests.jsonl
  license.txt
```

验收表可以包括：

| 项目 | 验收标准 |
| --- | --- |
| 文件完整 | manifest 声明的文件都存在 |
| 校验通过 | checksum 与实际文件一致 |
| 版本完整 | model、runtime、schema、index 版本都有记录 |
| 冒烟成功 | 固定样本能输出合法结构 |
| 性能达标 | 目标设备上加载和推理低于预算 |
| 授权清楚 | license 和来源可追溯 |
| 回滚可行 | 上一稳定版本仍可加载 |

这张表能阻止“模型文件丢给客户端就算完成”的协作方式。对移动端来说，模型包是要发布、观测和回滚的产品资产。

## 2.15 硬件加速失败处理

硬件加速不是总能成功。某些设备可能不支持特定算子，某些系统版本可能存在 delegate 问题，某些模型在 GPU/NPU 上反而更慢。工程上必须把硬件加速作为可失败能力处理。

常见策略：

1. 启动时查询可用后端。
2. 优先尝试高性能后端。
3. 初始化失败时回退 CPU 或轻量模型。
4. 记录失败原因和设备信息。
5. 通过远程配置禁用问题设备的后端。

不要把 delegate 初始化失败直接变成用户可见崩溃。用户关心的是功能能否继续使用，而不是底层使用 CPU、GPU 还是 NPU。只要性能和功耗可接受，回退路径就是有效方案。

## 2.16 Debug 信息设计

端侧 AI 排障需要诊断信息，但不能记录敏感输入。建议 debug 信息分两类。

开发调试信息：

- 模型路径。
- 输入 shape。
- 后端类型。
- 每阶段耗时。
- 中间 tensor 统计。
- 输出结构。

线上诊断信息：

- 模型版本。
- runtime 版本。
- 设备等级。
- 输入大小区间。
- 加载耗时。
- 推理耗时。
- 错误码。
- 降级原因。

线上诊断不应包含原始文本、图片、音频、OCR 全文、文件路径和用户标识。需要真实样本时，应通过用户授权反馈通道单独收集，并设置过期和访问控制。

## 2.17 版本矩阵

端侧 AI 至少涉及四类版本：

| 版本 | 说明 |
| --- | --- |
| App 版本 | 客户端代码和 UI |
| Runtime 版本 | 推理引擎和后端 |
| Model 版本 | 权重、tokenizer 和配置 |
| Schema 版本 | 输入输出和索引结构 |

线上问题定位时要能同时看到这些版本。比如某个模型在 App 6.2.0 正常，在 App 6.3.0 异常，可能是预处理逻辑变了；同一模型在不同 runtime 版本表现不同，可能是后端优化或算子行为变化；embedding 模型版本变化后检索变差，可能是索引没有重建。

版本矩阵建议进入观测字段和发布清单。没有版本维度，端侧 AI 的线上问题会非常难排查。

## 本章小结

模型格式和 runtime 是端侧 AI 的执行底座。选型时要同时考虑能力、性能、更新、调试和团队维护成本。端侧模型落地不是把文件放进工程，而是建立加载、校验、推理、失败处理和版本治理链路。
