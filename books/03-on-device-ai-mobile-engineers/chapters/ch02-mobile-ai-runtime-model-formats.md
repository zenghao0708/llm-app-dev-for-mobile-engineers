# 第 2 章 移动端 AI Runtime 与模型格式

## 本章导读

端侧 AI 需要 runtime、模型格式和硬件加速共同工作。工程师不一定要训练模型，但必须理解模型如何被加载、执行、量化、缓存和更新。否则，模型文件只是一个黑盒：能跑时看起来很神奇，不能跑时很难定位原因。

本章从移动端工程角度介绍 runtime 和模型格式。重点不是比较谁绝对更好，而是建立选型和排查框架。

## 学习目标

- 理解 Core ML、TensorFlow Lite、ONNX Runtime 等 runtime 的定位。
- 知道模型格式、量化和硬件后端对移动端体验的影响。
- 能为项目建立 runtime 选择清单。
- 理解模型加载、推理、缓存和更新的基本链路。

## 2.1 Runtime 是什么

Runtime 是模型在设备上执行的环境。它负责加载模型、准备输入、执行算子、调用硬件后端、返回输出，并处理内存和线程等底层细节。

移动端工程师可以把 runtime 类比成“模型的执行引擎”。同一个模型思路，落到不同 runtime 上，会遇到不同问题：

- 支持的算子不同。
- 模型格式不同。
- 量化方式不同。
- 硬件加速后端不同。
- 调试和 profiling 工具不同。
- iOS/Android 集成方式不同。

因此，端侧 AI 选型时不能只问“模型效果如何”，还要问“这个模型能否稳定进入目标 runtime”。

## 2.2 常见 Runtime 类型

| Runtime | 常见平台 | 适合场景 | 关注点 |
| --- | --- | --- | --- |
| Core ML | iOS、macOS | Apple 平台视觉、文本、小模型推理 | 模型转换、Neural Engine、系统版本 |
| TensorFlow Lite | Android、跨平台 | 移动端轻量模型、视觉、语音 | delegate、量化、设备兼容 |
| ONNX Runtime | 跨平台 | 跨框架模型部署 | 包体、算子支持、平台集成 |
| MediaPipe 等任务库 | 移动端多媒体任务 | 视觉、手势、姿态、音频 | 任务抽象好，但定制边界需确认 |
| 自研或第三方 LLM runtime | iOS/Android | 本地 LLM、embedding、tokenizer | 内存、KV cache、量化格式、性能 |

正式出版前，本书会按官方文档重新核验这些 runtime 的最新能力、命令和支持边界。读者在真实项目中也应以官方文档和自己的目标设备测试为准。

## 2.3 模型格式与转换

模型通常不会直接以训练框架原始格式进入移动端。常见流程是：

1. 训练或获取模型。
2. 导出中间格式。
3. 转换为目标 runtime 格式。
4. 量化或压缩。
5. 在目标设备验证效果和性能。
6. 集成到 App。

模型转换可能带来效果差异。某些算子不支持，需要替换；某些动态 shape 需要固定；某些量化方式会影响精度。移动端工程师至少要保留转换记录：

```text
模型名称：
来源版本：
转换脚本：
输入 shape：
量化方式：
目标 runtime：
验证数据集：
转换日期：
负责人：
```

没有这些记录，后续模型效果变化、线上问题和回滚都会很难追踪。

## 2.4 量化的工程意义

量化是端侧部署中的常见手段。它通过降低数值精度来减少模型体积和计算成本。工程上，量化通常带来：

- 包体下降。
- 内存占用下降。
- 推理速度提升。
- 能耗下降。

但量化也可能带来：

- 准确率下降。
- 边界样本表现变差。
- 不同设备后端表现不同。
- 调试更困难。

因此，量化不是“越低越好”。移动端项目需要建立量化前后评测表：

| 指标 | 原模型 | 量化模型 | 是否可接受 |
| --- | --- | --- | --- |
| 模型大小 | 待填 | 待填 | 待填 |
| 平均延迟 | 待填 | 待填 | 待填 |
| 峰值内存 | 待填 | 待填 | 待填 |
| 准确率 | 待填 | 待填 | 待填 |
| 关键样本失败数 | 待填 | 待填 | 待填 |

没有评测表的量化，只是压缩文件，不是工程优化。

## 2.5 模型加载链路

端侧模型通常要经过：

1. 检查模型是否存在。
2. 校验版本和完整性。
3. 加载 runtime。
4. 加载模型文件。
5. 初始化 tokenizer 或预处理器。
6. 预热或懒加载。
7. 执行推理。
8. 释放或缓存资源。

每一步都可能失败。App 应定义错误类型：

- 模型文件不存在。
- 校验失败。
- runtime 初始化失败。
- 设备不支持硬件加速。
- 内存不足。
- 推理超时。
- 输出解析失败。

这些错误不能只写进日志。用户体验、降级策略和线上观测都需要它们。

## 2.6 Runtime 选择清单

选型时建议填写：

| 维度 | 问题 |
| --- | --- |
| 平台 | 只支持 iOS，还是 iOS/Android 都要支持？ |
| 设备 | 最低系统版本和最低硬件是什么？ |
| 模型 | 文本、语音、视觉、embedding 还是 LLM？ |
| 性能 | 首次加载、单次推理、峰值内存预算是多少？ |
| 包体 | 模型和 runtime 允许增加多少 MB？ |
| 更新 | 模型随包发布还是远程下载？ |
| 调试 | 是否能定位加载失败和推理失败？ |
| 法务 | runtime 和模型许可证是否可用？ |

选型记录应进入仓库文档，避免半年后没人知道当初为什么选这个 runtime。

## 2.7 移动端集成建议

Runtime 不应直接散落在 UI 层。建议分层：

```text
UI -> ViewModel -> AiUseCase -> ModelRuntime -> ModelFileManager
```

其中：

- UI 只关心 loading、结果和错误。
- ViewModel 负责状态和取消。
- UseCase 负责业务任务。
- Runtime 封装推理细节。
- ModelFileManager 负责版本、校验、下载和缓存。

这种分层能让模型更容易替换，也能让测试更清晰。

## 本章小结

模型格式和 runtime 是端侧 AI 的执行底座。选型时要同时考虑能力、性能、更新、调试和团队维护成本。端侧模型落地不是把文件放进工程，而是建立加载、校验、推理、失败处理和版本治理链路。
