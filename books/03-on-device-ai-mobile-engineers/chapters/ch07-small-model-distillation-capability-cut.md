# 第 7 章 小模型、蒸馏与能力裁剪

## 本章导读

端侧模型通常需要更小、更快、更省电。移动设备没有无限内存，也不能长时间高负载运行。一个云端模型可以追求通用能力，一个端侧模型更应该追求“在明确任务上足够好”。这就需要能力裁剪、量化、蒸馏、任务专用模型和严格评测共同工作。

本章讨论的重点不是训练算法细节，而是移动端工程师如何定义模型边界。很多端侧 AI 项目失败，并不是模型不够先进，而是任务定义过大：既想聊天、又想总结、又想搜索、又想识图，还要求离线、低延迟、低功耗。端侧模型设计的第一原则是收缩问题。

## 学习目标

- 区分通用大模型和任务专用小模型。
- 理解蒸馏、量化和裁剪的工程意义。
- 能为移动端场景制定模型能力边界。
- 能用评测数据判断模型是否足够进入 App。

## 7.1 小模型不是低配大模型

小模型不应该被理解为“大模型的缩水版”。端侧小模型更像一个任务专家：只解决清晰边界内的问题，并把不能解决的问题交给传统逻辑、用户确认或云端模型。

例如：

- 输入法联想不需要完整对话能力。
- 离线语音命令不需要开放式问答。
- 截图敏感区域检测不需要长文本生成。
- 本地笔记分类不需要最新互联网知识。
- 相册模糊检测不需要多轮推理。

如果任务可以被拆成分类、提取、排序、过滤、召回、短摘要或意图识别，小模型往往比通用大模型更适合端侧部署。

## 7.2 能力裁剪方法

能力裁剪是把模糊需求变成可执行边界。可以按下面步骤进行：

1. 写出用户任务，而不是模型能力。
2. 列出必须支持的输入类型。
3. 列出明确不支持的输入类型。
4. 定义输出格式。
5. 定义置信度和拒答策略。
6. 定义失败时的降级路径。

以“本地语音命令”为例：

| 项目 | 裁剪结果 |
| --- | --- |
| 支持 | 新建笔记、开始录音、暂停、继续、保存、搜索 |
| 不支持 | 任意聊天、复杂设置修改、跨 App 操作 |
| 输出 | command、slot、confidence |
| 拒绝 | 置信度低于阈值时要求用户重说 |
| 降级 | 展示文字按钮或搜索入口 |

裁剪之后，模型目标会清楚很多：它不是“理解所有语音”，而是“稳定识别少量命令”。

## 7.3 蒸馏的工程意义

蒸馏可以理解为让小模型学习大模型或强模型的行为。对移动端团队来说，蒸馏的价值通常在于：

- 用更小模型完成固定任务。
- 保持端侧低延迟。
- 降低模型包体和内存。
- 减少云端调用成本。
- 在隐私场景中减少上传。

但蒸馏不是自动成功。工程上要关注：

- 教师模型输出是否可靠。
- 训练样本是否覆盖真实移动端输入。
- 小模型是否只学到表面模式。
- 边界样本是否明显退化。
- 蒸馏后是否仍能解释错误。

如果教师模型在某类任务上本身不稳定，蒸馏只会把不稳定复制到端侧。移动端团队需要准备真实样本和人工校验样本，而不是只用合成数据。

## 7.4 量化与精度取舍

量化通过降低数值精度来减少模型大小和计算成本。移动端常见收益是包体下降、内存下降、速度提升和能耗降低。但量化可能改变模型行为，尤其是在边界输入上。

量化前后必须评测：

| 指标 | 关注点 |
| --- | --- |
| 模型大小 | 是否满足包体或下载预算 |
| 首次加载 | 是否减少冷启动等待 |
| 单次延迟 | 是否达到交互要求 |
| 峰值内存 | 是否降低 OOM 风险 |
| 准确率 | 主指标是否下降 |
| 边界样本 | 弱网、噪声、多语言、低质量图片是否退化 |

不要只看平均准确率。端侧产品中，最容易引发用户不满的是边界场景：噪声环境下误触发语音命令、证件照 OCR 错误、删除建议误判、敏感内容漏检。

## 7.5 任务专用模型组合

端侧 AI 不一定只有一个模型。很多产品更适合使用多个小模型组合：

```text
输入 -> 轻量分类 -> 专用模型 -> 规则校验 -> 业务动作
```

例如截图理解可以由多个能力组成：

- OCR 模型提取文字。
- 敏感区域检测模型遮挡隐私。
- 页面分类模型判断截图类型。
- 云端模型在用户确认后做复杂解释。

这种组合比一个端侧通用大模型更容易控制。每个模型的输入、输出和评测指标都更清楚，也更容易回滚。

## 7.6 置信度和拒答

端侧模型必须有“不知道”的能力。很多移动端任务连接真实操作，如果模型低置信度仍然输出确定结果，就会造成误导。

建议设计三段式策略：

| 置信度 | 动作 |
| --- | --- |
| 高 | 自动展示建议或执行低风险动作 |
| 中 | 展示候选，让用户确认 |
| 低 | 拒绝执行，提示用户换一种输入 |

例如语音识别“删除这条记录”属于高风险动作，即使置信度高，也应确认。置信度不是唯一决策依据，还要结合风险等级。

## 7.7 模型选择评审表

模型进入移动端工程前，建议填写：

| 项目 | 问题 |
| --- | --- |
| 任务边界 | 模型具体解决什么问题？不解决什么问题？ |
| 输入范围 | 支持哪些语言、长度、图片尺寸、音频场景？ |
| 输出格式 | 是否结构化？是否可校验？ |
| 性能预算 | 加载、延迟、内存、功耗预算是多少？ |
| 评测集 | 是否包含真实设备和真实用户样本？ |
| 降级 | 模型失败时 App 如何继续可用？ |
| 更新 | 模型如何灰度、回滚和兼容旧数据？ |
| 风险 | 是否连接删除、支付、权限、发布等高风险动作？ |

这张表的目的不是增加流程负担，而是避免“模型已经有了，先接进去看看”的随意集成。

## 7.8 案例：本地敏感信息检测

假设 App 需要在截图上传前检测手机号、邮箱、地址和头像。团队可以选择端侧小模型加规则：

1. OCR 提取文字。
2. 规则识别手机号、邮箱和证件号。
3. 小模型检测模糊地址和人名。
4. 视觉模型检测头像或人脸区域。
5. 合并结果并遮挡。
6. 用户确认后再上传。

这里不需要一个通用多模态大模型在端侧完整解释截图。能力裁剪后，任务变成“上传前隐私防护”，评测也更清楚：漏检率、误检率、遮挡位置、处理延迟和用户取消率。

## 7.9 小结前检查

端侧小模型上线前可以问：

- 是否存在更简单的规则或传统算法？
- 是否明确了不支持场景？
- 是否有拒答和用户确认？
- 是否有量化前后评测？
- 是否覆盖低端设备和边界样本？
- 是否能独立更新或回滚？
- 是否会让用户误以为能力更强？

这些问题能帮助团队保持克制。端侧 AI 的价值来自稳定解决具体问题，而不是展示模型存在感。

## 7.10 模型选型矩阵

模型选型不能只比较榜单分数。移动端团队更应该把候选模型放到同一个工程矩阵中评估：

| 维度 | 需要回答的问题 |
| --- | --- |
| 能力 | 是否覆盖当前任务的主路径和边界输入？ |
| 尺寸 | 权重、tokenizer、配置和 runtime 合计多大？ |
| 速度 | 首次加载和 P95 推理延迟是否满足交互预算？ |
| 内存 | 峰值内存是否能覆盖目标低端设备？ |
| 量化 | 量化后主指标和边界指标下降多少？ |
| 授权 | 模型许可证是否允许当前产品形态使用？ |
| 可解释 | 错误能否通过样本、日志和中间结果定位？ |
| 可更新 | 是否支持独立灰度、回滚和兼容旧数据？ |

一个常见误区是选择“能力最强”的模型，然后再想办法压缩。更稳妥的顺序是先确定任务、输入、输出和预算，再选择刚好覆盖边界的模型。端侧模型越大，后续在包体、性能、发热、下载和回滚上付出的工程成本越高。

如果两个模型在核心指标上接近，通常优先选择更小、更快、行为更稳定的模型。对移动端用户来说，稳定完成任务比偶尔输出惊艳结果更重要。

## 7.11 蒸馏数据集设计

蒸馏能否成功，很大程度取决于数据集，而不是训练脚本。移动端场景的数据集至少应该包含四类样本：

| 样本类型 | 作用 | 示例 |
| --- | --- | --- |
| 主路径样本 | 保证常见输入稳定 | 清晰语音命令、标准文档截图 |
| 边界样本 | 观察退化点 | 噪声、模糊、截断、多语言混合 |
| 反例样本 | 训练拒答和低置信度 | 与任务无关的问题、危险操作 |
| 回归样本 | 防止新版模型破坏旧能力 | 线上问题样本、投诉样本 |

教师模型输出不能直接等同于真值。建议使用“教师模型生成、规则过滤、人工抽样确认、线上问题回流”的流程。对于高风险任务，人工校验比例要更高；对于低风险任务，可以先用规则和小规模人工复核控制成本。

蒸馏数据还要保留来源和版本。否则当模型线上退化时，团队很难回答：问题来自教师模型、样本分布、标注规则，还是训练配置。模型能力越小，越依赖数据边界清晰。

## 7.12 能力裁剪文档模板

每个准备进入 App 的端侧模型，都建议有一页能力裁剪文档。它可以很短，但必须明确：

```text
功能名称：
目标用户任务：
支持输入：
不支持输入：
输出结构：
置信度字段：
失败处理：
高风险动作：
端侧预算：
云端兜底：
灰度指标：
回滚方式：
```

这份文档的读者不是算法团队内部，而是产品、客户端、服务端、测试和运营。它让所有人知道模型能做什么、不能做什么、什么时候应该让用户确认。

例如“本地会议纪要摘要”可以写成：支持 20 分钟以内的普通话转写文本；不支持多人说话人分离、不支持法律合同结论、不支持医疗建议；输出包括 `title`、`bullets`、`action_items`；如果文本过长则分段摘要；如果置信度低则提示用户检查原文。

这种文档能减少需求膨胀。没有裁剪文档的模型，往往会在迭代中被不断追加能力，最后变成一个既慢又不稳定的端侧通用入口。

## 7.13 量化回归样本

量化前后的评测要有固定回归集。回归集不要只覆盖“正常输入”，还要专门放入容易受精度影响的样本：

- 很短文本和很长文本。
- 低对比度图片。
- 含噪声音频。
- 相似类别之间的样本。
- 含数字、金额、日期和人名的文本。
- 多语言混合输入。
- 用户真实投诉样本。

评测结果建议拆成三层：

| 层级 | 判断方式 |
| --- | --- |
| 指标层 | 准确率、召回率、延迟、内存是否达标 |
| 样本层 | 关键样本是否出现不可接受退化 |
| 产品层 | 退化是否会触发误操作、误删、误导或投诉 |

如果量化后平均指标只下降很少，但关键样本出现高风险错误，仍然不能上线。移动端模型常常连接真实用户数据和真实操作，平均分不能替代产品风险判断。

## 7.14 小模型组合的接口规范

多个小模型组合时，最重要的是接口稳定。每个模型都应该用结构化输入输出连接，而不是让调用方解析自然语言。

建议每个能力节点都定义：

| 字段 | 说明 |
| --- | --- |
| `task_id` | 当前请求唯一标识，便于串联日志 |
| `model_id` | 模型标识 |
| `model_version` | 模型版本 |
| `input_schema` | 输入结构版本 |
| `output_schema` | 输出结构版本 |
| `confidence` | 置信度或可信区间 |
| `fallback_reason` | 拒绝、降级或转云原因 |

当 OCR、分类、检索和摘要组合时，任何一个节点失败都不应让整条链路不可解释。客户端至少要知道失败发生在哪个节点、是否可以重试、是否需要用户确认、是否可以切到云端。

接口规范还有一个好处：模型升级时可以独立验证。只要新模型保持同样 schema，调用方就能通过灰度比较新旧版本；如果 schema 变化，则必须进入兼容性评审。

## 7.15 任务风险分级

模型能力裁剪还要结合任务风险。相同模型输出，在不同业务动作中风险不同。例如识别“保存”失败，通常只是体验问题；识别“删除”失败，可能造成数据损失。

可以把端侧 AI 任务分为四级：

| 风险等级 | 示例 | 策略 |
| --- | --- | --- |
| 低 | 标签建议、输入联想、图片质量提示 | 可自动展示，允许用户忽略 |
| 中 | 分类、摘要、搜索排序 | 展示证据，允许编辑和反馈 |
| 高 | 删除、发送、上传、权限修改 | 必须用户确认，不直接执行 |
| 禁止自动化 | 支付、法律承诺、医疗判断、企业敏感外发 | 拒绝或进入人工/合规流程 |

风险分级应写进能力裁剪文档。模型输出不是最终动作，业务风险才决定是否自动执行。移动端工程师在接入模型时，要把模型置信度、任务风险和用户确认结合起来，而不是只看模型分数。

## 7.16 小模型上线门槛

小模型进入 App 前，应有清晰上线门槛。门槛可以分成效果、性能、体验和治理四类：

| 类型 | 门槛示例 |
| --- | --- |
| 效果 | 主路径样本达到目标指标，关键反例不误触发 |
| 性能 | 目标低端设备 P95 延迟低于预算 |
| 体验 | 低置信度有候选、拒答或手动入口 |
| 治理 | 模型版本、schema、日志字段和回滚策略明确 |

上线门槛应尽量可执行。例如“效果不错”不是门槛，“删除类命令在反例集中 0 次自动执行”才是门槛。对于高风险任务，可以允许召回率稍低，但不能允许危险误触发。

小模型如果暂时达不到门槛，不一定要放弃。可以缩小任务边界、提高确认比例、只对部分设备开放、或先作为后台建议而不是自动动作。能力裁剪不是一次性动作，而是模型和产品共同收缩到可发布边界的过程。

## 7.17 模型能力退化处理

模型更新后可能退化。退化不一定表现为崩溃，更多时候是低置信度变多、误报增加、拒答变多或用户接受率下降。团队需要提前定义退化处理：

1. 比较新旧模型在固定回归集上的结果。
2. 灰度期间按模型版本观察用户接受率和低置信度比例。
3. 发现关键任务退化时，立即停止扩大灰度。
4. 高风险误触发时，远程关闭自动执行路径。
5. 回滚后把退化样本加入回归集。

退化处理要和模型版本系统打通。如果客户端只知道“当前模型”，不知道具体版本，就无法判断问题来自哪个模型包。每次推理日志至少要包含模型 id、模型版本、schema 版本、任务类型和结果状态。

## 本章小结

端侧 AI 的核心是用足够小的模型解决足够明确的问题。模型越小，越需要清晰任务边界、结构化输出、置信度策略和严格评测。能力裁剪不是降低产品价值，而是把模型能力放到能发布、能解释、能回滚的工程边界内。
