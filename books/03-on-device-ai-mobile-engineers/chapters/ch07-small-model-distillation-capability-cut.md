# 第 7 章 小模型、蒸馏与能力裁剪

## 本章导读

端侧模型通常需要更小、更快、更省电。移动设备没有无限内存，也不能长时间高负载运行。一个云端模型可以追求通用能力，一个端侧模型更应该追求“在明确任务上足够好”。这就需要能力裁剪、量化、蒸馏、任务专用模型和严格评测共同工作。

本章讨论的重点不是训练算法细节，而是移动端工程师如何定义模型边界。很多端侧 AI 项目失败，并不是模型不够先进，而是任务定义过大：既想聊天、又想总结、又想搜索、又想识图，还要求离线、低延迟、低功耗。端侧模型设计的第一原则是收缩问题。

## 学习目标

- 区分通用大模型和任务专用小模型。
- 理解蒸馏、量化和裁剪的工程意义。
- 能为移动端场景制定模型能力边界。
- 能用评测数据判断模型是否足够进入 App。

## 7.1 小模型不是低配大模型

小模型不应该被理解为“大模型的缩水版”。端侧小模型更像一个任务专家：只解决清晰边界内的问题，并把不能解决的问题交给传统逻辑、用户确认或云端模型。

例如：

- 输入法联想不需要完整对话能力。
- 离线语音命令不需要开放式问答。
- 截图敏感区域检测不需要长文本生成。
- 本地笔记分类不需要最新互联网知识。
- 相册模糊检测不需要多轮推理。

如果任务可以被拆成分类、提取、排序、过滤、召回、短摘要或意图识别，小模型往往比通用大模型更适合端侧部署。

## 7.2 能力裁剪方法

能力裁剪是把模糊需求变成可执行边界。可以按下面步骤进行：

1. 写出用户任务，而不是模型能力。
2. 列出必须支持的输入类型。
3. 列出明确不支持的输入类型。
4. 定义输出格式。
5. 定义置信度和拒答策略。
6. 定义失败时的降级路径。

以“本地语音命令”为例：

| 项目 | 裁剪结果 |
| --- | --- |
| 支持 | 新建笔记、开始录音、暂停、继续、保存、搜索 |
| 不支持 | 任意聊天、复杂设置修改、跨 App 操作 |
| 输出 | command、slot、confidence |
| 拒绝 | 置信度低于阈值时要求用户重说 |
| 降级 | 展示文字按钮或搜索入口 |

裁剪之后，模型目标会清楚很多：它不是“理解所有语音”，而是“稳定识别少量命令”。

## 7.3 蒸馏的工程意义

蒸馏可以理解为让小模型学习大模型或强模型的行为。对移动端团队来说，蒸馏的价值通常在于：

- 用更小模型完成固定任务。
- 保持端侧低延迟。
- 降低模型包体和内存。
- 减少云端调用成本。
- 在隐私场景中减少上传。

但蒸馏不是自动成功。工程上要关注：

- 教师模型输出是否可靠。
- 训练样本是否覆盖真实移动端输入。
- 小模型是否只学到表面模式。
- 边界样本是否明显退化。
- 蒸馏后是否仍能解释错误。

如果教师模型在某类任务上本身不稳定，蒸馏只会把不稳定复制到端侧。移动端团队需要准备真实样本和人工校验样本，而不是只用合成数据。

## 7.4 量化与精度取舍

量化通过降低数值精度来减少模型大小和计算成本。移动端常见收益是包体下降、内存下降、速度提升和能耗降低。但量化可能改变模型行为，尤其是在边界输入上。

量化前后必须评测：

| 指标 | 关注点 |
| --- | --- |
| 模型大小 | 是否满足包体或下载预算 |
| 首次加载 | 是否减少冷启动等待 |
| 单次延迟 | 是否达到交互要求 |
| 峰值内存 | 是否降低 OOM 风险 |
| 准确率 | 主指标是否下降 |
| 边界样本 | 弱网、噪声、多语言、低质量图片是否退化 |

不要只看平均准确率。端侧产品中，最容易引发用户不满的是边界场景：噪声环境下误触发语音命令、证件照 OCR 错误、删除建议误判、敏感内容漏检。

## 7.5 任务专用模型组合

端侧 AI 不一定只有一个模型。很多产品更适合使用多个小模型组合：

```text
输入 -> 轻量分类 -> 专用模型 -> 规则校验 -> 业务动作
```

例如截图理解可以由多个能力组成：

- OCR 模型提取文字。
- 敏感区域检测模型遮挡隐私。
- 页面分类模型判断截图类型。
- 云端模型在用户确认后做复杂解释。

这种组合比一个端侧通用大模型更容易控制。每个模型的输入、输出和评测指标都更清楚，也更容易回滚。

## 7.6 置信度和拒答

端侧模型必须有“不知道”的能力。很多移动端任务连接真实操作，如果模型低置信度仍然输出确定结果，就会造成误导。

建议设计三段式策略：

| 置信度 | 动作 |
| --- | --- |
| 高 | 自动展示建议或执行低风险动作 |
| 中 | 展示候选，让用户确认 |
| 低 | 拒绝执行，提示用户换一种输入 |

例如语音识别“删除这条记录”属于高风险动作，即使置信度高，也应确认。置信度不是唯一决策依据，还要结合风险等级。

## 7.7 模型选择评审表

模型进入移动端工程前，建议填写：

| 项目 | 问题 |
| --- | --- |
| 任务边界 | 模型具体解决什么问题？不解决什么问题？ |
| 输入范围 | 支持哪些语言、长度、图片尺寸、音频场景？ |
| 输出格式 | 是否结构化？是否可校验？ |
| 性能预算 | 加载、延迟、内存、功耗预算是多少？ |
| 评测集 | 是否包含真实设备和真实用户样本？ |
| 降级 | 模型失败时 App 如何继续可用？ |
| 更新 | 模型如何灰度、回滚和兼容旧数据？ |
| 风险 | 是否连接删除、支付、权限、发布等高风险动作？ |

这张表的目的不是增加流程负担，而是避免“模型已经有了，先接进去看看”的随意集成。

## 7.8 案例：本地敏感信息检测

假设 App 需要在截图上传前检测手机号、邮箱、地址和头像。团队可以选择端侧小模型加规则：

1. OCR 提取文字。
2. 规则识别手机号、邮箱和证件号。
3. 小模型检测模糊地址和人名。
4. 视觉模型检测头像或人脸区域。
5. 合并结果并遮挡。
6. 用户确认后再上传。

这里不需要一个通用多模态大模型在端侧完整解释截图。能力裁剪后，任务变成“上传前隐私防护”，评测也更清楚：漏检率、误检率、遮挡位置、处理延迟和用户取消率。

## 7.9 小结前检查

端侧小模型上线前可以问：

- 是否存在更简单的规则或传统算法？
- 是否明确了不支持场景？
- 是否有拒答和用户确认？
- 是否有量化前后评测？
- 是否覆盖低端设备和边界样本？
- 是否能独立更新或回滚？
- 是否会让用户误以为能力更强？

这些问题能帮助团队保持克制。端侧 AI 的价值来自稳定解决具体问题，而不是展示模型存在感。

## 本章小结

端侧 AI 的核心是用足够小的模型解决足够明确的问题。模型越小，越需要清晰任务边界、结构化输出、置信度策略和严格评测。能力裁剪不是降低产品价值，而是把模型能力放到能发布、能解释、能回滚的工程边界内。
