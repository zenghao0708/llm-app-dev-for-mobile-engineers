# 第 11 章 端侧 AI 体验设计

## 本章导读

端侧 AI 的用户体验不只是响应快。用户需要知道能力边界、等待状态、失败原因和隐私承诺。端侧能力通常被包装成“更快、更隐私、离线可用”，但这些价值只有在交互中被清楚呈现，用户才会真正感受到。

AI 体验设计的核心是处理不确定性。模型可能置信度低，设备可能不支持，模型可能正在下载，权限可能被拒绝，离线时能力可能缩水。好的体验不是隐藏这些不确定性，而是把它们转化成可理解、可控制、可恢复的状态。

## 学习目标

- 设计可解释的 AI 交互。
- 处理等待、取消、失败和降级。
- 避免让用户误解模型能力。
- 能为端侧 AI 功能设计状态、反馈和控制权。

## 11.1 明确入口

端侧 AI 不应该在用户完全无感的情况下执行高影响动作。对于低风险建议，例如输入联想、图片质量提醒，可以自动出现；对于涉及上传、删除、修改、分享和权限的动作，应有明确入口和确认。

入口设计要回答：

- 这个 AI 功能做什么？
- 使用哪些数据？
- 是否在本地完成？
- 失败后会怎样？
- 用户如何关闭？

不要把 AI 能力藏在模糊按钮里，例如只写“智能处理”。更好的文案是“本地识别文字”“离线语音命令”“整理相似照片”“生成本地摘要”。

## 11.2 等待与取消

端侧 AI 仍然会等待：模型加载、图片处理、OCR、索引构建、下载模型、生成摘要都需要时间。等待状态要做到：

- 告诉用户正在做什么。
- 告诉用户是否在本地处理。
- 允许取消。
- 取消后释放资源。
- 长任务支持后台或稍后继续。

例如相册索引不应显示一个不可取消的全屏 loading。更好的体验是“正在设备空闲时整理照片，可随时暂停”，并显示已处理数量。

对于短任务，可以使用轻量状态；对于长任务，要显示进度、可取消和失败恢复。

## 11.3 能力边界

用户容易高估 AI 能力。端侧 AI 更需要明确边界，因为它通常比云端大模型能力更窄。

可以用三种方式表达边界：

| 方式 | 示例 |
| --- | --- |
| 功能命名 | “离线语音命令”而不是“语音助手” |
| 空状态文案 | “可识别新建、保存、搜索等常用命令” |
| 失败反馈 | “暂不支持这个命令，可以试试搜索或打开设置” |

边界不是削弱产品，而是减少误解。用户知道一个功能擅长什么，反而更容易形成稳定预期。

## 11.4 置信度与确认

AI 输出不应只有“成功/失败”。很多结果有置信度。体验上可以分层：

| 置信度 | 体验 |
| --- | --- |
| 高 | 直接展示建议或执行低风险动作 |
| 中 | 展示候选项，让用户选择 |
| 低 | 不执行，提示用户补充或重试 |

高风险动作即使高置信度也要确认。例如删除照片、发送消息、上传截图、修改权限设置、清空索引，都应该保留人工确认。

确认界面要展示关键证据。例如截图上传前展示脱敏预览；本地检索回答前展示引用片段；相册整理删除前展示候选图片。

## 11.5 离线与降级体验

端侧 AI 的价值之一是离线可用，但离线不一定代表完整能力。体验上应区分：

- 完全本地可用。
- 本地可用但能力较弱。
- 需要下载模型后可用。
- 需要网络增强。
- 当前设备不支持。

建议用明确状态表达：

```text
当前为离线模式：可以使用本地搜索和常用语音命令，复杂总结将在联网后继续。
```

不要让用户在离线时点击后才发现“失败”。如果功能需要网络增强，应在入口或结果阶段说明。

## 11.6 隐私反馈

端侧 AI 的隐私优势要在体验中呈现。用户不一定理解 runtime、embedding、索引，但能理解：

- 数据是否离开设备。
- 哪些内容会被上传。
- 是否可删除。
- 是否可关闭。

可以在关键节点显示简洁提示：

- “本次识别在设备上完成。”
- “将上传脱敏后的文字片段用于生成解释。”
- “本地索引只保存在此设备，可随时清空。”

隐私提示不要堆成长文，而要放在用户做决定的位置。

## 11.7 错误提示

端侧 AI 错误提示要可操作。常见错误包括：

| 错误 | 好的提示 |
| --- | --- |
| 模型未下载 | “需要下载本地识别组件，约 25 MB” |
| 权限拒绝 | “需要麦克风权限才能使用离线语音命令” |
| 设备不支持 | “当前设备暂不支持本地识别，可使用云端模式” |
| 内存不足 | “图片过大，已为你降低清晰度后重试” |
| 低置信度 | “没有识别到明确命令，可以重说或手动选择” |

避免使用“AI 处理失败”这种无行动价值的提示。

## 11.8 体验案例：本地会议摘要

一个本地会议摘要功能可以设计为：

1. 录音前说明“音频默认在设备上处理”。
2. 录音中显示状态和暂停按钮。
3. 处理摘要时显示“正在本地生成摘要，可取消”。
4. 长音频提示“可能耗时较长，建议连接电源”。
5. 如果本地处理失败，提供“稍后重试”或“用户确认后云端增强”。
6. 摘要结果展示引用时间点。
7. 用户可以删除录音、摘要和索引。

这个体验把隐私、等待、降级和删除都放进主流程，而不是隐藏在设置里。

## 11.9 体验检查清单

上线前建议检查：

| 项目 | 问题 |
| --- | --- |
| 入口 | 用户是否知道 AI 功能会做什么？ |
| 状态 | 等待、下载、处理、取消是否清楚？ |
| 控制 | 用户是否能取消、关闭、删除？ |
| 边界 | 是否说明不支持场景？ |
| 置信度 | 中低置信度是否有候选或拒答？ |
| 隐私 | 是否说明本地处理和上云边界？ |
| 降级 | 离线、失败、设备不支持时是否可用？ |
| 证据 | 结果是否有引用、来源或预览？ |

## 本章小结

AI 体验设计要把不确定性显式化。端侧能力再快，也需要清楚的入口、反馈、取消、边界、隐私说明和降级路径。用户真正需要的不是“模型很强”，而是知道发生了什么、数据去了哪里、结果能否信任、自己如何控制。
