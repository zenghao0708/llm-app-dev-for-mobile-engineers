# 第 11 章 端侧 AI 体验设计

## 本章导读

端侧 AI 的用户体验不只是响应快。用户需要知道能力边界、等待状态、失败原因和隐私承诺。端侧能力通常被包装成“更快、更隐私、离线可用”，但这些价值只有在交互中被清楚呈现，用户才会真正感受到。

AI 体验设计的核心是处理不确定性。模型可能置信度低，设备可能不支持，模型可能正在下载，权限可能被拒绝，离线时能力可能缩水。好的体验不是隐藏这些不确定性，而是把它们转化成可理解、可控制、可恢复的状态。

## 学习目标

- 设计可解释的 AI 交互。
- 处理等待、取消、失败和降级。
- 避免让用户误解模型能力。
- 能为端侧 AI 功能设计状态、反馈和控制权。

## 11.1 明确入口

端侧 AI 不应该在用户完全无感的情况下执行高影响动作。对于低风险建议，例如输入联想、图片质量提醒，可以自动出现；对于涉及上传、删除、修改、分享和权限的动作，应有明确入口和确认。

入口设计要回答：

- 这个 AI 功能做什么？
- 使用哪些数据？
- 是否在本地完成？
- 失败后会怎样？
- 用户如何关闭？

不要把 AI 能力藏在模糊按钮里，例如只写“智能处理”。更好的文案是“本地识别文字”“离线语音命令”“整理相似照片”“生成本地摘要”。

## 11.2 等待与取消

端侧 AI 仍然会等待：模型加载、图片处理、OCR、索引构建、下载模型、生成摘要都需要时间。等待状态要做到：

- 告诉用户正在做什么。
- 告诉用户是否在本地处理。
- 允许取消。
- 取消后释放资源。
- 长任务支持后台或稍后继续。

例如相册索引不应显示一个不可取消的全屏 loading。更好的体验是“正在设备空闲时整理照片，可随时暂停”，并显示已处理数量。

对于短任务，可以使用轻量状态；对于长任务，要显示进度、可取消和失败恢复。

## 11.3 能力边界

用户容易高估 AI 能力。端侧 AI 更需要明确边界，因为它通常比云端大模型能力更窄。

可以用三种方式表达边界：

| 方式 | 示例 |
| --- | --- |
| 功能命名 | “离线语音命令”而不是“语音助手” |
| 空状态文案 | “可识别新建、保存、搜索等常用命令” |
| 失败反馈 | “暂不支持这个命令，可以试试搜索或打开设置” |

边界不是削弱产品，而是减少误解。用户知道一个功能擅长什么，反而更容易形成稳定预期。

## 11.4 置信度与确认

AI 输出不应只有“成功/失败”。很多结果有置信度。体验上可以分层：

| 置信度 | 体验 |
| --- | --- |
| 高 | 直接展示建议或执行低风险动作 |
| 中 | 展示候选项，让用户选择 |
| 低 | 不执行，提示用户补充或重试 |

高风险动作即使高置信度也要确认。例如删除照片、发送消息、上传截图、修改权限设置、清空索引，都应该保留人工确认。

确认界面要展示关键证据。例如截图上传前展示脱敏预览；本地检索回答前展示引用片段；相册整理删除前展示候选图片。

## 11.5 离线与降级体验

端侧 AI 的价值之一是离线可用，但离线不一定代表完整能力。体验上应区分：

- 完全本地可用。
- 本地可用但能力较弱。
- 需要下载模型后可用。
- 需要网络增强。
- 当前设备不支持。

建议用明确状态表达：

```text
当前为离线模式：可以使用本地搜索和常用语音命令，复杂总结将在联网后继续。
```

不要让用户在离线时点击后才发现“失败”。如果功能需要网络增强，应在入口或结果阶段说明。

## 11.6 隐私反馈

端侧 AI 的隐私优势要在体验中呈现。用户不一定理解 runtime、embedding、索引，但能理解：

- 数据是否离开设备。
- 哪些内容会被上传。
- 是否可删除。
- 是否可关闭。

可以在关键节点显示简洁提示：

- “本次识别在设备上完成。”
- “将上传脱敏后的文字片段用于生成解释。”
- “本地索引只保存在此设备，可随时清空。”

隐私提示不要堆成长文，而要放在用户做决定的位置。

## 11.7 错误提示

端侧 AI 错误提示要可操作。常见错误包括：

| 错误 | 好的提示 |
| --- | --- |
| 模型未下载 | “需要下载本地识别组件，约 25 MB” |
| 权限拒绝 | “需要麦克风权限才能使用离线语音命令” |
| 设备不支持 | “当前设备暂不支持本地识别，可使用云端模式” |
| 内存不足 | “图片过大，已为你降低清晰度后重试” |
| 低置信度 | “没有识别到明确命令，可以重说或手动选择” |

避免使用“AI 处理失败”这种无行动价值的提示。

## 11.8 体验案例：本地会议摘要

一个本地会议摘要功能可以设计为：

1. 录音前说明“音频默认在设备上处理”。
2. 录音中显示状态和暂停按钮。
3. 处理摘要时显示“正在本地生成摘要，可取消”。
4. 长音频提示“可能耗时较长，建议连接电源”。
5. 如果本地处理失败，提供“稍后重试”或“用户确认后云端增强”。
6. 摘要结果展示引用时间点。
7. 用户可以删除录音、摘要和索引。

这个体验把隐私、等待、降级和删除都放进主流程，而不是隐藏在设置里。

## 11.9 体验检查清单

上线前建议检查：

| 项目 | 问题 |
| --- | --- |
| 入口 | 用户是否知道 AI 功能会做什么？ |
| 状态 | 等待、下载、处理、取消是否清楚？ |
| 控制 | 用户是否能取消、关闭、删除？ |
| 边界 | 是否说明不支持场景？ |
| 置信度 | 中低置信度是否有候选或拒答？ |
| 隐私 | 是否说明本地处理和上云边界？ |
| 降级 | 离线、失败、设备不支持时是否可用？ |
| 证据 | 结果是否有引用、来源或预览？ |

## 11.10 AI 功能状态机

端侧 AI 功能经常同时涉及权限、模型下载、模型加载、推理、取消和降级。只靠几个布尔值很容易出现 UI 状态混乱。建议把用户可见状态整理成状态机：

```text
unavailable
  -> permission_required
  -> model_required
  -> downloading
  -> ready
  -> processing
  -> result
  -> failed
  -> degraded
```

每个状态都要定义：

| 状态 | 用户看到什么 | 用户能做什么 |
| --- | --- | --- |
| `permission_required` | 需要授权的原因 | 去授权、取消 |
| `model_required` | 需要下载本地组件 | 下载、稍后 |
| `downloading` | 下载大小和进度 | 暂停、取消 |
| `ready` | 功能可用 | 开始任务 |
| `processing` | 正在处理和可取消 | 取消、后台继续 |
| `result` | 结果和证据 | 接受、编辑、删除 |
| `failed` | 失败原因 | 重试、降级 |
| `degraded` | 当前能力缩水 | 使用可用能力 |

状态机的价值是让设计、客户端和测试对齐。测试用例可以按状态覆盖，而不是只测“正常成功”。用户体验中的很多问题都来自状态漏处理：下载中退出页面、权限撤销后仍显示入口、模型加载失败后按钮不可恢复。

## 11.11 结果可解释与可编辑

AI 输出不能像系统通知一样不可质疑。端侧 AI 结果更应该提供证据和编辑入口：

- 本地检索回答展示引用片段。
- 图片识别展示识别区域或缩略图。
- 会议摘要展示对应时间点。
- 相册清理展示候选图片对比。
- 隐私遮挡展示遮挡前后的预览。

可解释不等于展示模型内部细节，而是让用户知道结果从哪里来、为什么可信、哪里可以修改。对于摘要、分类、标签和建议，用户应能编辑、删除或反馈错误。

结果可编辑还有工程价值。用户修改结果可以成为本地偏好或反馈样本，但必须经过授权和脱敏设计。不要把用户编辑过的隐私内容直接上传为训练数据。

## 11.12 下载模型的体验

模型下载是端侧 AI 独有的体验环节。它会影响首次使用、弱网、存储和用户信任。一个好的下载体验应该包含：

| 信息 | 示例 |
| --- | --- |
| 下载原因 | “需要下载本地 OCR 组件才能离线识别文字” |
| 大小 | “约 25 MB” |
| 网络建议 | “建议在 Wi-Fi 下下载” |
| 存储提示 | “设备空间不足，无法完成下载” |
| 隐私说明 | “识别将在设备上完成” |
| 退出路径 | “稍后下载，继续拍照保存” |

不要在用户点击功能后悄悄下载大模型，也不要把下载失败伪装成普通功能失败。用户需要知道等待的原因，并能选择是否继续。

如果模型是按需下载，入口要支持未安装状态；如果模型正在灰度，用户可能看到“当前设备暂不支持”。这些状态都应该有明确文案，而不是让按钮消失。

## 11.13 低端设备体验

端侧 AI 的体验不能只在旗舰机上成立。低端设备可能出现加载慢、内存不足、发热、模型不可用等问题。产品上可以设计分层体验：

| 设备能力 | 体验策略 |
| --- | --- |
| 高性能设备 | 默认启用本地模型和后台索引 |
| 主流设备 | 启用核心本地能力，限制长任务 |
| 低内存设备 | 默认关闭重模型，提供云端模式 |
| 不支持设备 | 隐藏入口或展示替代能力 |

分层体验要透明。用户不需要知道芯片细节，但需要知道为什么某个功能不可用，以及是否有替代方案。比如“当前设备暂不支持本地识别，可在联网时使用云端识别”比“功能不可用”更好。

测试时要专门走低端设备路径。很多 AI 功能在高端设备上表现流畅，但低端设备会频繁触发降级。如果降级体验没有设计，用户看到的就是失败。

## 11.14 用户反馈闭环

端侧 AI 需要用户反馈，但反馈入口要克制。常见反馈包括：

- 结果正确或错误。
- 隐私遮挡是否准确。
- 摘要是否有遗漏。
- 检索引用是否相关。
- 语音命令是否识别错误。

反馈设计要遵守两个原则：第一，用户知道反馈会发送什么；第二，默认不上传原始敏感内容。可以让用户选择是否附带样本，或者只上传脱敏后的错误类型和模型版本。

反馈结果要回到工程流程：错误样本进入回归集，体验问题进入文案和状态机调整，性能问题进入设备矩阵。否则反馈只是收集情绪，不能提升产品质量。

## 11.15 多模态交互的一致性

端侧 AI 可能同时处理文字、语音、图片和文件。不同入口要保持一致的控制权：

| 输入方式 | 需要一致的体验 |
| --- | --- |
| 文字 | 可编辑、可清空、可取消 |
| 语音 | 可暂停、可重录、可删除音频 |
| 图片 | 可预览、可裁剪、可删除临时图 |
| 文件 | 可确认范围、可取消解析 |

如果文字输入可以撤销，语音输入也应能删除；如果图片上传前有脱敏预览，文件解析前也应说明会读取哪些内容。多模态体验的一致性比单个入口的炫酷更重要。

## 本章小结

AI 体验设计要把不确定性显式化。端侧能力再快，也需要清楚的入口、反馈、取消、边界、隐私说明和降级路径。用户真正需要的不是“模型很强”，而是知道发生了什么、数据去了哪里、结果能否信任、自己如何控制。
