# 第 10 章 隐私、安全与本地数据治理

## 本章导读

端侧 AI 常以隐私为卖点，但本地处理不自动等于安全。模型、缓存、索引、日志、用户输入、调试文件和同步链路都需要治理。只要数据被收集、存储、索引、展示、上传或写入日志，就存在边界和责任。

移动端工程师要建立一个清醒认识：端侧处理只是隐私设计的起点，不是终点。真正能保护用户的是数据最小化、明确授权、加密存储、可删除、可解释、可审计和可回滚的工程体系。

## 学习目标

- 理解本地数据生命周期。
- 设计本地索引、缓存和模型文件的安全策略。
- 避免日志、调试和同步链路泄露隐私。
- 能为端侧 AI 功能建立数据治理清单。

## 10.1 本地不等于安全

端侧 AI 的隐私优势主要来自“原始数据不必上传”。但如果本地处理结果被写入日志、同步到云端、进入崩溃上报、被其他 App 读取或无法删除，隐私风险仍然存在。

常见误区包括：

- 认为只要不上传原图，就可以任意保存 OCR 结果。
- 认为本地 embedding 不含原文，就一定不敏感。
- 认为缓存只是临时文件，不需要清理。
- 认为调试日志不会进入线上。
- 认为用户删除原文后，索引可以保留。

这些误区会让“端侧隐私”变成口号。工程上必须把本地数据当成真实敏感资产管理。

## 10.2 数据生命周期

本地数据要回答：

- 从哪里来？
- 保存在哪里？
- 保存多久？
- 如何删除？
- 是否加密？
- 是否同步？
- 是否进入日志或崩溃上报？

建议为每类数据建立生命周期表：

| 数据 | 来源 | 存储 | 保留 | 删除 | 是否上云 |
| --- | --- | --- | --- | --- | --- |
| 原始音频 | 麦克风 | 默认不保存 | 任务结束即释放 | 自动释放 | 默认不上云 |
| OCR 文本 | 截图/文档 | 本地加密缓存 | 用户可配置 | 随原图删除 | 用户确认后 |
| embedding | 本地笔记 | 本地索引 | 功能开启期间 | 原文删除后删除 | 不直接上传 |
| 模型日志 | runtime | 聚合指标 | 短期 | 自动清理 | 只上传统计 |

没有生命周期表，团队很难在需求迭代中保持一致。

## 10.3 数据最小化

数据最小化是端侧 AI 的第一原则。能不收集就不收集，能不保存就不保存，能保存派生低敏数据就不保存原始高敏数据。

例如截图解释功能可以：

1. 本地 OCR。
2. 本地检测敏感区域。
3. 只把脱敏后的文字片段用于云端解释。
4. 不保存原始截图。
5. 不把 OCR 全文写入日志。

这比“上传整张截图给云端模型”更符合端侧 AI 的价值。

## 10.4 加密与访问控制

本地索引、缓存和模型配置可能包含敏感信息。移动端应考虑：

- 使用系统提供的安全存储保存密钥。
- 对本地数据库或索引加密。
- 避免把敏感缓存放在可被备份或共享的位置。
- 区分用户登录态和游客态的数据。
- 用户退出登录时清理个人索引。
- 企业场景中支持远程擦除或策略禁用。

模型文件本身也要保护完整性。虽然模型不一定包含用户数据，但被篡改后可能输出错误结果或绕过安全策略。因此模型下载后要校验 checksum，必要时做签名验证。

## 10.5 日志和崩溃上报

端侧 AI 最容易泄露隐私的地方不是模型，而是日志。开发阶段为了排查问题，工程师可能记录输入文本、OCR 结果、模型输出、提示词、错误上下文和文件路径。如果这些日志进入线上，就会造成风险。

建议日志只记录：

- 任务类型。
- 输入大小区间。
- 模型版本。
- 运行耗时。
- 错误类型。
- 是否降级。

不要记录：

- 原始文本。
- 原始图片。
- 原始音频。
- 完整 OCR 内容。
- 用户文件路径。
- 联系人、地址、手机号、token。

如果确实需要问题样本，应通过用户授权的反馈通道单独收集，并做脱敏和过期清理。

## 10.6 同步与备份

本地索引是否参与系统备份或多设备同步，需要明确设计。默认把所有缓存交给系统备份，可能导致用户换设备后恢复出已经删除或不希望同步的数据。

建议区分：

| 数据 | 是否同步 |
| --- | --- |
| 用户原始内容 | 按产品原有策略 |
| 可重建索引 | 通常不需要同步 |
| 模型文件 | 可重新下载，不必备份 |
| 用户偏好 | 可同步，但要尊重关闭设置 |
| 处理日志 | 不同步 |

如果跨设备同步个性化索引，要处理删除事件、版本兼容和授权差异。不要让 A 设备已删除的数据在 B 设备继续被搜索到。

## 10.7 安全边界

端侧 AI 还要考虑攻击面：

- 恶意输入触发模型异常或性能耗尽。
- 篡改模型文件。
- 伪造模型配置。
- 通过日志读取敏感输出。
- 越权访问其他用户的本地索引。
- WebView 或插件读取本地缓存。

防护策略包括输入长度限制、文件完整性校验、权限隔离、最小日志、沙箱路径、加密存储和异常降级。

## 10.8 案例：本地截图助手的数据治理

一个截图助手可能处理用户选择的截图，并给出解释。治理方案可以是：

1. 用户主动选择截图，不自动读取相册。
2. 本地生成缩略图和 OCR。
3. 本地检测手机号、头像、地址、二维码。
4. 展示脱敏预览。
5. 用户确认后，上传脱敏文本或裁剪后的区域。
6. 任务结束后删除临时图像。
7. 只记录处理耗时和错误类型。
8. 用户可在设置中清空历史和索引。

这个流程清楚地区分了本地处理、用户确认和云端增强。它比一句“截图在本地处理，更安全”更可信。

## 10.9 数据治理清单

上线前建议检查：

| 项目 | 问题 |
| --- | --- |
| 授权 | 用户是否知道哪些数据会被处理？ |
| 最小化 | 是否避免保存原始高敏数据？ |
| 存储 | 本地索引和缓存是否加密？ |
| 日志 | 是否禁止原文、图片、音频进入日志？ |
| 删除 | 用户删除原文后索引是否同步删除？ |
| 同步 | 缓存和索引是否被系统备份或多设备同步？ |
| 上云 | 上传前是否脱敏并经过用户确认？ |
| 审计 | 是否能说明数据生命周期？ |

## 10.10 数据分类分级

端侧 AI 功能上线前，应先把涉及的数据分类。不同数据的敏感程度不同，治理策略也不同：

| 数据类型 | 敏感程度 | 治理策略 |
| --- | --- | --- |
| 原始图片、音频、文档 | 高 | 默认不上传，短期缓存，加密存储 |
| OCR 文本、转写文本 | 高 | 按原始内容治理，禁止进入日志 |
| embedding 向量 | 中到高 | 绑定原文生命周期，模型变更后重建 |
| 模型输出摘要 | 中 | 展示给用户确认，随原文删除 |
| 聚合性能指标 | 低 | 只记录区间和状态，不记录内容 |
| 模型文件 | 低到中 | 校验完整性，控制来源和版本 |

不要因为数据是“派生结果”就降低治理等级。OCR 文本、摘要、标签、向量索引都可能还原用户兴趣、身份、位置、工作内容和社交关系。移动端工程师要从用户能否被识别、能否被推断、能否被误用的角度判断敏感程度。

分类分级的结果要进入技术方案，而不是只留在隐私合规文档中。客户端存储路径、日志字段、崩溃上报、清理策略、同步策略都应该引用这张分类表。

## 10.11 权限撤销与数据删除

用户授予权限不代表永久授权。端侧 AI 功能必须处理权限撤销和数据删除：

- 用户关闭相册权限后，停止新增图片索引。
- 用户删除照片后，清理对应缩略图、embedding 和标签。
- 用户关闭麦克风权限后，语音命令入口进入不可用状态。
- 用户退出账号后，清理个人模型缓存和索引。
- 用户关闭 AI 功能后，删除可重建数据并停止后台任务。

删除不能只删 UI 上可见的记录。真正的删除要覆盖：

| 位置 | 需要处理的内容 |
| --- | --- |
| 数据库 | 原文、摘要、标签、状态记录 |
| 文件系统 | 临时文件、缩略图、模型中间产物 |
| 索引 | embedding、倒排索引、向量索引 |
| 缓存 | 最近结果、预处理结果、token cache |
| 队列 | 未完成任务、重试任务、后台任务 |

如果删除链路做不到同步完成，可以采用“标记删除 + 异步清理 + 完成确认”的方式，但用户层面要避免继续展示已删除内容。隐私体验中，删除的一致性比清理速度更重要。

## 10.12 企业与未成年人场景

某些移动端产品会进入企业、教育或未成年人场景，数据治理要求更高。端侧 AI 能降低上传风险，但不能替代组织策略。

企业场景常见要求包括：

- 禁止把企业文档上传到外部模型。
- 支持管理员关闭某些 AI 功能。
- 支持远程擦除本地索引。
- 支持审计模型版本和数据处理类型。
- 支持设备丢失后的数据保护。

教育和未成年人场景还要关注：

- 不诱导上传身份、位置、通讯录等敏感信息。
- 对输出建议进行风险限制。
- 避免长期保留学习行为画像。
- 为监护或组织策略预留关闭入口。

工程上可以把这些要求抽象成策略层：功能入口、模型下载、上云增强、日志采集、索引保留时间都由策略控制。这样同一个 App 可以在个人版、企业版和教育版中复用代码，但应用不同治理边界。

## 10.13 本地模型安全

端侧 AI 不只处理用户数据，也引入新的本地资产：模型文件、tokenizer、prompt 模板、规则库和安全配置。它们一旦被篡改，可能造成错误结果或绕过保护策略。

本地模型安全至少包括：

| 风险 | 防护 |
| --- | --- |
| 模型包被替换 | checksum、签名验证、HTTPS 下载 |
| 配置被篡改 | 配置签名、版本白名单 |
| 调试开关误开 | 发布包关闭调试入口 |
| prompt 模板泄露 | 不写入日志，不暴露内部策略 |
| 规则库过期 | 独立版本和灰度更新 |

移动端沙箱能提供基础隔离，但不能假设文件永远可信。模型加载前做完整性校验，发现异常时要删除本地文件、回退稳定版本，并上报不含敏感输入的错误类型。

## 10.14 隐私审计记录

当功能越来越多，团队需要能回答“某个端侧 AI 功能到底处理了什么数据”。建议为每个功能维护一份隐私审计记录：

```text
feature_id:
processed_data:
local_storage:
cloud_upload:
retention_period:
delete_trigger:
log_fields:
model_versions:
fallback_path:
owner:
last_review_date:
```

这份记录不是给用户阅读的说明文，而是给研发、测试、法务、运营和安全团队共同维护的工程事实。每次新增字段、改日志、改缓存、改云端兜底，都要同步更新。

审计记录还有助于事故排查。例如用户反馈删除后仍能搜索到旧内容，团队可以顺着 `delete_trigger` 和 `local_storage` 找到遗漏的索引或缓存，而不是在代码里盲查。

## 本章小结

端侧 AI 的隐私优势要靠工程实现证明。数据最小化、明确授权、加密存储、日志治理、删除一致性和同步边界是基础要求。移动端团队要把本地数据当成真实资产管理，而不是因为“没有上云”就放松治理。
