# 第 1 章 端侧大模型的机会与边界

## 本章导读

端侧大模型不是把云端模型缩小后放进手机。移动设备有严格的电量、内存、温度、网络、权限和发布约束。一个模型能在实验环境中运行，不代表能进入真实 App。对移动端工程师来说，真正的问题不是“模型能不能跑”，而是“模型是否值得在端上跑，运行失败时如何降级，用户数据如何保护，发版后如何观测和回滚”。

第一册讨论了大模型应用的基本链路，第二册讨论了 AI coding 如何改造工程过程。第三册把视角放回移动设备：当模型能力进入用户手里的手机、平板、车机、可穿戴设备，工程师要面对的约束会更具体，也更难绕过。

## 学习目标

- 理解端侧智能的收益和边界。
- 区分端侧推理、云端推理和端云协同。
- 知道哪些场景适合本地优先，哪些场景必须云端兜底。
- 建立端侧 AI 的产品、工程和治理视角。

## 1.1 为什么移动端需要端侧 AI

端侧 AI 的价值主要来自四点。

第一，低延迟。某些交互不能等待网络往返。例如输入法联想、拍照取景框中的实时提示、离线语音命令和本地分类，都要求用户感知上“马上有结果”。即使云端模型更强，延迟也可能破坏体验。

第二，隐私保护。很多数据天然属于用户设备：照片、截图、剪贴板、本地笔记、浏览历史、输入内容、位置轨迹。如果任务可以在端侧完成，就不必把原始数据上传到云端。这不是自动合规，但提供了更好的隐私设计起点。

第三，离线可用。移动端用户经常处于弱网、地铁、飞机、海外漫游、企业内网或系统省电模式。端侧能力可以在网络不可用时提供基础体验。

第四，成本控制。云端大模型调用有成本，也有并发和限流问题。把简单任务放到本地，可以减少云端请求，把预算留给真正需要大模型能力的复杂任务。

这些价值解释了为什么端侧 AI 重要，但它们不意味着所有能力都应该端侧化。

## 1.2 端侧 AI 的工程代价

端侧 AI 会引入新的成本：

- 模型包体变大，影响下载和安装。
- 模型加载增加冷启动或功能首次使用延迟。
- 推理占用 CPU/GPU/NPU，影响电量和温度。
- 内存峰值可能导致低端设备崩溃。
- 模型更新比云端慢，需要版本和灰度策略。
- 端侧结果更难统一观测，需要本地指标和上报设计。
- 平台差异明显，iOS、Android、不同芯片和系统版本表现可能不同。

因此，端侧 AI 的决策不能只看模型效果。一个本地模型准确率不错，但让页面首次打开增加 3 秒、包体增加 200 MB、低端设备频繁 OOM，这在产品上可能不可接受。

## 1.3 端侧、云端与端云协同

可以把移动端 AI 能力分为三种形态：

| 形态 | 说明 | 优势 | 风险 |
| --- | --- | --- | --- |
| 端侧推理 | 模型在设备上运行 | 低延迟、隐私、离线 | 受设备性能和包体限制 |
| 云端推理 | App 调用服务端模型 | 能力强、更新快、易观测 | 网络、成本、隐私边界 |
| 端云协同 | 端侧和云端分工 | 兼顾体验和能力 | 架构、评测和降级复杂 |

成熟移动端产品通常不会简单选择其中一种，而是按任务分层。例如本地小模型判断用户意图，如果是简单命令就在端上执行；如果需要复杂生成，再把经过脱敏和压缩的上下文交给云端。

## 1.4 场景判断表

| 场景 | 适合端侧 | 适合云端 | 判断理由 |
| --- | --- | --- | --- |
| 输入法联想、简单分类 | 是 | 可选 | 对延迟敏感，任务边界明确 |
| 离线语音命令 | 是 | 可选 | 网络不可用时仍需基础能力 |
| 本地图片预处理 | 是 | 可选 | 可减少上传数据和云端成本 |
| 大规模知识问答 | 通常不适合 | 是 | 需要大模型和最新知识 |
| 隐私敏感本地摘要 | 是 | 需谨慎 | 可先端侧处理，再决定是否上云 |
| 高质量长文本生成 | 视设备而定 | 是 | 端侧能力和速度可能不足 |
| 企业内部文档问答 | 端侧可做预处理 | 是 | 需要权限、检索和审计 |

判断时可以问四个问题：

1. 用户是否能接受网络等待？
2. 原始输入是否敏感？
3. 端侧模型的效果是否足够？
4. 失败时是否有清晰降级？

如果四个问题无法回答，不应急着落地端侧模型。

## 1.5 移动端案例：本地会议摘要

假设 App 希望提供会议录音摘要。这个任务看起来适合 AI，但端侧与云端取舍并不简单。

端侧方案的优势是隐私好、弱网可用、短录音响应快。风险是转写和摘要模型可能很大，长音频耗电明显，低端设备容易发热。

云端方案的优势是效果好、模型更新快、长音频处理能力强。风险是录音上传涉及隐私和合规，网络不好时体验不稳定。

端云协同方案可以这样设计：

- 端侧先做音频切分、降噪和敏感片段检测。
- 短音频或隐私模式使用本地摘要。
- 长音频在用户确认后上传云端处理。
- 云端返回结果后，本地保存摘要和索引。
- 网络失败时保留待处理队列。

这个例子说明，端侧 AI 不是单点技术，而是一组产品和工程决策。

## 1.6 端侧 AI 决策清单

在立项前，团队应填写：

| 问题 | 说明 |
| --- | --- |
| 用户价值 | 为什么必须在端上做？ |
| 模型能力 | 本地模型是否达到可用效果？ |
| 设备范围 | 支持哪些系统版本和硬件？ |
| 性能预算 | 延迟、内存、电量、包体上限是多少？ |
| 数据边界 | 哪些数据留在本地，哪些会上云？ |
| 降级策略 | 模型不可用时如何处理？ |
| 更新策略 | 模型如何发布、灰度和回滚？ |
| 观测指标 | 如何知道线上是否稳定？ |

如果这些问题没有答案，技术选型再漂亮也可能无法发布。

## 1.7 立项评审：从想法到可做

端侧 AI 功能进入研发前，建议先做一次轻量立项评审。评审不是为了阻止创新，而是让团队在投入模型、客户端、服务端、测试和设计资源前，先确认问题值得做。

评审可以分为五类问题。

第一，用户场景是否真实。端侧 AI 很容易被包装成“智能功能”，但如果用户没有明确痛点，模型能力会变成展示项。例如“本地总结所有内容”听起来强，但用户真正需要的可能只是“快速找到上周会议里的待办”。后者更适合从本地检索和引用做起。

第二，端侧价值是否明确。选择端侧应该有理由：低延迟、隐私、离线、成本或体验连续性。如果只是因为“现在端侧 AI 很热”，那不是工程理由。

第三，任务边界是否可测。端侧功能必须能评测。例如“理解用户意图”太宽；“识别 20 个离线语音命令，置信度低时拒绝执行”就可测。

第四，失败是否可接受。模型会失败，设备会失败，权限会失败，网络会失败。立项时要确认失败路径不会破坏核心体验。

第五，数据治理是否清楚。哪些数据留在本地，哪些数据会上云，哪些数据会进入日志，用户如何删除，这些问题不能等到上线前才讨论。

可以使用下面的立项表：

| 项目 | 问题 | 结论 |
| --- | --- | --- |
| 场景 | 用户在什么时刻需要这个能力？ | 待填 |
| 端侧理由 | 为什么不能只用云端？ | 待填 |
| 任务边界 | 输入、输出和拒答条件是什么？ | 待填 |
| 设备范围 | 支持哪些机型和系统版本？ | 待填 |
| 数据边界 | 原始数据是否上传？是否记录日志？ | 待填 |
| 验证方式 | 用什么样本和设备证明可用？ | 待填 |
| 降级 | 模型不可用时用户还能做什么？ | 待填 |

这张表填不出来时，不应急着写代码。

## 1.8 常见误判

端侧 AI 立项中常见五类误判。

第一，把“能跑”当成“能发布”。开发机上跑通模型，只证明技术链路存在，不证明它满足包体、延迟、内存、电量、权限和审核要求。

第二，把“本地处理”当成“隐私合规”。本地缓存、本地索引、日志、崩溃上报和备份都可能带来隐私风险。

第三，把“模型效果好”当成“产品体验好”。如果用户不知道 AI 正在做什么、等待不可取消、结果没有引用、错误提示不可操作，体验仍然失败。

第四，把“端云协同”写成“失败就上云”。端云协同需要用户确认、数据脱敏、任务队列、超时和回滚，不是简单兜底。

第五，把“低端设备关闭”当成“兼容方案”。有些高价值能力可以在低端设备上降级为规则、云端或手动入口，而不是直接消失。

这些误判在早期看不明显，到了发版前会变成跨团队问题。移动端工程师应在需求阶段就把它们拉到桌面上。

## 1.9 从小能力开始

端侧 AI 的务实路径通常是从小能力开始：

1. 一个低风险任务。
2. 一个明确模型。
3. 一个目标设备范围。
4. 一个可重复评测集。
5. 一个清晰降级路径。
6. 一个线上观测闭环。

例如先做“本地识别 10 个语音命令”，再扩展到“本地语音命令 + 云端复杂问答”；先做“本地笔记召回”，再扩展到“本地召回 + 云端总结”；先做“截图敏感区域遮挡”，再扩展到“截图解释助手”。

这不是保守，而是移动端工程的现实。小能力稳定后，团队才有信心扩大模型、数据源和产品入口。

## 本章小结

端侧大模型的关键不是“能不能跑”，而是“是否值得跑、怎样稳定跑、失败时如何降级”。端侧 AI 的价值来自低延迟、隐私、离线和成本控制，但代价是包体、性能、兼容、评测和治理复杂度。后续章节会从 runtime、端云架构、性能、隐私和产品化角度展开。
