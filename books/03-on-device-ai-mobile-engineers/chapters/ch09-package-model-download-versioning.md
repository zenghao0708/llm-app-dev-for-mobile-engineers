# 第 9 章 包体、模型下载与版本更新

## 本章导读

模型文件会改变移动端发版方式。它可能随 App 打包，也可能首次启动下载，还可能按灰度策略更新。对移动端团队来说，模型不是普通资源文件：它影响安装包大小、首次体验、存储空间、网络流量、版本兼容、灰度回滚和线上观测。

本章讨论模型分发和版本治理。一个端侧 AI 功能能否进入生产环境，很大程度取决于模型如何到达用户设备，以及出问题时能否安全回滚。

## 学习目标

- 理解模型分发方式。
- 设计版本、校验、回滚和灰度策略。
- 处理弱网、断点续传和存储空间不足。
- 能为移动端模型建立可发布、可观测、可回滚的分发链路。

## 9.1 模型分发方式

| 方式 | 优点 | 风险 |
| --- | --- | --- |
| 随包发布 | 简单稳定 | 包体变大，更新慢 |
| 首次下载 | 包体小 | 首次体验差，依赖网络 |
| 按需下载 | 精细控制 | 状态管理复杂 |
| 远程灰度 | 可回滚 | 需要平台和监控 |

随包发布适合小模型、核心功能和必须离线可用的能力。首次下载适合模型较大但功能重要的场景。按需下载适合低频功能或多模型组合。远程灰度适合模型迭代快、需要分批发布和回滚的场景。

实际项目中经常混合使用。例如基础敏感信息检测模型随包发布，视觉理解模型按需下载，个性化索引模型通过远程灰度更新。

## 9.2 包体预算

模型会直接增加安装包或 App 资源大小。包体变大可能影响：

- 下载转化率。
- 应用商店审核和分发。
- 用户存储空间。
- 企业内部分发。
- 低端设备安装体验。

团队应为模型设置包体预算：

| 类型 | 建议策略 |
| --- | --- |
| 核心小模型 | 可随包，但必须有大小上限 |
| 中等模型 | 评估是否按需下载 |
| 大模型 | 默认远程下载或云端兜底 |
| 多语言模型 | 按语言或地区拆分 |
| 实验模型 | 不随正式包全量发布 |

包体评审不能只看单个模型。多个模型、runtime、tokenizer、词表和资源文件加起来，才是用户真正要下载的大小。

## 9.3 模型包结构

建议模型包包含：

```text
model_package/
  manifest.json
  model.bin
  tokenizer.json
  config.json
  checksum.txt
  license.txt
```

`manifest.json` 至少记录：

```json
{
  "model_id": "local-intent-v1",
  "version": "1.2.0",
  "runtime": "example-runtime",
  "min_app_version": "5.8.0",
  "platform": "ios",
  "checksum": "sha256-placeholder",
  "size_bytes": 12345678,
  "created_at": "2026-06-28"
}
```

模型文件、配置、tokenizer 和版本信息必须绑定。不要只更新模型权重，却让旧 tokenizer 继续工作。很多线上问题来自资源版本不一致。

## 9.4 下载与校验

模型下载链路要处理：

- 弱网。
- 断点续传。
- 下载中断。
- 存储空间不足。
- 校验失败。
- 下载完成但加载失败。
- 用户清理缓存。
- App 升级后资源迁移。

下载完成后必须做完整性校验。校验失败不能继续加载。校验通过后，应采用原子替换：先下载到临时目录，校验成功后再切换到正式目录，避免半包状态。

状态机可以设计为：

```text
not_installed -> downloading -> verified -> ready
                       |            |
                    failed       load_failed
```

UI 不应只显示“下载失败”。要区分网络失败、空间不足、校验失败和版本不兼容，因为用户和工程师的处理方式不同。

## 9.5 版本兼容

模型版本要和 App 版本、runtime 版本、索引版本兼容。常见问题包括：

- 新模型需要新 runtime，但用户 App 版本过旧。
- embedding 模型更新后，旧索引不可复用。
- tokenizer 更新后，旧缓存失效。
- 云端下发了不支持当前系统的模型。
- 回滚模型后，本地数据格式不兼容。

因此，模型 manifest 中应包含 `min_app_version`、`runtime_version`、`schema_version` 和 `index_version`。App 在加载前先判断兼容性，不兼容则不下载或不启用。

## 9.6 灰度发布

模型灰度发布要比普通配置更谨慎。建议从小比例开始：

1. 内部设备。
2. 员工或测试用户。
3. 1% 真实用户。
4. 5%。
5. 20%。
6. 全量。

每个阶段观察：

- 下载成功率。
- 加载成功率。
- 推理成功率。
- 延迟和内存。
- 崩溃和 ANR。
- 用户取消率。
- 业务指标。

如果模型涉及隐私、安全或高风险动作，还要增加人工抽样评审。灰度不是只看崩溃率，也要看错误结果是否变多。

## 9.7 回滚策略

模型必须可回滚。回滚策略包括：

- 保留上一个稳定版本。
- 新版本加载失败时自动回退。
- 远程配置关闭新模型。
- 清理坏版本缓存。
- 兼容旧索引或触发重建。
- 回滚后停止继续下载坏版本。

回滚不能依赖发新版 App。模型既然可以远程更新，就必须可以远程禁用或回退。

## 9.8 存储清理

移动设备存储有限。模型、索引、缓存和临时文件都要有清理策略：

- 删除未使用模型。
- 清理下载临时文件。
- 限制最多保留版本数。
- 用户关闭功能时清理相关资源。
- 系统存储不足时释放可重建资源。

要区分“可重建”和“不可重建”数据。模型和索引通常可重建；用户原始数据不可随意删除。清理策略必须谨慎。

## 9.9 案例：按需下载 OCR 模型

假设文档扫描功能需要 OCR 模型，但不是所有用户都会使用。可以设计：

1. App 首次安装不下载 OCR 模型。
2. 用户进入文档扫描页时提示需要下载本地识别组件。
3. 下载前检查网络和存储空间。
4. 下载支持断点续传。
5. 下载后校验 checksum。
6. 校验成功后启用本地 OCR。
7. 如果下载失败，允许用户继续使用拍照保存，不启用 OCR。
8. 新版本 OCR 通过灰度更新。
9. 发现问题时远程回滚到上个版本。

这个流程比“进入页面自动下载模型”更复杂，但更符合移动端真实环境。

## 9.10 发布检查清单

模型发布前建议检查：

| 项目 | 问题 |
| --- | --- |
| 包体 | 模型、runtime、tokenizer 总大小是否可接受？ |
| 版本 | manifest 是否记录模型、runtime、App 兼容信息？ |
| 校验 | 是否有 checksum 和原子替换？ |
| 弱网 | 是否支持失败重试和断点续传？ |
| 空间 | 存储不足时是否有明确提示？ |
| 灰度 | 是否能按用户、设备、版本分批发布？ |
| 回滚 | 是否能远程禁用或恢复旧模型？ |
| 观测 | 是否能看到下载、加载、推理和失败指标？ |
| 清理 | 是否限制历史版本和临时文件？ |

## 9.11 模型包验收标准

模型包进入发布系统前，应该先通过验收。验收不是只看文件能否下载，而是确认模型包完整、可校验、可加载、可回滚。

| 验收项 | 要求 |
| --- | --- |
| manifest | 字段完整，包含模型、runtime、App、schema 和 checksum |
| 文件完整性 | 所有声明文件存在，大小和 checksum 匹配 |
| 加载测试 | 在目标 runtime 上能正常初始化 |
| 推理冒烟 | 使用固定样本输出合法结构 |
| 兼容性 | min app version、系统版本和设备架构判断正确 |
| 授权 | license 文件和来源信息可追溯 |
| 回滚 | 上一稳定版本仍可被加载 |

如果模型包缺少 manifest 或 checksum，客户端就无法判断它是否可用。如果模型包没有固定冒烟样本，发布系统也无法在上线前发现明显损坏。模型包越大，越不能依赖人工检查。

## 9.12 下载状态机

下载模型时，状态机要比普通图片资源更严格。推荐至少区分：

```text
idle
  -> checking_compatibility
  -> waiting_for_network
  -> waiting_for_storage
  -> downloading
  -> verifying
  -> installing
  -> ready
  -> failed
```

其中 `failed` 不应该是唯一失败状态。工程上还需要记录失败原因：

| 失败原因 | 用户体验 | 工程处理 |
| --- | --- | --- |
| 网络不可用 | 提示稍后下载 | 等待网络恢复 |
| 空间不足 | 提示释放空间 | 不继续重试 |
| 校验失败 | 提示下载失败 | 删除临时文件并上报 |
| 不兼容 | 隐藏或禁用功能 | 等待 App 升级 |
| 加载失败 | 切回旧版本 | 上报模型包问题 |

状态机要持久化。App 被杀掉、系统重启或用户切后台后，下载状态仍要能恢复。否则容易出现临时文件残留、重复下载或半安装状态。

## 9.13 断点续传与原子安装

大模型文件下载时间长，必须考虑断点续传。断点续传要记录已下载范围、目标版本、文件大小和校验信息。如果服务端模型版本已经变化，旧的断点文件不能继续复用。

安装过程建议分为三个目录：

```text
tmp_download/   下载中的临时文件
staging/        校验完成、等待安装的文件
active/         当前正在使用的稳定版本
```

流程可以是：

1. 下载到 `tmp_download`。
2. 下载完成后计算 checksum。
3. 校验通过后移动到 `staging`。
4. 停止旧模型的新任务入口。
5. 将 `staging` 原子切换为 `active`。
6. 加载新模型做冒烟测试。
7. 冒烟失败则恢复旧 `active`。

不要边下载边覆盖正在使用的模型文件。移动端 App 随时可能被系统杀掉，半覆盖状态会造成无法加载、重复崩溃或功能不可恢复。

## 9.14 版本命名与兼容规则

模型版本命名要表达兼容关系，而不是随意递增。可以把版本拆成：

```text
model_family / capability_version / package_revision
```

例如 `local-ocr / 2.1 / r3` 表示 OCR 能力的 2.1 版，第 3 个包修订。能力版本变化通常意味着输出行为或 schema 可能变化；包修订可能只是重新压缩、替换 tokenizer 小文件或修复 manifest。

兼容规则建议写入 manifest：

| 字段 | 作用 |
| --- | --- |
| `min_app_version` | 低于该 App 版本不得启用 |
| `max_app_version` | 必要时阻止过新 App 加载旧模型 |
| `runtime_version` | 绑定推理引擎能力 |
| `schema_version` | 绑定输入输出结构 |
| `index_version` | 判断旧索引是否可复用 |
| `rollback_to` | 指定可回退的稳定版本 |

版本治理的关键是让客户端可以自动决策。不要把“这个模型能不能用”写在运营备注里，也不要依赖人工记忆。

## 9.15 多模型依赖关系

端侧 AI 功能经常依赖多个模型和资源。例如本地文档理解可能依赖 OCR、版面分析、embedding、摘要模型和检索索引。任何一个资源版本不兼容，整条链路都会出错。

建议为功能定义能力包：

```text
document_ai_bundle:
  ocr_model: ocr-v2.1-r3
  layout_model: layout-v1.4-r1
  embedding_model: emb-v3.0-r2
  summarizer_model: sum-v1.2-r5
  index_schema: doc-index-v4
```

客户端加载功能时，不是逐个随意选择最新模型，而是加载一个经过验证的 bundle。bundle 通过灰度发布和回滚，内部资源版本保持一致。

这种方式能避免“部分资源更新”导致的问题。比如 embedding 模型升级后，如果索引 schema 没有同步升级，检索结果可能变差但不一定崩溃，更难发现。bundle 化可以把兼容性问题提前到发布阶段处理。

## 9.16 运营与成本视角

模型分发还会影响运营成本。大模型下载会消耗 CDN 流量，频繁更新会增加用户等待和失败率。发布策略要考虑：

- 是否所有用户都需要下载。
- 是否可以按地区、语言、设备能力分发。
- 是否允许用户仅在 Wi-Fi 下下载。
- 是否需要企业客户内部分发包。
- 是否需要离线安装包。
- 是否有下载失败后的客服和诊断入口。

对于低频能力，不要默认全量预下载。对于高频核心能力，要在包体和首次体验之间做清楚取舍。端侧 AI 的成本不只在训练和推理，也在模型到达设备的每一次下载、更新和回滚。

## 本章小结

模型版本管理是产品工程问题，不是下载文件那么简单。移动端团队必须设计包体预算、模型包结构、下载校验、版本兼容、灰度、回滚和清理策略。只有模型能够安全到达设备，并在出问题时快速退出，端侧 AI 才能成为可运营能力。
