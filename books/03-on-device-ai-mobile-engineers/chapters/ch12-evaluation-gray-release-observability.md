# 第 12 章 评测、灰度与线上观测

## 本章导读

端侧 AI 发布前需要离线评测，发布后需要灰度和观测。模型效果、性能、崩溃和用户体验都要进入指标体系。与普通功能不同，端侧 AI 的结果可能受模型版本、runtime、设备性能、输入分布和用户授权影响。只在实验数据上看准确率，无法证明它适合发布。

本章讨论端侧 AI 的发布闭环：离线评测、设备验证、灰度发布、线上观测、告警、回滚和复盘。

## 学习目标

- 设计端侧 AI 评测任务集。
- 建立灰度、回滚和观测方案。
- 区分模型质量问题和移动端工程问题。
- 能为端侧 AI 功能制定发布门槛。

## 12.1 评测不只看模型效果

模型评测通常关注准确率、召回率、BLEU、ROUGE、人工打分等指标。移动端发布还要看：

- 是否能在目标设备运行。
- 首次加载是否可接受。
- 推理是否稳定。
- 内存峰值是否安全。
- 电量和温度是否可控。
- 权限拒绝时是否可用。
- 结果是否有可解释证据。
- 是否有降级和回滚。

一个模型在服务器评测集上表现好，但在低端 Android 设备上加载失败，仍然不能发布。

## 12.2 评测任务集

端侧 AI 评测任务集应包含真实移动端输入：

| 任务 | 样本 |
| --- | --- |
| 文本分类 | 短文本、长文本、多语言、错别字 |
| 语音命令 | 安静、噪声、耳机、蓝牙、方言 |
| 图片识别 | 大图、长截图、模糊、夜景、旋转 |
| 本地检索 | 新增、删除、重复、旧数据 |
| 隐私检测 | 手机号、地址、头像、二维码、误报样本 |
| 性能 | 低端设备、大输入、连续任务 |

评测集要区分开发集、回归集和发布集。发布集不应被频繁用于调参，否则会失去检验价值。

## 12.3 指标体系

| 维度 | 指标 |
| --- | --- |
| 效果 | 准确率、召回率、用户接受率 |
| 性能 | 延迟、内存、电量、温度 |
| 稳定性 | 崩溃、ANR、模型加载失败 |
| 体验 | 取消率、重试率、降级率 |
| 隐私 | 本地处理比例、上传数据类型 |

还应记录版本维度：

- App 版本。
- 模型版本。
- runtime 版本。
- 配置版本。
- 设备型号。
- 系统版本。

没有版本维度，线上指标变化很难定位。比如加载失败率上升，可能是模型版本问题，也可能是某个系统版本兼容问题。

## 12.4 设备矩阵

端侧 AI 必须在设备矩阵上验证：

- 高端设备。
- 主流设备。
- 低端设备。
- 老系统版本。
- 新系统版本。
- 不同芯片或硬件加速能力。
- 不同语言和地区设置。

设备矩阵不必覆盖所有机型，但要覆盖目标用户的主要分布。对于性能敏感功能，低端设备结果比高端设备更有决策价值。

## 12.5 灰度策略

端侧 AI 灰度可以按多维度控制：

| 维度 | 说明 |
| --- | --- |
| 用户比例 | 1%、5%、20%、全量 |
| App 版本 | 只对支持版本开启 |
| 设备能力 | 排除低内存或不支持设备 |
| 地区语言 | 按语言模型能力开放 |
| 模型版本 | 新旧模型并行观察 |
| 功能入口 | 先隐藏入口，再逐步开放 |

灰度阶段要设置停止条件。例如加载失败率超过阈值、崩溃率上升、P95 延迟超预算、用户取消率异常，就暂停扩大。

## 12.6 线上观测

线上观测要在保护隐私的前提下记录：

```text
feature
model_version
runtime_version
device_class
input_size_bucket
load_time_ms
inference_time_ms
result_status
fallback_reason
user_action
```

`input_size_bucket` 代表输入大小区间，不记录原始输入。`result_status` 可以是 success、low_confidence、timeout、cancelled、model_load_failed。`fallback_reason` 记录降级原因。

观测目标不是收集更多数据，而是回答发布问题：功能是否稳定、哪些设备有问题、哪个版本有回归、用户是否接受。

## 12.7 告警与回滚

端侧 AI 应设置告警：

- 模型下载失败率异常。
- 模型加载失败率异常。
- 推理超时率异常。
- 崩溃或 ANR 上升。
- 降级率上升。
- 用户取消率上升。
- 低置信度比例异常。

回滚手段包括：

- 关闭功能入口。
- 回退模型版本。
- 禁用特定设备或系统版本。
- 切换到云端模式。
- 降低输入尺寸或关闭后台任务。

回滚要提前演练。不要等线上事故发生后才发现模型已经下载到本地且无法禁用。

## 12.8 区分问题类型

线上指标异常时，要区分：

| 问题 | 表现 | 处理 |
| --- | --- | --- |
| 模型质量 | 准确率低、误报多、低置信度高 | 回退模型或补评测集 |
| 性能 | 延迟高、内存高、发热 | 优化 runtime、降级设备 |
| 分发 | 下载失败、校验失败 | 修复 CDN、断点续传、版本配置 |
| 体验 | 取消率高、使用率低 | 调整入口、文案、等待状态 |
| 隐私 | 用户拒绝上传、投诉 | 收缩数据范围、增强透明度 |

不要把所有问题都归因于模型。移动端 AI 的线上问题常常来自工程链路。

## 12.9 发布门槛

发布前建议设定门槛：

| 类型 | 门槛示例 |
| --- | --- |
| 效果 | 发布集指标不低于上一版本 |
| 性能 | P95 延迟低于预算 |
| 稳定性 | 加载失败率低于阈值 |
| 兼容 | 主力设备矩阵通过 |
| 隐私 | 日志不包含原始输入 |
| 体验 | 低置信度和失败状态有明确文案 |
| 回滚 | 远程关闭和模型回退已验证 |

门槛应写入发布检查清单，而不是口头约定。

## 12.10 复盘机制

每次模型或端侧 AI 功能发布后，应复盘：

- 哪些指标符合预期？
- 哪些设备表现异常？
- 哪些输入类型失败最多？
- 是否出现隐私或体验投诉？
- 是否触发降级或回滚？
- 评测集是否遗漏真实场景？
- 下次发布要更新哪些门槛？

复盘结果要反哺评测集、设备矩阵、性能预算和体验文案。端侧 AI 的发布能力，是在多次发布中逐步成熟的。

## 12.11 实验设计

端侧 AI 的灰度不是简单开关。团队通常需要比较新旧模型、新旧 runtime、新旧提示词、新旧体验入口。实验设计要提前规定分组、样本量、观察窗口和停止条件。

常见实验类型包括：

| 实验 | 目的 |
| --- | --- |
| 新旧模型对比 | 判断效果、性能和失败率变化 |
| runtime 对比 | 判断加载、延迟、内存和兼容性 |
| 入口文案对比 | 判断用户是否理解功能边界 |
| 本地与云端对比 | 判断本地能力是否足够替代云端 |
| 下载策略对比 | 判断首次使用和下载完成率 |

实验要避免把多个变量混在一起。例如同时更换模型、文案和下载策略，如果指标变差，很难知道原因。对移动端 AI 来说，变量拆分比快速全量更重要。

实验指标也要分层。模型效果提升但取消率上升，可能说明等待体验变差；加载成功率提升但隐私投诉增加，可能说明文案或授权边界不清楚。实验结论要同时看效果、性能、体验和隐私。

## 12.12 线上看板

端侧 AI 的看板应围绕发布决策设计，而不是堆满指标。一个可用看板至少包含：

| 模块 | 指标 |
| --- | --- |
| 覆盖 | 功能开启人数、模型下载人数、模型就绪人数 |
| 分发 | 下载成功率、校验失败率、平均下载耗时 |
| 加载 | 加载成功率、加载耗时、加载失败原因 |
| 推理 | 成功率、P50/P95/P99 延迟、超时率 |
| 体验 | 使用率、取消率、重试率、用户接受率 |
| 降级 | 低置信度、设备不支持、云端兜底比例 |
| 稳定性 | 崩溃、ANR、OOM、发热相关停止 |

看板必须支持按版本和设备切分：

- App 版本。
- 模型版本。
- runtime 版本。
- 操作系统版本。
- 设备等级。
- 地区和语言。
- 功能开关版本。

没有切分能力，看板只能告诉团队“出问题了”，不能告诉团队“哪里出问题”。端侧 AI 的问题常常集中在某类设备、某个模型版本或某个输入大小区间。

## 12.13 事故演练

端侧模型一旦全量分发，事故处理会比服务端功能更慢。团队应在正式发布前演练：

1. 远程关闭功能入口。
2. 回退模型版本。
3. 阻止坏版本继续下载。
4. 清理本地坏包。
5. 限制某类设备继续使用。
6. 切换到云端兜底。
7. 发布用户可理解的错误提示。

演练要验证实际链路，而不是只确认“理论上可以回滚”。例如远程配置下发后，客户端多久生效？已经下载的坏模型是否会继续被加载？用户离线时是否仍会使用坏版本？这些问题必须提前知道。

事故演练还要覆盖隐私类问题。如果发现日志误上传了敏感字段，团队需要能停止采集、清理管道、确认影响范围，并发布修复版本。端侧 AI 的事故不一定是崩溃，也可能是数据处理边界错误。

## 12.14 评测集演进

评测集不能一次建完。每次发布后，真实线上问题都应该反哺评测集：

| 来源 | 进入评测集的方式 |
| --- | --- |
| 用户反馈 | 脱敏后加入回归样本 |
| 低置信度样本 | 抽样人工标注 |
| 性能异常 | 加入大输入和低端设备用例 |
| 误报漏报 | 标记为边界样本 |
| 隐私投诉 | 加入敏感数据治理用例 |

评测集要有版本。新模型不能只在最新评测集上看分数，还要对比上一版发布集、关键回归集和事故样本集。否则团队可能修复一个问题，又引入另一个老问题。

对于隐私和安全类样本，评测集本身也要治理。真实用户样本需要脱敏、授权和访问控制，不能为了评测而制造新的数据风险。

## 12.15 发布决策会

当端侧 AI 功能影响范围较大时，建议在灰度前开一次发布决策会。参会角色不需要很多，但要覆盖关键责任：

| 角色 | 关注点 |
| --- | --- |
| 客户端 | 性能、兼容、降级、回滚 |
| 算法 | 效果、边界、模型版本 |
| 服务端 | 配置、下载、云端兜底 |
| 测试 | 设备矩阵、回归集、事故演练 |
| 产品 | 入口、文案、用户预期 |
| 安全/合规 | 数据流、日志、授权、删除 |

会议输出应该是清晰结论：是否进入灰度、灰度比例、观察指标、停止条件、负责人和回滚路径。不要用“先上看看”作为发布策略。端侧 AI 一旦分发到设备，错误恢复成本比普通 UI 改动更高。

## 12.16 从一次发布到发布体系

成熟团队不会把每个端侧 AI 功能都当成一次性项目。随着模型数量增加，团队需要形成发布体系：

- 统一模型包规范。
- 统一评测任务和回归集。
- 统一设备矩阵。
- 统一灰度平台。
- 统一观测字段。
- 统一隐私审计。
- 统一事故演练。

这样做的目的不是增加流程，而是降低每次发布的不确定性。第一版端侧 AI 功能可以轻量，但从第二个、第三个功能开始，如果没有共用体系，团队会反复踩同样的问题：下载失败、版本不兼容、日志泄露、低端设备卡顿、回滚困难。

## 本章小结

端侧 AI 的发布不能只看模型效果。移动端工程指标、用户体验指标、隐私指标和版本维度同样重要。团队需要用离线评测、设备矩阵、灰度、线上观测、告警和回滚构成闭环，让模型能力在真实设备上可控发布。
