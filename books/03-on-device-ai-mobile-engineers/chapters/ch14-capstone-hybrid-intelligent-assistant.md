# 第 14 章 综合项目：构建端云协同智能助手

## 本章导读

本章规划一个综合项目：构建端云协同智能助手。它结合本地意图识别、本地检索、云端生成、隐私过滤、模型版本、灰度和观测。这个项目不是为了展示“模型有多强”，而是为了把前面章节的工程方法串起来：如何定义边界、如何端云分工、如何保护用户数据、如何在弱网和低端设备上降级、如何上线和回滚。

综合项目的目标是做一个移动端可发布的智能助手骨架。它可以回答用户授权内容中的问题，执行少量本地命令，在复杂问题上请求云端增强，并在隐私、性能和失败路径上可控。

## 项目目标

- 本地处理简单命令和隐私过滤。
- 本地检索用户授权内容。
- 云端处理复杂生成。
- 弱网时提供降级答案。
- 记录性能、失败和用户反馈。
- 支持模型和索引版本管理。
- 支持灰度、远程关闭和回滚。

## 14.1 需求边界

先定义助手支持什么：

| 能力 | 说明 |
| --- | --- |
| 本地命令 | 打开收藏、搜索笔记、开始录音、切换模式 |
| 本地检索 | 查询用户授权的笔记、摘要、收藏 |
| 本地隐私过滤 | 检测手机号、邮箱、地址、头像、敏感截图 |
| 云端增强 | 用户确认后基于选中片段生成完整答案 |
| 离线模式 | 提供本地搜索、引用和简单摘要 |

也要定义不支持什么：

- 不直接操作支付、删除、发消息等高风险动作。
- 不自动读取未授权相册或文件。
- 不在用户无确认时上传原始截图、录音或文档。
- 不承诺回答最新互联网事实。
- 不把模型输出作为唯一事实来源。

边界写清楚后，项目才可评测、可发布。

## 14.2 系统架构

可以采用下面的分层：

```text
Assistant UI
  -> Assistant StateMachine
  -> Task Router
  -> Local Intent / Local Retrieval / Privacy Filter
  -> Cloud Generation Gateway
  -> Observability / Model Manager / Index Manager
```

各模块职责：

| 模块 | 职责 |
| --- | --- |
| Assistant UI | 输入、状态、引用、确认、反馈 |
| StateMachine | 处理 loading、cancel、retry、fallback |
| Task Router | 决定本地、云端或端云协同 |
| Local Intent | 识别少量本地命令 |
| Local Retrieval | 召回用户授权内容 |
| Privacy Filter | 上云前脱敏和风险提示 |
| Cloud Gateway | 调用服务端模型并处理超时 |
| Model Manager | 模型下载、校验、版本和回滚 |
| Index Manager | 本地索引构建、更新和删除 |
| Observability | 指标、错误、用户反馈 |

这个架构强调“助手”不是一个模型，而是一组受控能力组合。

## 14.3 端云路由

端云路由表是项目核心：

| 用户任务 | 默认路径 | 上云条件 | 降级 |
| --- | --- | --- | --- |
| 打开收藏 | 本地意图 | 不上云 | 展示手动入口 |
| 搜索本地笔记 | 本地检索 | 用户要求总结时 | 展示引用 |
| 解释截图 | 本地 OCR + 脱敏 | 用户确认上传脱敏内容 | 只展示 OCR |
| 总结长文档 | 本地片段召回 | 网络可用且用户确认 | 本地短摘要 |
| 查询最新知识 | 云端 | 用户输入非隐私内容 | 提示需要联网 |

路由表要进入仓库文档和测试，而不是散落在代码里。

## 14.4 本地数据索引

本地索引只处理用户授权内容：

1. 用户选择笔记本、收藏夹或文档目录。
2. 抽取文本和元数据。
3. 按标题、段落或时间分段。
4. 生成 embedding。
5. 写入本地加密索引。
6. 记录 source_id、chunk_id、模型版本和更新时间。
7. 原文删除时同步删除索引。

查询时先本地召回，再展示引用。不要直接把召回片段全部交给云端。用户应能看到将使用哪些片段。

## 14.5 隐私过滤

隐私过滤位于上云前。它处理：

- 手机号。
- 邮箱。
- 地址。
- 证件号。
- 联系人。
- 头像和人脸。
- 二维码。
- 企业敏感词。

过滤结果分三类：

| 结果 | 处理 |
| --- | --- |
| 无敏感内容 | 可继续，但仍需遵守任务策略 |
| 可脱敏内容 | 展示脱敏预览，用户确认后上传 |
| 高敏内容 | 默认不上云，只提供本地结果 |

隐私过滤不是一次正则替换，而是产品决策点。

## 14.6 状态机

助手状态机可以包含：

```text
idle
input_received
routing
local_processing
need_user_confirmation
cloud_processing
showing_result
fallback
failed
cancelled
```

状态机的价值是让等待、取消、降级和失败都可测试。比如云端超时后进入 fallback，展示本地引用；用户取消后停止云端请求并释放本地资源；隐私过滤命中后进入 need_user_confirmation。

## 14.7 示例任务：搜索本地会议笔记

用户输入：“上次关于预算的会议结论是什么？”

流程：

1. Task Router 判断这是本地知识查询。
2. Local Retrieval 在授权笔记中召回相关片段。
3. UI 展示 3 条引用。
4. 如果用户只需要查看引用，任务结束。
5. 如果用户点击“生成总结”，进入云端确认。
6. Privacy Filter 检查片段是否包含敏感信息。
7. 用户确认后上传脱敏片段。
8. Cloud Gateway 返回总结。
9. UI 展示总结和引用来源。

这个流程保证即使没有网络，用户也能看到本地引用。

## 14.8 示例任务：解释截图错误

用户选择一张 App 错误截图。

流程：

1. 本地读取截图。
2. 本地 OCR 提取错误文案。
3. Privacy Filter 检测手机号、头像、地址。
4. 如果包含敏感区域，展示遮挡预览。
5. 本地判断截图类型是“错误页面”。
6. 用户确认后上传脱敏文字。
7. 云端生成解释和下一步建议。
8. 如果云端失败，只展示 OCR 和通用排查建议。

这个任务体现了端云协同：本地负责输入治理和隐私，云端负责复杂解释。

## 14.9 评测集

项目至少需要四类评测集：

| 评测集 | 样本 |
| --- | --- |
| 意图识别 | 常用命令、多语言、错别字、拒答样本 |
| 本地检索 | 查询、期望引用、删除一致性 |
| 隐私过滤 | 手机号、邮箱、地址、头像、误报样本 |
| 端云路由 | 本地、云端、确认、降级路径 |

评测指标包括：命令识别准确率、召回命中率、隐私漏检率、云端确认率、降级成功率、P95 延迟和用户接受率。

## 14.10 发布计划

建议按阶段发布：

1. 内部版本：只启用本地命令和本地检索。
2. 小流量：启用隐私过滤和脱敏预览。
3. 灰度：开放云端增强，但默认需要确认。
4. 扩大：加入更多数据源和场景。
5. 全量：建立固定观测和回滚机制。

每阶段都要有停止条件。比如隐私漏检、模型加载失败、崩溃上升或用户取消率异常，都应暂停扩大。

一个示例目录：

目录设计要让新工程师看得出：这里不是一个“AI 调用工具类”，而是一套端云协同系统。

## 工作流

1. 定义任务边界。
2. 设计端云路由。
3. 建立本地数据索引。
4. 接入云端生成。
5. 增加隐私过滤。
6. 建立评测集。
7. 灰度发布。
8. 线上观测和回滚。

## 14.12 验收清单

- 本地路径可离线运行。
- 云端路径有超时和降级。
- 用户数据进入云端前有过滤。
- 模型和索引有版本。
- 性能指标可观测。
- 失败路径有用户提示。
- 删除原文后索引同步删除。
- 模型和云端配置可灰度回滚。
- 高风险动作有用户确认。
- 结果展示引用或依据。

## 14.13 最小可行版本

综合项目不要一开始就做完整助手。建议先做最小可行版本：

| 模块 | MVP 范围 |
| --- | --- |
| 输入 | 文本输入，不接语音和图片 |
| 本地检索 | 只索引用户选择的 20 篇笔记 |
| 本地命令 | 支持搜索、打开收藏、切换模式 |
| 云端增强 | 只对用户确认的引用片段生成摘要 |
| 隐私过滤 | 先覆盖手机号、邮箱、地址和证件号 |
| 观测 | 记录加载、检索、云端超时和用户取消 |
| 回滚 | 远程关闭云端增强，保留本地检索 |

MVP 的目标是验证架构和闭环，而不是验证所有模型能力。只要能跑通“本地召回 -> 用户确认 -> 脱敏 -> 云端生成 -> 引用展示 -> 反馈记录”，项目就具备继续扩展的基础。

如果第一版就接入语音、图片、长文档、多数据源和多模型，很容易在权限、性能、隐私和状态机上同时失控。综合项目的正确节奏是先做闭环，再扩能力。

## 14.14 关键接口设计

项目中建议把接口拆成稳定协议，而不是让 UI 直接调用模型：

```text
AssistantRequest
AssistantRoute
AssistantEvidence
AssistantResult
AssistantError
```

示例字段：

```json
{
  "request_id": "req_001",
  "input_type": "text",
  "user_text": "上次关于预算的会议结论是什么？",
  "allowed_sources": ["notes"],
  "privacy_mode": "local_first"
}
```

路由结果可以是：

```json
{
  "route": "local_retrieval_then_cloud",
  "requires_confirmation": true,
  "evidence_count": 3,
  "fallback": "show_local_evidence"
}
```

结构化接口的好处是测试清楚。测试可以断言某个输入必须走本地，某个输入必须要求确认，某个输入必须拒绝上云。不要让端云路由隐藏在自然语言 prompt 或零散 if 判断里。

## 14.15 错误与降级表

综合项目要把错误当成正常路径处理：

| 错误 | 用户体验 | 系统动作 |
| --- | --- | --- |
| 本地索引未就绪 | 展示“正在建立本地索引” | 返回最近可用结果或等待 |
| 模型未下载 | 提示下载本地组件 | 允许稍后下载 |
| 云端超时 | 展示本地引用和简短建议 | 取消云端请求 |
| 隐私高风险 | 默认不上云 | 只展示本地结果 |
| 低置信度命令 | 要求用户确认 | 不执行动作 |
| 权限撤销 | 入口降级 | 停止索引和后台任务 |
| 存储不足 | 暂停索引 | 清理可重建缓存 |

降级路径要在设计阶段写清楚。否则实现时容易出现“云端失败就报错”“索引未完成就空白”“模型未下载就无限 loading”。端云协同助手的质量，很大程度取决于失败路径是否可用。

## 14.16 开发迭代路线

建议按 6 个迭代推进：

1. **任务协议与状态机**：定义 request、route、result、error、cancel。
2. **本地检索闭环**：建立授权数据源、分段、索引、引用展示。
3. **隐私过滤**：上云前检测敏感字段，提供脱敏预览。
4. **云端增强**：只上传确认后的片段，处理超时和失败。
5. **模型与索引版本**：加入 manifest、index_version、回滚和重建。
6. **灰度与观测**：接入指标、告警、远程关闭和用户反馈。

每个迭代都应可独立验收。比如第二个迭代完成后，即使没有云端增强，也应该能离线搜索和展示引用。这样项目不会因为后续模型能力迟迟未稳定而整体不可发布。

## 14.17 测试计划

综合项目的测试应覆盖功能、性能、隐私和发布：

| 测试类型 | 用例 |
| --- | --- |
| 单元测试 | 路由规则、隐私过滤、错误映射 |
| 集成测试 | 本地检索到云端增强完整流程 |
| 设备测试 | 低端设备、大输入、前后台切换 |
| 离线测试 | 无网络、弱网、云端超时 |
| 权限测试 | 相册、文件、麦克风拒绝和撤销 |
| 数据测试 | 删除原文后索引同步删除 |
| 灰度测试 | 远程关闭、模型回滚、配置冻结 |

测试样本要覆盖正例和反例。比如本地命令不仅要测试“打开收藏”，还要测试“删除所有收藏”这种高风险命令是否拒绝执行。隐私过滤不仅要测试手机号命中，还要测试普通数字不被误报。

## 14.18 读者练习

读者可以基于本章设计完成三个练习：

1. 为一个“本地笔记问答助手”写端云路由表，至少包含 8 类用户任务。
2. 为截图解释功能写隐私过滤规则，区分可脱敏和高风险内容。
3. 为本地索引功能写发布检查清单，覆盖下载、索引、删除和回滚。

练习的重点不是写出复杂模型代码，而是把任务边界、数据边界和失败路径说清楚。移动端 AI 工程的成熟度，首先体现在这些设计文档和测试用例中。

## 14.19 项目复盘模板

综合项目上线后，可以用下面模板复盘：

```text
版本：
开放比例：
启用模型：
启用数据源：
核心指标：
主要失败原因：
用户反馈：
隐私或安全问题：
性能异常设备：
是否触发降级：
是否触发回滚：
评测集需要补充：
下一版计划：
```

复盘要回到事实。不要只讨论“用户觉得好不好”，而要看引用点击率、云端确认率、降级率、超时率、删除一致性、隐私命中率和低端设备表现。综合项目的价值在于长期迭代，复盘是把一次发布变成下一次改进的入口。

## 本章小结

综合项目的重点是端云分工。真正的移动智能工程不是展示模型能力，而是把能力做成可发布、可维护、可回滚的产品系统。一个合格的端云协同智能助手，应同时具备本地能力、云端增强、隐私过滤、状态机、评测集、观测和回滚机制。
