# 第 14 章 综合项目：构建端云协同智能助手

## 本章导读

本章规划一个综合项目：构建端云协同智能助手。它结合本地意图识别、本地检索、云端生成、隐私过滤、模型版本、灰度和观测。这个项目不是为了展示“模型有多强”，而是为了把前面章节的工程方法串起来：如何定义边界、如何端云分工、如何保护用户数据、如何在弱网和低端设备上降级、如何上线和回滚。

综合项目的目标是做一个移动端可发布的智能助手骨架。它可以回答用户授权内容中的问题，执行少量本地命令，在复杂问题上请求云端增强，并在隐私、性能和失败路径上可控。

## 项目目标

- 本地处理简单命令和隐私过滤。
- 本地检索用户授权内容。
- 云端处理复杂生成。
- 弱网时提供降级答案。
- 记录性能、失败和用户反馈。
- 支持模型和索引版本管理。
- 支持灰度、远程关闭和回滚。

## 14.1 需求边界

先定义助手支持什么：

| 能力 | 说明 |
| --- | --- |
| 本地命令 | 打开收藏、搜索笔记、开始录音、切换模式 |
| 本地检索 | 查询用户授权的笔记、摘要、收藏 |
| 本地隐私过滤 | 检测手机号、邮箱、地址、头像、敏感截图 |
| 云端增强 | 用户确认后基于选中片段生成完整答案 |
| 离线模式 | 提供本地搜索、引用和简单摘要 |

也要定义不支持什么：

- 不直接操作支付、删除、发消息等高风险动作。
- 不自动读取未授权相册或文件。
- 不在用户无确认时上传原始截图、录音或文档。
- 不承诺回答最新互联网事实。
- 不把模型输出作为唯一事实来源。

边界写清楚后，项目才可评测、可发布。

## 14.2 系统架构

可以采用下面的分层：

```text
Assistant UI
  -> Assistant StateMachine
  -> Task Router
  -> Local Intent / Local Retrieval / Privacy Filter
  -> Cloud Generation Gateway
  -> Observability / Model Manager / Index Manager
```

各模块职责：

| 模块 | 职责 |
| --- | --- |
| Assistant UI | 输入、状态、引用、确认、反馈 |
| StateMachine | 处理 loading、cancel、retry、fallback |
| Task Router | 决定本地、云端或端云协同 |
| Local Intent | 识别少量本地命令 |
| Local Retrieval | 召回用户授权内容 |
| Privacy Filter | 上云前脱敏和风险提示 |
| Cloud Gateway | 调用服务端模型并处理超时 |
| Model Manager | 模型下载、校验、版本和回滚 |
| Index Manager | 本地索引构建、更新和删除 |
| Observability | 指标、错误、用户反馈 |

这个架构强调“助手”不是一个模型，而是一组受控能力组合。

## 14.3 端云路由

端云路由表是项目核心：

| 用户任务 | 默认路径 | 上云条件 | 降级 |
| --- | --- | --- | --- |
| 打开收藏 | 本地意图 | 不上云 | 展示手动入口 |
| 搜索本地笔记 | 本地检索 | 用户要求总结时 | 展示引用 |
| 解释截图 | 本地 OCR + 脱敏 | 用户确认上传脱敏内容 | 只展示 OCR |
| 总结长文档 | 本地片段召回 | 网络可用且用户确认 | 本地短摘要 |
| 查询最新知识 | 云端 | 用户输入非隐私内容 | 提示需要联网 |

路由表要进入仓库文档和测试，而不是散落在代码里。

## 14.4 本地数据索引

本地索引只处理用户授权内容：

1. 用户选择笔记本、收藏夹或文档目录。
2. 抽取文本和元数据。
3. 按标题、段落或时间分段。
4. 生成 embedding。
5. 写入本地加密索引。
6. 记录 source_id、chunk_id、模型版本和更新时间。
7. 原文删除时同步删除索引。

查询时先本地召回，再展示引用。不要直接把召回片段全部交给云端。用户应能看到将使用哪些片段。

## 14.5 隐私过滤

隐私过滤位于上云前。它处理：

- 手机号。
- 邮箱。
- 地址。
- 证件号。
- 联系人。
- 头像和人脸。
- 二维码。
- 企业敏感词。

过滤结果分三类：

| 结果 | 处理 |
| --- | --- |
| 无敏感内容 | 可继续，但仍需遵守任务策略 |
| 可脱敏内容 | 展示脱敏预览，用户确认后上传 |
| 高敏内容 | 默认不上云，只提供本地结果 |

隐私过滤不是一次正则替换，而是产品决策点。

## 14.6 状态机

助手状态机可以包含：

```text
idle
input_received
routing
local_processing
need_user_confirmation
cloud_processing
showing_result
fallback
failed
cancelled
```

状态机的价值是让等待、取消、降级和失败都可测试。比如云端超时后进入 fallback，展示本地引用；用户取消后停止云端请求并释放本地资源；隐私过滤命中后进入 need_user_confirmation。

## 14.7 示例任务：搜索本地会议笔记

用户输入：“上次关于预算的会议结论是什么？”

流程：

1. Task Router 判断这是本地知识查询。
2. Local Retrieval 在授权笔记中召回相关片段。
3. UI 展示 3 条引用。
4. 如果用户只需要查看引用，任务结束。
5. 如果用户点击“生成总结”，进入云端确认。
6. Privacy Filter 检查片段是否包含敏感信息。
7. 用户确认后上传脱敏片段。
8. Cloud Gateway 返回总结。
9. UI 展示总结和引用来源。

这个流程保证即使没有网络，用户也能看到本地引用。

## 14.8 示例任务：解释截图错误

用户选择一张 App 错误截图。

流程：

1. 本地读取截图。
2. 本地 OCR 提取错误文案。
3. Privacy Filter 检测手机号、头像、地址。
4. 如果包含敏感区域，展示遮挡预览。
5. 本地判断截图类型是“错误页面”。
6. 用户确认后上传脱敏文字。
7. 云端生成解释和下一步建议。
8. 如果云端失败，只展示 OCR 和通用排查建议。

这个任务体现了端云协同：本地负责输入治理和隐私，云端负责复杂解释。

## 14.9 评测集

项目至少需要四类评测集：

| 评测集 | 样本 |
| --- | --- |
| 意图识别 | 常用命令、多语言、错别字、拒答样本 |
| 本地检索 | 查询、期望引用、删除一致性 |
| 隐私过滤 | 手机号、邮箱、地址、头像、误报样本 |
| 端云路由 | 本地、云端、确认、降级路径 |

评测指标包括：命令识别准确率、召回命中率、隐私漏检率、云端确认率、降级成功率、P95 延迟和用户接受率。

## 14.10 发布计划

建议按阶段发布：

1. 内部版本：只启用本地命令和本地检索。
2. 小流量：启用隐私过滤和脱敏预览。
3. 灰度：开放云端增强，但默认需要确认。
4. 扩大：加入更多数据源和场景。
5. 全量：建立固定观测和回滚机制。

每阶段都要有停止条件。比如隐私漏检、模型加载失败、崩溃上升或用户取消率异常，都应暂停扩大。

## 工作流

1. 定义任务边界。
2. 设计端云路由。
3. 建立本地数据索引。
4. 接入云端生成。
5. 增加隐私过滤。
6. 建立评测集。
7. 灰度发布。
8. 线上观测和回滚。

## 14.12 验收清单

- 本地路径可离线运行。
- 云端路径有超时和降级。
- 用户数据进入云端前有过滤。
- 模型和索引有版本。
- 性能指标可观测。
- 失败路径有用户提示。
- 删除原文后索引同步删除。
- 模型和云端配置可灰度回滚。
- 高风险动作有用户确认。
- 结果展示引用或依据。

## 14.13 最小可行版本

综合项目不要一开始就做完整助手。建议先做最小可行版本：

| 模块 | MVP 范围 |
| --- | --- |
| 输入 | 文本输入，不接语音和图片 |
| 本地检索 | 只索引用户选择的 20 篇笔记 |
| 本地命令 | 支持搜索、打开收藏、切换模式 |
| 云端增强 | 只对用户确认的引用片段生成摘要 |
| 隐私过滤 | 先覆盖手机号、邮箱、地址和证件号 |
| 观测 | 记录加载、检索、云端超时和用户取消 |
| 回滚 | 远程关闭云端增强，保留本地检索 |

MVP 的目标是验证架构和闭环，而不是验证所有模型能力。只要能跑通“本地召回 -> 用户确认 -> 脱敏 -> 云端生成 -> 引用展示 -> 反馈记录”，项目就具备继续扩展的基础。

如果第一版就接入语音、图片、长文档、多数据源和多模型，很容易在权限、性能、隐私和状态机上同时失控。综合项目的正确节奏是先做闭环，再扩能力。

## 14.14 关键接口设计

项目中建议把接口拆成稳定协议，而不是让 UI 直接调用模型：

```text
AssistantRequest
AssistantRoute
AssistantEvidence
AssistantResult
AssistantError
```

示例字段：

```json
{
  "request_id": "req_001",
  "input_type": "text",
  "user_text": "上次关于预算的会议结论是什么？",
  "allowed_sources": ["notes"],
  "privacy_mode": "local_first"
}
```

路由结果可以是：

```json
{
  "route": "local_retrieval_then_cloud",
  "requires_confirmation": true,
  "evidence_count": 3,
  "fallback": "show_local_evidence"
}
```

结构化接口的好处是测试清楚。测试可以断言某个输入必须走本地，某个输入必须要求确认，某个输入必须拒绝上云。不要让端云路由隐藏在自然语言 prompt 或零散 if 判断里。

## 14.15 错误与降级表

综合项目要把错误当成正常路径处理：

| 错误 | 用户体验 | 系统动作 |
| --- | --- | --- |
| 本地索引未就绪 | 展示“正在建立本地索引” | 返回最近可用结果或等待 |
| 模型未下载 | 提示下载本地组件 | 允许稍后下载 |
| 云端超时 | 展示本地引用和简短建议 | 取消云端请求 |
| 隐私高风险 | 默认不上云 | 只展示本地结果 |
| 低置信度命令 | 要求用户确认 | 不执行动作 |
| 权限撤销 | 入口降级 | 停止索引和后台任务 |
| 存储不足 | 暂停索引 | 清理可重建缓存 |

降级路径要在设计阶段写清楚。否则实现时容易出现“云端失败就报错”“索引未完成就空白”“模型未下载就无限 loading”。端云协同助手的质量，很大程度取决于失败路径是否可用。

## 14.16 开发迭代路线

建议按 6 个迭代推进：

1. **任务协议与状态机**：定义 request、route、result、error、cancel。
2. **本地检索闭环**：建立授权数据源、分段、索引、引用展示。
3. **隐私过滤**：上云前检测敏感字段，提供脱敏预览。
4. **云端增强**：只上传确认后的片段，处理超时和失败。
5. **模型与索引版本**：加入 manifest、index_version、回滚和重建。
6. **灰度与观测**：接入指标、告警、远程关闭和用户反馈。

每个迭代都应可独立验收。比如第二个迭代完成后，即使没有云端增强，也应该能离线搜索和展示引用。这样项目不会因为后续模型能力迟迟未稳定而整体不可发布。

## 14.17 测试计划

综合项目的测试应覆盖功能、性能、隐私和发布：

| 测试类型 | 用例 |
| --- | --- |
| 单元测试 | 路由规则、隐私过滤、错误映射 |
| 集成测试 | 本地检索到云端增强完整流程 |
| 设备测试 | 低端设备、大输入、前后台切换 |
| 离线测试 | 无网络、弱网、云端超时 |
| 权限测试 | 相册、文件、麦克风拒绝和撤销 |
| 数据测试 | 删除原文后索引同步删除 |
| 灰度测试 | 远程关闭、模型回滚、配置冻结 |

测试样本要覆盖正例和反例。比如本地命令不仅要测试“打开收藏”，还要测试“删除所有收藏”这种高风险命令是否拒绝执行。隐私过滤不仅要测试手机号命中，还要测试普通数字不被误报。

## 14.18 读者练习

读者可以基于本章设计完成三个练习：

1. 为一个“本地笔记问答助手”写端云路由表，至少包含 8 类用户任务。
2. 为截图解释功能写隐私过滤规则，区分可脱敏和高风险内容。
3. 为本地索引功能写发布检查清单，覆盖下载、索引、删除和回滚。

练习的重点不是写出复杂模型代码，而是把任务边界、数据边界和失败路径说清楚。移动端 AI 工程的成熟度，首先体现在这些设计文档和测试用例中。

## 14.20 交付物清单

综合项目完成后，不应该只交付代码。一个可以进入团队协作和长期维护的端云协同助手，至少应交付以下材料：

| 交付物 | 内容 |
| --- | --- |
| 架构说明 | 端云分工、模块边界、核心状态机 |
| 路由表 | 用户任务、默认路径、上云条件、降级路径 |
| 隐私边界 | 数据类型、是否上传、保留时间、删除触发 |
| 模型清单 | 本地模型、云端模型、版本和回滚方式 |
| 索引说明 | 数据源、分段规则、索引版本、迁移策略 |
| 测试集 | 意图、检索、隐私、端云路由、异常路径 |
| 观测字段 | request id、模型版本、阶段耗时、失败原因 |
| 发布计划 | 灰度比例、停止条件、回滚负责人 |
| 用户文案 | 权限、下载、隐私、失败和降级提示 |

这些交付物的价值在于降低未来维护成本。半年后模型升级、App 改版、隐私策略变化或团队成员变动时，新的工程师可以从这些材料理解系统边界，而不是从代码里猜。

## 14.21 代码实现顺序

如果把本章项目真正落到代码中，建议实现顺序如下：

1. 定义数据结构：`AssistantRequest`、`AssistantRoute`、`AssistantResult`、`AssistantError`。
2. 实现纯函数路由器，不接模型，只根据输入和环境返回路由。
3. 实现本地 mock 检索，用固定样本返回引用。
4. 实现隐私过滤规则，先覆盖手机号、邮箱、地址。
5. 实现 UI 状态机，覆盖 loading、confirm、fallback、failed。
6. 接入真实本地索引。
7. 接入云端网关。
8. 加入模型版本、索引版本和远程配置。
9. 加入观测、灰度和回滚。

这个顺序可以让项目很早就有可测试骨架。不要等模型、索引和云端全部就绪后才开始做状态机和路由。端云协同项目最容易失败的地方，不是某个模型没有接好，而是所有不确定性同时出现，导致团队不知道该先修哪一层。

## 14.22 示例验收用例

综合项目可以使用下面的验收用例：

| 用例 | 期望 |
| --- | --- |
| 用户离线搜索本地笔记 | 展示本地引用，不调用云端 |
| 用户要求总结本地笔记 | 展示引用并请求上云确认 |
| 片段包含手机号 | 展示脱敏预览，默认不上云原文 |
| 云端超时 | 保留本地结果，展示稍后重试 |
| 用户取消任务 | 停止本地任务和云端请求 |
| 删除原始笔记 | 索引结果不再返回该片段 |
| 模型新版本加载失败 | 回退旧版本或进入降级 |
| 低端设备大输入 | 限制输入规模或切云端 |

这些用例要自动化一部分，也要在真实设备上手动验证一部分。自动化适合路由、隐私过滤和错误映射；真实设备验证适合性能、权限、前后台、弱网和用户体验。

## 14.23 从项目到平台

当综合助手跑通后，团队会发现很多能力可以复用：模型包管理、索引构建、隐私过滤、路由、观测、灰度、回滚、反馈入口。这时可以把项目能力沉淀为平台能力。

平台化不意味着一开始就做大平台。更务实的做法是从重复代码中提取：

- 统一模型包 manifest。
- 统一能力查询接口。
- 统一端云路由事件。
- 统一隐私过滤结果结构。
- 统一发布检查清单。
- 统一观测字段。

当第二个、第三个端侧 AI 功能出现时，这些统一能力会明显降低成本。团队不需要重新设计下载状态机、错误码、日志白名单和回滚开关。平台化的目标，是让每个新 AI 功能都站在已有工程边界上，而不是重新发明一套风险控制。

## 14.24 最终发布判断

综合项目是否能发布，可以用四句话判断：

1. 用户知道功能会处理哪些数据。
2. 工程师知道每个任务走本地还是云端。
3. 测试知道每条失败路径如何验证。
4. 运营知道指标异常时如何停止和回滚。

如果这四句话都成立，项目就具备进入灰度的基础。如果任何一句不成立，就说明系统仍有关键不确定性。端侧 AI 的发布不是模型能力展示，而是一次跨产品、客户端、服务端、算法、测试和安全的工程协作。

## 14.25 读者实践路线

读者可以用四周时间把本章项目压缩成一个可运行原型。

第一周只做路由和本地检索。准备 20 条脱敏笔记样本，实现本地关键词检索或简单向量检索，保证用户输入问题后能看到引用片段。此时不要接云端模型，也不要做复杂 UI，重点是把数据源、片段、引用和删除关系跑通。

第二周加入隐私过滤和确认。为手机号、邮箱、地址、证件号准备测试样本，上云前展示脱敏预览。即使还没有真实云端调用，也要把“需要确认”“拒绝上云”“只显示本地结果”这些状态做出来。端云协同的可靠性，先来自状态机，而不是来自模型能力。

第三周接入云端增强。云端只接收用户确认后的片段，返回摘要或解释，并在结果中保留引用。要同时实现超时、取消、重试和本地降级。不要让云端增强破坏本地可用性：离线时仍能检索，云端失败时仍能展示引用。

第四周补齐发布工程。加入模型和索引版本，记录 request id、路由结果、阶段耗时和失败原因；准备一份发布检查清单，演练远程关闭云端增强和清理本地索引。到这一步，原型虽然还不强大，但已经具备端侧 AI 产品的关键骨架。

这条实践路线的目的，是让读者从“调用一次模型”转向“交付一条可发布链路”。真正的移动端 AI 工程能力，不是把某个模型接进 App，而是把模型能力、用户数据、系统资源、端云路由、发布观测和回滚机制放到同一个工程闭环中。

## 本章小结

> **综合项目重点**：综合项目的重点是端云分工。真正的移动智能工程不是展示模型能力，而是把能力做成可发布、可维护、可回滚的产品系统。一个合格的端云协同智能助手，应同时具备本地能力、云端增强、隐私过滤、状态机、评测集、观测和回滚机制。
