# 第 3 章 端云协同架构

## 本章导读

端侧和云端不是二选一。成熟移动端 AI 产品通常采用端云协同：本地处理低延迟、隐私敏感、离线可用和低成本任务，云端处理复杂推理、大模型生成、最新知识和集中治理任务。

端云协同的难点不在“端侧调用云端接口”，而在决策：什么时候端侧处理，什么时候上云，什么时候降级，什么时候让用户确认。这个决策如果没有工程化，就会变成散落在页面里的 if/else。

## 学习目标

- 掌握端云协同的常见架构。
- 设计本地优先、云端兜底和失败降级策略。
- 理解端侧缓存、队列和同步边界。
- 能为移动端 AI 功能设计端云路由表。

## 3.1 端云协同的基本分工

端侧适合：

- 快速分类和路由。
- 本地隐私过滤。
- 本地数据检索。
- 简单摘要和格式化。
- 离线命令。
- 输入预处理和输出后处理。

云端适合：

- 复杂长文本生成。
- 大规模知识问答。
- 多轮复杂推理。
- 高质量多模态理解。
- 集中模型更新。
- 统一审计和策略控制。

这个分工不是固定的。随着端侧模型能力提升，一些任务会从云端迁回本地；随着产品要求变化，一些任务又可能回到云端。架构应允许演进。

## 3.2 架构模式

| 模式 | 说明 | 适用场景 |
| --- | --- | --- |
| 本地优先 | 先端侧推理，失败或置信度低时上云 | 离线摘要、简单分类 |
| 云端优先 | 云端生成，本地做预处理或后处理 | 长文本生成、知识问答 |
| 分层决策 | 本地小模型做路由，云端大模型做复杂任务 | 智能助手、搜索入口 |
| 延迟容忍 | 本地先给粗结果，云端异步补全 | 图片理解、内容推荐 |
| 隐私保护 | 本地脱敏后再上云 | 截图分析、文档摘要 |

团队应该为每个 AI 功能选择明确模式，而不是让每个页面自己决定。

## 3.3 端云路由表

建议为功能建立路由表：

| 任务 | 默认路径 | 上云条件 | 本地降级 | 用户确认 |
| --- | --- | --- | --- | --- |
| 本地笔记搜索 | 端侧检索 | 无结果且用户允许联网 | 返回本地候选 | 可选 |
| 截图解释 | 本地脱敏 + 云端理解 | 截图不含敏感区域 | 只做本地 OCR | 必须 |
| 语音命令 | 端侧意图识别 | 置信度低 | 展示候选命令 | 可选 |
| 长文摘要 | 云端生成 | 网络可用且用户授权 | 本地生成粗摘要 | 必须 |

路由表的作用是把产品、隐私和工程决策显式化。没有路由表，端云协同很容易变成不可审计的隐式逻辑。

## 3.4 降级设计

端云协同必须有降级：

- 网络不可用时本地能力是否足够。
- 模型加载失败时是否回退传统逻辑。
- 云端超时时是否展示部分结果。
- 低端设备是否禁用部分能力。
- 用户拒绝上传时是否有本地替代。

降级不是异常处理的最后补丁，而是架构的一部分。用户看到的失败提示、重试按钮、离线结果和本地缓存，都应该在设计阶段确定。

## 3.5 数据边界

端云协同最敏感的是数据边界。移动端应区分：

- 永不上传：真实密钥、未授权相册、精确位置、敏感本地文件。
- 脱敏后上传：截图 OCR 结果、日志摘要、错误上下文。
- 用户确认后上传：录音、图片、文档、聊天历史。
- 可直接上传：公开查询、非个人化命令、低敏文本。

这个分类要进入代码和文档。仅靠口头约定，无法保证长期一致。

## 3.6 队列与同步

弱网环境下，端云协同需要队列：

- 待上传任务队列。
- 待下载模型队列。
- 待同步索引队列。
- 待重试生成队列。

队列要记录状态：pending、running、failed、cancelled、done。还要支持用户取消、任务过期和版本变化。否则，App 可能在网络恢复后上传用户已经不再需要的内容。

## 3.7 端云协同 Prompt 模板

```text
请为 [功能名] 设计端云协同架构。

输入：
- 用户数据类型：
- 延迟要求：
- 离线要求：
- 隐私等级：
- 目标平台：

输出：
- 默认端侧/云端路径。
- 上云条件。
- 本地降级策略。
- 用户确认点。
- 需要观测的指标。
- 需要禁止上传的数据。
```

这个模板适合架构评审，也适合在开发前让 AI coding agent 做方案分析。

## 3.8 本地优先案例：个人知识助手

假设 App 提供个人知识助手，能搜索用户本地笔记并回答问题。合理架构是：

1. 本地抽取用户授权笔记。
2. 在端侧生成或保存 embedding。
3. 查询时先做本地召回。
4. 如果召回结果足够，端侧生成简短答案或展示引用。
5. 如果用户允许云端增强，再上传脱敏片段。
6. 云端生成更完整答案。
7. 本地保存结果和引用，不保存敏感原文到日志。

这个架构兼顾隐私、离线和能力，但也更复杂。工程师需要处理索引更新、删除、同步、授权和上云确认。

## 3.9 端云状态机

端云协同最好用状态机表达，而不是散落在页面逻辑中。一个通用状态机可以包含：

```text
idle
input_received
local_checking
local_processing
need_user_confirmation
cloud_processing
showing_result
fallback
failed
cancelled
```

状态机的价值是让异常路径变得可测试。例如：

- 本地模型加载失败，进入 fallback。
- 隐私过滤命中，进入 need_user_confirmation。
- 用户拒绝上传，回到 local result。
- 云端超时，展示本地候选。
- 用户取消，停止本地任务和云端请求。

如果没有状态机，端云协同会很快变成多处 `if/else`，难以审查和测试。

## 3.10 端云一致性

端侧和云端可能给出不同结果。团队要提前定义一致性规则。

| 场景 | 一致性策略 |
| --- | --- |
| 本地分类与云端分类不同 | 以高置信度且高优先级策略为准 |
| 本地召回和云端回答不一致 | 展示引用，云端回答必须基于引用 |
| 本地隐私过滤命中，云端可处理 | 仍以本地隐私策略为准 |
| 云端返回过慢 | 先展示本地结果，云端结果异步补充 |
| 云端失败 | 保留本地结果和可重试入口 |

端云协同不是让云端覆盖端侧，也不是让端侧拒绝云端，而是定义清楚谁在什么条件下有决策权。

## 3.11 队列设计细节

端云队列需要记录更多信息：

```text
task_id
task_type
created_at
expires_at
local_result
upload_payload
privacy_status
network_status
retry_count
user_cancelled
model_version
```

尤其要注意任务过期。用户今天选择上传一张截图，明天网络恢复时，这个任务可能已经不再需要。队列不能无条件重试，也不能在用户取消后继续上传。

建议规则：

- 用户取消后不再自动重试。
- 隐私确认只对当前任务有效。
- 任务过期后丢弃或重新确认。
- App 升级后校验队列 schema。
- 网络恢复后按风险排序重试。

## 3.12 服务端网关

移动端不应直接把所有请求打到模型服务。服务端网关应负责：

- 鉴权。
- 限流。
- 模型路由。
- Prompt 模板。
- 安全过滤。
- 审计日志。
- 成本控制。
- 超时和重试。

移动端负责本地输入治理和用户确认，服务端负责集中策略和模型调用。这样可以避免把密钥、模型路由和安全策略放在客户端。

## 3.13 架构评审清单

端云协同方案上线前建议检查：

| 项目 | 问题 |
| --- | --- |
| 路由 | 是否有明确端云路由表？ |
| 状态 | 是否有状态机覆盖等待、取消和失败？ |
| 隐私 | 上云前是否脱敏并确认？ |
| 队列 | 弱网任务是否可取消、过期和重试？ |
| 一致性 | 本地和云端结果冲突时谁优先？ |
| 网关 | 客户端是否避免保存模型密钥？ |
| 观测 | 是否能区分本地失败和云端失败？ |
| 回滚 | 是否能关闭云端增强或本地模型？ |

## 3.14 路由策略治理

端云路由不能只写在客户端代码里。随着功能增加，路由会受到模型版本、设备能力、用户授权、网络状态、隐私策略和远程配置影响。建议把路由策略治理成可配置、可观测、可回滚的模块。

一个路由决策至少应记录：

```text
request_id
task_type
input_type
privacy_level
device_class
network_state
local_capability
cloud_capability
route_decision
decision_reason
config_version
```

`decision_reason` 很重要。它可以说明为什么走本地、为什么要求用户确认、为什么拒绝上云、为什么进入云端兜底。没有这个字段，线上看到“云端调用下降”或“本地失败上升”时，很难判断是策略变化、模型失败还是用户授权变化。

路由策略也要有回归测试。给定一组输入和环境，路由结果应该稳定：

| 输入 | 环境 | 期望路由 |
| --- | --- | --- |
| 打开收藏 | 离线 | 本地命令 |
| 总结长文档 | Wi-Fi、已授权 | 用户确认后云端 |
| 解释含手机号截图 | 在线 | 本地脱敏预览 |
| 低端设备大图分析 | 在线 | 降低尺寸或云端 |

这类测试能防止路由规则在迭代中被无意改坏。

## 3.15 云端网关契约

服务端网关应提供稳定契约，而不是让客户端直接拼模型请求。推荐契约包含：

```json
{
  "request_id": "req_001",
  "task_type": "summarize_notes",
  "payload": {
    "chunks": [],
    "locale": "zh-CN"
  },
  "privacy": {
    "redacted": true,
    "user_confirmed": true
  },
  "client_context": {
    "app_version": "6.3.0",
    "model_version": "local-embed-v2",
    "route_version": "route-2026-06"
  }
}
```

网关返回也要结构化：

```json
{
  "status": "success",
  "answer": "...",
  "citations": [],
  "safety_flags": [],
  "fallback_suggestion": null
}
```

这样客户端可以统一处理成功、拒绝、超时、限流和安全拦截。不要把网关设计成“传 prompt，返回文本”。那会让客户端无法稳定展示引用、错误原因、降级建议和安全状态。

## 3.16 端云观测口径

端云协同需要同时观测端侧和云端。建议统一口径：

| 阶段 | 指标 |
| --- | --- |
| 本地检查 | 能力查询结果、权限状态、模型就绪状态 |
| 本地处理 | 加载耗时、推理耗时、失败码 |
| 隐私过滤 | 命中类型、脱敏比例、用户确认结果 |
| 云端请求 | 请求耗时、超时、限流、安全拦截 |
| 结果展示 | 引用数量、用户接受、编辑、取消 |
| 降级 | 降级原因、降级路径、是否成功 |

观测字段要使用同一个 `request_id` 串起来。否则端侧日志和云端日志无法关联，排障时只能猜。关联时仍要避免上传原始输入，可以使用任务 ID、版本、耗时和状态，而不是用户内容。

## 3.17 端云一致性的产品表达

端云结果不一致时，产品体验要提前设计。用户不应该看到两个互相冲突的答案，却不知道该相信谁。

常见表达方式：

- 本地先展示“设备上的初步结果”，云端完成后标记“增强结果”。
- 云端回答必须展示所依据的本地引用。
- 本地隐私策略拒绝上传时，明确说明“仅显示设备上的结果”。
- 云端结果覆盖本地结果时，保留查看本地引用的入口。
- 如果两者冲突，优先展示可验证证据。

端云协同的体验重点不是让用户理解架构，而是让用户知道结果来源和可信边界。特别是在笔记、文档、截图和会议场景中，引用和预览比一句“AI 认为”更有价值。

## 3.18 事故场景演练

端云协同上线前应演练事故：

1. 云端模型超时率突然升高。
2. 本地模型新版本加载失败。
3. 隐私过滤误漏检。
4. 路由配置把高敏任务错误上云。
5. 队列在网络恢复后重复上传。
6. 服务端限流导致大面积降级。
7. 某类低端设备推理 OOM。

每个事故都要回答：

| 问题 | 要求 |
| --- | --- |
| 如何发现 | 指标、告警或用户反馈 |
| 如何止血 | 关闭入口、回退模型、禁用路由 |
| 如何恢复 | 清理本地状态、重新下载、重建索引 |
| 如何复盘 | 补评测集、补路由测试、补发布门槛 |

端云协同复杂度高，事故不一定来自模型质量。很多事故来自配置、队列、权限和状态机。提前演练能减少真实事故中的决策时间。

## 3.19 端云协同最小实现

如果团队刚开始做端云协同，不必一次实现完整平台。可以先做最小实现：

1. 一个端云路由表。
2. 一个本地能力查询接口。
3. 一个隐私确认点。
4. 一个云端网关接口。
5. 一个超时和降级路径。
6. 一组端云观测字段。
7. 一个远程关闭开关。

这七项具备后，功能虽然简单，但已经具备可发布的基本闭环。后续可以逐步增加模型下载、队列、复杂灰度和多模型 bundle。反过来，如果没有路由表、确认点和关闭开关，即使模型效果很好，也不应该贸然上线。

## 本章小结

> **架构重点**：端云协同的重点是分工、路由、降级和数据边界。不要把端侧和云端当成竞争关系，而要把它们设计成互补的产品能力。移动端工程师需要把这些决策写成路由表、状态机和检查清单，让架构可审查、可测试、可演进。
