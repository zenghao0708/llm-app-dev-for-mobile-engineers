# 第 3 章 端云协同架构

## 本章导读

端侧和云端不是二选一。成熟移动端 AI 产品通常采用端云协同：本地处理低延迟、隐私敏感、离线可用和低成本任务，云端处理复杂推理、大模型生成、最新知识和集中治理任务。

端云协同的难点不在“端侧调用云端接口”，而在决策：什么时候端侧处理，什么时候上云，什么时候降级，什么时候让用户确认。这个决策如果没有工程化，就会变成散落在页面里的 if/else。

## 学习目标

- 掌握端云协同的常见架构。
- 设计本地优先、云端兜底和失败降级策略。
- 理解端侧缓存、队列和同步边界。
- 能为移动端 AI 功能设计端云路由表。

## 3.1 端云协同的基本分工

端侧适合：

- 快速分类和路由。
- 本地隐私过滤。
- 本地数据检索。
- 简单摘要和格式化。
- 离线命令。
- 输入预处理和输出后处理。

云端适合：

- 复杂长文本生成。
- 大规模知识问答。
- 多轮复杂推理。
- 高质量多模态理解。
- 集中模型更新。
- 统一审计和策略控制。

这个分工不是固定的。随着端侧模型能力提升，一些任务会从云端迁回本地；随着产品要求变化，一些任务又可能回到云端。架构应允许演进。

## 3.2 架构模式

| 模式 | 说明 | 适用场景 |
| --- | --- | --- |
| 本地优先 | 先端侧推理，失败或置信度低时上云 | 离线摘要、简单分类 |
| 云端优先 | 云端生成，本地做预处理或后处理 | 长文本生成、知识问答 |
| 分层决策 | 本地小模型做路由，云端大模型做复杂任务 | 智能助手、搜索入口 |
| 延迟容忍 | 本地先给粗结果，云端异步补全 | 图片理解、内容推荐 |
| 隐私保护 | 本地脱敏后再上云 | 截图分析、文档摘要 |

团队应该为每个 AI 功能选择明确模式，而不是让每个页面自己决定。

## 3.3 端云路由表

建议为功能建立路由表：

| 任务 | 默认路径 | 上云条件 | 本地降级 | 用户确认 |
| --- | --- | --- | --- | --- |
| 本地笔记搜索 | 端侧检索 | 无结果且用户允许联网 | 返回本地候选 | 可选 |
| 截图解释 | 本地脱敏 + 云端理解 | 截图不含敏感区域 | 只做本地 OCR | 必须 |
| 语音命令 | 端侧意图识别 | 置信度低 | 展示候选命令 | 可选 |
| 长文摘要 | 云端生成 | 网络可用且用户授权 | 本地生成粗摘要 | 必须 |

路由表的作用是把产品、隐私和工程决策显式化。没有路由表，端云协同很容易变成不可审计的隐式逻辑。

## 3.4 降级设计

端云协同必须有降级：

- 网络不可用时本地能力是否足够。
- 模型加载失败时是否回退传统逻辑。
- 云端超时时是否展示部分结果。
- 低端设备是否禁用部分能力。
- 用户拒绝上传时是否有本地替代。

降级不是异常处理的最后补丁，而是架构的一部分。用户看到的失败提示、重试按钮、离线结果和本地缓存，都应该在设计阶段确定。

## 3.5 数据边界

端云协同最敏感的是数据边界。移动端应区分：

- 永不上传：真实密钥、未授权相册、精确位置、敏感本地文件。
- 脱敏后上传：截图 OCR 结果、日志摘要、错误上下文。
- 用户确认后上传：录音、图片、文档、聊天历史。
- 可直接上传：公开查询、非个人化命令、低敏文本。

这个分类要进入代码和文档。仅靠口头约定，无法保证长期一致。

## 3.6 队列与同步

弱网环境下，端云协同需要队列：

- 待上传任务队列。
- 待下载模型队列。
- 待同步索引队列。
- 待重试生成队列。

队列要记录状态：pending、running、failed、cancelled、done。还要支持用户取消、任务过期和版本变化。否则，App 可能在网络恢复后上传用户已经不再需要的内容。

## 3.7 端云协同 Prompt 模板

```text
请为 [功能名] 设计端云协同架构。

输入：
- 用户数据类型：
- 延迟要求：
- 离线要求：
- 隐私等级：
- 目标平台：

输出：
- 默认端侧/云端路径。
- 上云条件。
- 本地降级策略。
- 用户确认点。
- 需要观测的指标。
- 需要禁止上传的数据。
```

这个模板适合架构评审，也适合在开发前让 AI coding agent 做方案分析。

## 3.8 本地优先案例：个人知识助手

假设 App 提供个人知识助手，能搜索用户本地笔记并回答问题。合理架构是：

1. 本地抽取用户授权笔记。
2. 在端侧生成或保存 embedding。
3. 查询时先做本地召回。
4. 如果召回结果足够，端侧生成简短答案或展示引用。
5. 如果用户允许云端增强，再上传脱敏片段。
6. 云端生成更完整答案。
7. 本地保存结果和引用，不保存敏感原文到日志。

这个架构兼顾隐私、离线和能力，但也更复杂。工程师需要处理索引更新、删除、同步、授权和上云确认。

## 3.9 端云状态机

端云协同最好用状态机表达，而不是散落在页面逻辑中。一个通用状态机可以包含：

```text
idle
input_received
local_checking
local_processing
need_user_confirmation
cloud_processing
showing_result
fallback
failed
cancelled
```

状态机的价值是让异常路径变得可测试。例如：

- 本地模型加载失败，进入 fallback。
- 隐私过滤命中，进入 need_user_confirmation。
- 用户拒绝上传，回到 local result。
- 云端超时，展示本地候选。
- 用户取消，停止本地任务和云端请求。

如果没有状态机，端云协同会很快变成多处 `if/else`，难以审查和测试。

## 3.10 端云一致性

端侧和云端可能给出不同结果。团队要提前定义一致性规则。

| 场景 | 一致性策略 |
| --- | --- |
| 本地分类与云端分类不同 | 以高置信度且高优先级策略为准 |
| 本地召回和云端回答不一致 | 展示引用，云端回答必须基于引用 |
| 本地隐私过滤命中，云端可处理 | 仍以本地隐私策略为准 |
| 云端返回过慢 | 先展示本地结果，云端结果异步补充 |
| 云端失败 | 保留本地结果和可重试入口 |

端云协同不是让云端覆盖端侧，也不是让端侧拒绝云端，而是定义清楚谁在什么条件下有决策权。

## 3.11 队列设计细节

端云队列需要记录更多信息：

```text
task_id
task_type
created_at
expires_at
local_result
upload_payload
privacy_status
network_status
retry_count
user_cancelled
model_version
```

尤其要注意任务过期。用户今天选择上传一张截图，明天网络恢复时，这个任务可能已经不再需要。队列不能无条件重试，也不能在用户取消后继续上传。

建议规则：

- 用户取消后不再自动重试。
- 隐私确认只对当前任务有效。
- 任务过期后丢弃或重新确认。
- App 升级后校验队列 schema。
- 网络恢复后按风险排序重试。

## 3.12 服务端网关

移动端不应直接把所有请求打到模型服务。服务端网关应负责：

- 鉴权。
- 限流。
- 模型路由。
- Prompt 模板。
- 安全过滤。
- 审计日志。
- 成本控制。
- 超时和重试。

移动端负责本地输入治理和用户确认，服务端负责集中策略和模型调用。这样可以避免把密钥、模型路由和安全策略放在客户端。

## 3.13 架构评审清单

端云协同方案上线前建议检查：

| 项目 | 问题 |
| --- | --- |
| 路由 | 是否有明确端云路由表？ |
| 状态 | 是否有状态机覆盖等待、取消和失败？ |
| 隐私 | 上云前是否脱敏并确认？ |
| 队列 | 弱网任务是否可取消、过期和重试？ |
| 一致性 | 本地和云端结果冲突时谁优先？ |
| 网关 | 客户端是否避免保存模型密钥？ |
| 观测 | 是否能区分本地失败和云端失败？ |
| 回滚 | 是否能关闭云端增强或本地模型？ |

## 本章小结

端云协同的重点是分工、路由、降级和数据边界。不要把端侧和云端当成竞争关系，而要把它们设计成互补的产品能力。移动端工程师需要把这些决策写成路由表、状态机和检查清单，让架构可审查、可测试、可演进。
