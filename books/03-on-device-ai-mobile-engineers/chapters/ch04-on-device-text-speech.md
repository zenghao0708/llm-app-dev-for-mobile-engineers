# 第 4 章 端侧文本与语音能力

## 本章导读

文本和语音是端侧 AI 最常见的入口。用户在移动设备上输入文字、说话、听播报、搜索本地内容、执行语音命令，这些行为都可以被 AI 能力增强。相比云端文本生成，端侧文本与语音能力更强调低延迟、隐私、离线和交互连续性。

移动端工程师在这一类场景中经常遇到一个误区：把文本或语音任务看成“调用模型得到结果”。真实产品中，难点往往不在模型 API，而在输入采集、权限、噪声、取消、前后台切换、缓存、错误提示和降级。一个语音命令模型在安静会议室里可用，不代表它在地铁、车载蓝牙、耳机、后台恢复和低电量模式下也可靠。

## 学习目标

- 理解端侧文本处理和语音处理的典型任务。
- 知道延迟、隐私、噪声和设备差异如何影响实现。
- 能设计本地语音命令或文本分类的验证清单。
- 能把文本和语音能力拆成可测试、可降级的移动端工程链路。

## 4.1 文本任务的端侧边界

端侧文本任务适合从小能力开始，而不是一上来做完整聊天助手。常见任务包括：

- 本地文本分类：判断消息、笔记、工单或评论属于哪个类别。
- 敏感信息检测：识别手机号、邮箱、地址、证件号、密钥片段等。
- 本地意图识别：把“打开设置”“搜索昨天的截图”映射到 App 操作。
- 短文本摘要：对本地笔记、剪贴板或通知做简短摘要。
- 本地改写：对输入内容做格式化、纠错或风格调整。
- 离线问答入口：在本地可用范围内回答简单问题。

这些任务有一个共同特点：输入较短、边界较清楚、结果可以验证。端侧文本能力不适合一开始承担开放式复杂推理。如果任务需要最新知识、大量上下文、高质量长文本生成或严格事实校验，通常要端云协同。

## 4.2 文本处理链路

一个可发布的端侧文本链路通常包括：

1. 输入收集：来自输入框、剪贴板、通知、本地文件或语音转写结果。
2. 预处理：去除空白、截断超长输入、识别语言、过滤不可处理内容。
3. 隐私判断：判断是否含敏感信息，决定是否只在本地处理。
4. 模型推理：执行分类、摘要、意图识别或改写。
5. 后处理：解析结构化结果、映射到业务枚举、检查置信度。
6. 展示或执行：展示建议、触发操作、进入二次确认。
7. 记录指标：延迟、失败类型、用户接受率、降级次数。

移动端工程师需要避免让模型输出直接驱动高风险操作。例如模型识别“删除全部草稿”后，不应直接删除数据，而应进入确认流程。文本任务看似轻量，但一旦连接到业务动作，就必须有权限和确认边界。

## 4.3 语音任务的端侧价值

语音任务的端侧价值主要来自三点：

第一，离线可用。很多语音命令不需要云端大模型，例如“开始录音”“暂停播放”“打开扫一扫”“切换深色模式”。这些命令适合在端侧完成。

第二，低延迟。语音交互最怕等待。如果每个短命令都要上传云端，用户会明显感到不连贯。本地识别可以先给出快速反馈。

第三，隐私。录音、会议、家庭场景和车内对话都可能敏感。本地处理可以减少原始音频上传。

但语音任务的工程代价也更高。它涉及麦克风权限、音频会话、前后台生命周期、耳机和蓝牙、噪声环境、回声、方言、多语言和设备性能。模型效果只是其中一部分。

## 4.4 语音命令链路

一个本地语音命令功能可以拆成：

```text
麦克风输入 -> VAD/端点检测 -> 转写或命令识别 -> 意图解析 -> 置信度判断 -> 执行/确认/降级
```

其中 VAD 用于判断用户是否开始和结束说话。没有端点检测，App 可能一直录音、错误截断或让用户等待。转写可以是完整语音转文本，也可以是面向少量命令的轻量识别。意图解析要把自然语言映射到业务动作，例如：

| 用户说法 | 意图 | 是否需要确认 |
| --- | --- | --- |
| “打开收藏” | open_favorites | 否 |
| “搜索昨天的照片” | search_photos | 可选 |
| “删除这条记录” | delete_item | 是 |
| “发给张三” | send_to_contact | 是 |

置信度低时，不应硬执行。更好的做法是展示候选项：“你是想搜索照片，还是打开相册？”

## 4.5 移动端生命周期问题

文本和语音能力都要处理移动端生命周期：

- 用户在推理过程中切到后台。
- 系统回收资源。
- 麦克风权限被拒绝或撤销。
- 蓝牙耳机切换输入设备。
- 来电中断音频会话。
- 低电量模式限制后台任务。
- 屏幕锁定后任务是否继续。
- 用户取消任务或关闭页面。

这些场景如果不设计，线上会出现“偶现”问题。语音任务尤其要在状态机中表示 recording、processing、cancelled、interrupted、failed、done 等状态，而不是用一个布尔值表示是否正在识别。

## 4.6 本地文本分类案例

假设一个移动端客服 App 希望在本地把用户输入分为“退款”“物流”“账号”“投诉”“其他”。端侧方案可以这样设计：

1. 输入框内容变化时不立即推理，而是等待用户停顿。
2. 对超长输入只取前若干关键片段。
3. 本地模型输出类别和置信度。
4. 置信度高时展示快捷入口，例如“查看退款进度”。
5. 置信度低时不展示，避免干扰。
6. 用户点击建议后再记录接受率。

这个功能不需要一开始上云。它的价值是低延迟和隐私，验证指标也很清楚：分类准确率、建议点击率、误触率、平均延迟、低端设备表现。

## 4.7 本地语音命令案例

假设一个笔记 App 支持离线语音命令：

- “新建笔记”
- “开始录音”
- “暂停”
- “继续”
- “保存”
- “搜索会议”

实现时不要只关注语音识别模型，还要定义命令状态机。例如“保存”只有在编辑或录音状态下有效；“继续”只有暂停后有效；“搜索会议”需要进入搜索页而不是修改当前笔记。

测试时至少覆盖：

| 场景 | 验证点 |
| --- | --- |
| 安静环境 | 基础识别准确 |
| 噪声环境 | 不误触发高风险命令 |
| 蓝牙耳机 | 输入设备切换正常 |
| 权限拒绝 | 有明确替代入口 |
| 后台切回 | 不重复执行命令 |
| 低置信度 | 展示候选或要求重说 |
| 多语言系统 | 文案和命令表一致 |

语音命令的目标不是识别所有话，而是稳定识别少量高价值命令。

## 4.8 验证清单

端侧文本与语音能力上线前建议检查：

| 项目 | 检查问题 |
| --- | --- |
| 延迟 | 首次加载和单次推理是否在体验预算内？ |
| 隐私 | 原始文本或音频是否留在本地？是否记录日志？ |
| 权限 | 麦克风权限拒绝后是否有可用流程？ |
| 取消 | 用户离开页面后任务是否停止或安全完成？ |
| 噪声 | 语音任务是否覆盖噪声、耳机、蓝牙？ |
| 低端设备 | 内存和温度是否可接受？ |
| 降级 | 模型加载失败时是否回到传统入口？ |
| 观测 | 是否记录失败类型、耗时和用户接受率？ |

## 4.9 文本任务测试样例

文本任务测试不要只覆盖正常输入，还要覆盖边界输入。以本地意图识别为例，可以准备：

| 类型 | 样例 | 期望 |
| --- | --- | --- |
| 标准命令 | 打开收藏 | open_favorites |
| 同义表达 | 帮我看收藏夹 | open_favorites |
| 模糊表达 | 找一下昨天那个 | low_confidence |
| 高风险动作 | 删除全部笔记 | need_confirmation |
| 超长输入 | 多段无关文本后带命令 | 截断后仍安全 |
| 多语言 | open favorites | 按语言策略处理 |
| 噪声字符 | 打开！！收藏？？ | 正常归一化 |

测试样例应保存到仓库，而不是只写在文档里。每次模型、分词器、prompt 模板或规则变化，都要跑回归。

## 4.10 语音状态机

语音能力建议使用显式状态机：

```text
idle
permission_checking
recording
detecting_endpoint
recognizing
need_confirmation
executing
finished
interrupted
failed
cancelled
```

状态机要处理系统事件：

- 来电中断。
- 耳机断开。
- App 进入后台。
- 权限被撤销。
- 用户点击取消。
- 录音时间过长。
- 低置信度。

如果状态机不清楚，最容易出现重复执行命令、录音没有释放、页面退出后仍回调 UI、用户取消后继续处理等问题。

## 4.11 观测指标

文本和语音能力应记录聚合指标：

| 指标 | 用途 |
| --- | --- |
| model_load_success | 判断模型是否稳定加载 |
| inference_latency_ms | 判断交互延迟 |
| low_confidence_rate | 判断模型是否常常不确定 |
| user_accept_rate | 判断建议是否有价值 |
| cancel_rate | 判断等待是否过长 |
| permission_denied_rate | 判断入口和权限文案是否合理 |
| fallback_rate | 判断端侧能力是否可靠 |

不要记录原始文本或音频。对于语音命令，可以记录命令枚举和置信度区间，但不要记录完整录音。

## 4.12 迭代路径

端侧文本与语音能力可以按阶段推进：

1. 规则或小模型识别少量命令。
2. 加入置信度和拒答。
3. 加入状态机和完整错误提示。
4. 加入本地评测集。
5. 加入端云协同，例如云端处理复杂表达。
6. 加入个性化，例如用户常用命令排序。

这条路径比直接做“本地语音助手”更稳。移动端 AI 产品的成熟，往往来自连续的小能力，而不是一次性的大功能。

## 本章小结

端侧文本与语音能力适合从小任务做起。文本任务要关注输入边界、结构化输出和业务动作确认；语音任务要关注权限、噪声、生命周期和状态机。先保证延迟、隐私、失败兜底和验证指标，再扩大到复杂生成或端云协同。
