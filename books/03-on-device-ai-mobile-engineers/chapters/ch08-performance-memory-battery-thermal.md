# 第 8 章 性能、内存、电量与热管理

## 本章导读

端侧模型进入 App 后，性能问题会直接影响用户体验。延迟、内存、电量和发热必须在开发阶段被量化，而不是发版后再观察。对用户来说，“本地 AI 更隐私”不能成为卡顿、耗电和发热的理由。一个端侧模型如果让页面打开变慢、手机明显发热、后台被系统杀掉，即使准确率很高，也不是合格的移动端能力。

本章讨论如何建立端侧 AI 的性能预算、profiling 方法和线上观测。重点是把“感觉还行”变成可比较、可回归、可发布的指标。

## 学习目标

- 建立端侧 AI 性能指标。
- 设计本地 profiling 和线上观测。
- 理解低端设备和长时间运行风险。
- 能为端侧 AI 功能制定性能预算和降级策略。

## 8.1 性能预算

性能预算是上线前的约束。没有预算，优化就没有目标。端侧 AI 至少要定义：

| 指标 | 说明 |
| --- | --- |
| 首次加载时间 | 模型首次可用所需时间 |
| 单次推理延迟 | 用户等待时间 |
| 峰值内存 | 是否触发系统回收或崩溃 |
| 电量消耗 | 长任务和后台任务影响 |
| 温度 | 是否触发降频 |
| 包体 | 模型是否显著增加安装成本 |

不同任务的预算不同。输入法联想需要非常低延迟；文档扫描可以接受几秒处理；相册后台整理可以延迟完成，但不能持续发热；本地知识索引可以在充电和 Wi-Fi 条件下执行。

建议用表格记录：

| 任务 | 首次加载 | 单次延迟 | 峰值内存 | 可后台 | 可取消 |
| --- | --- | --- | --- | --- | --- |
| 语音命令 | 短 | 短 | 低 | 否 | 是 |
| 文档扫描 | 中 | 中 | 中 | 否 | 是 |
| 相册索引 | 长 | 长 | 可控 | 是 | 是 |
| 本地摘要 | 中 | 中 | 中 | 否 | 是 |

预算不必一开始完美，但必须存在。

## 8.2 冷启动与懒加载

模型加载常常比单次推理更影响体验。移动端工程师要决定：

- 模型是否随 App 启动加载。
- 是否在功能入口懒加载。
- 是否在用户可能使用前预热。
- 是否在内存紧张时释放。
- 是否缓存 tokenizer、特征提取器或 runtime。

不要把所有模型放到 App 启动路径。更好的策略是按功能懒加载，并在 UI 上明确 loading 状态。如果模型加载失败，功能应该降级，而不是让页面卡住。

对高频小模型，可以考虑预热；对低频大模型，应避免常驻内存。缓存策略要和系统内存压力结合，不能只从模型角度看。

## 8.3 内存峰值

端侧 AI 的内存不仅来自模型文件，还包括：

- runtime 初始化。
- 权重加载。
- tokenizer 或预处理资源。
- 输入张量。
- 中间激活。
- KV cache。
- 图片解码后的像素缓冲。
- 批处理临时数据。

图片和多模态任务尤其容易出现内存峰值。用户选择一张大图时，如果直接按原尺寸解码，再送入模型，峰值可能远高于模型本身。正确做法是按目标尺寸解码、分块处理或提前压缩。

移动端测试要覆盖低内存设备。高端机上没问题，不代表目标用户设备没问题。

## 8.4 电量与热管理

电量和发热是端侧 AI 的长期体验指标。短时间推理可能看不出问题，但连续语音、相册索引、长文本摘要、后台处理都可能触发降频或系统限制。

建议遵守：

- 长任务必须可取消。
- 后台任务要分批执行。
- 低电量模式下减少 AI 任务。
- 设备发热时暂停非关键任务。
- 充电和 Wi-Fi 条件下执行重索引。
- 避免在页面滚动或动画时执行重推理。

热管理不是等系统降频后再处理，而是产品策略的一部分。例如相册整理可以显示“将在设备空闲时继续处理”，而不是强制立即跑完。

## 8.5 Profiling 方法

本地 profiling 至少要记录：

```text
device_model
os_version
app_version
model_version
runtime
input_size
load_time_ms
inference_time_ms
peak_memory_mb
task_cancelled
error_type
```

测试时要覆盖：

- 首次安装后第一次使用。
- App 冷启动后使用。
- 连续多次使用。
- 前后台切换。
- 低端设备。
- 大输入。
- 权限拒绝和取消。

不要只记录平均值。P95、P99 和失败率更能反映真实体验。端侧 AI 的坏体验通常来自少数慢设备和大输入。

## 8.6 线上观测

端侧 AI 的线上观测要谨慎，不能上传敏感原文、图片或音频。可以上传聚合指标：

- 模型加载成功率。
- 推理成功率。
- 超时次数。
- 用户取消次数。
- 降级次数。
- 设备型号分布。
- 模型版本分布。
- 平均和分位延迟。

错误日志要避免包含原始输入。可以记录错误类型和输入大小，而不是输入内容。例如记录 `input_tokens_bucket=512-1024`，不要记录用户文本。

## 8.7 降级策略

性能降级应提前设计：

| 条件 | 降级 |
| --- | --- |
| 模型加载超时 | 使用传统规则或隐藏 AI 建议 |
| 内存不足 | 降低输入尺寸、减少 batch、释放缓存 |
| 设备发热 | 暂停后台索引或降低频率 |
| 低电量 | 延迟非必要任务 |
| 低端设备 | 默认关闭重模型功能 |
| 用户取消 | 立即停止任务并释放资源 |

降级不是失败，而是移动端产品质量的一部分。用户更能接受“稍后处理”或“当前设备暂不支持”，而不是卡死和崩溃。

## 8.8 案例：本地相册索引

相册索引是性能挑战集中的场景。它可能处理成千上万张图片，涉及解码、缩放、embedding、索引写入和增量更新。

合理策略：

1. 用户授权后不立即全量处理。
2. 先处理最近少量图片，让功能快速可用。
3. 后台分批索引，每批之间让出资源。
4. 只在设备空闲、充电或 Wi-Fi 条件下执行重任务。
5. 记录已处理资源和模型版本。
6. 用户关闭功能时停止并清理索引。

指标包括：每张图片处理时间、每批耗时、峰值内存、取消率、索引完成率、设备发热反馈。没有这些指标，相册索引很容易变成“测试机可用，真实用户不可用”。

## 8.9 性能评审清单

上线前建议检查：

| 项目 | 问题 |
| --- | --- |
| 预算 | 是否定义加载、推理、内存和电量目标？ |
| 设备 | 是否覆盖低端设备和主力机型？ |
| 输入 | 是否测试最大图片、最长文本、最长音频？ |
| 生命周期 | 前后台、取消、权限变化是否稳定？ |
| 观测 | 是否能区分加载失败、推理失败和超时？ |
| 隐私 | 性能日志是否不包含原始输入？ |
| 降级 | 超预算时是否有明确用户体验？ |
| 回归 | 模型更新后是否重复跑性能基准？ |

## 8.10 性能基线的建立

性能优化要先有基线。基线不是一次测试截图，而是一组可以重复执行、可以比较版本差异的数据。端侧 AI 功能至少应该维护三类基线：

| 基线 | 目的 | 触发时机 |
| --- | --- | --- |
| 开发基线 | 发现明显性能退化 | 每次模型、runtime 或预处理变化 |
| 发布基线 | 判断是否能进入灰度 | 发版候选构建 |
| 线上基线 | 观察真实设备分布 | 灰度和全量发布后 |

开发基线可以在固定设备上执行，重点是比较变化趋势；发布基线要覆盖目标机型、系统版本和大输入；线上基线要看 P95、P99、失败率和降级率。三类基线不能互相替代。

建议把每次基线结果记录为：

```text
feature_name
app_version
model_version
runtime_version
device_group
input_profile
load_time_p50_ms
load_time_p95_ms
inference_p50_ms
inference_p95_ms
peak_memory_mb
thermal_state
failure_rate
```

如果团队只记录“本地测试耗时 800ms”，后续几乎无法判断新版模型到底是变快、变慢，还是只在某类输入上变差。

## 8.11 任务调度策略

端侧 AI 任务不能只看模型本身，还要和页面生命周期、用户交互、系统状态协调。常见调度策略包括：

| 场景 | 策略 |
| --- | --- |
| 用户正在输入 | 避免执行重推理，优先低延迟小模型 |
| 页面正在滚动 | 延迟视觉处理，避免掉帧 |
| App 进入后台 | 只保留可中断、可恢复任务 |
| 设备低电量 | 暂停非必要索引和重计算 |
| 设备发热 | 降低任务频率或停止后台批处理 |
| 网络可用 | 可选择云端兜底或下载模型 |

移动端工程师要避免把 AI 任务塞进主线程或关键渲染路径。即使模型推理本身在后台线程，如果预处理、图片解码、结果合并或数据库写入发生在主线程，用户仍然会感觉卡顿。

长任务还要有任务队列和取消机制。用户离开页面后，除非任务明确可以后台继续，否则应该停止或降级。端侧 AI 功能的资源使用要服从产品生命周期，而不是让模型任务反过来控制 App。

## 8.12 输入规模控制

很多性能问题不是模型太慢，而是输入没有控制。移动端 AI 输入通常来自用户真实数据，大小分布很不稳定：

- 一张图片可能是几百 KB，也可能是几十 MB。
- 一段语音可能是 5 秒，也可能是 30 分钟。
- 一篇笔记可能只有一句话，也可能是长文档。
- 一个相册可能只有几十张图片，也可能有几万张。

因此，每个功能都要定义输入上限：

| 输入 | 控制方式 |
| --- | --- |
| 图片 | 目标尺寸解码、最大边限制、分块处理 |
| 文本 | token 上限、分段、摘要缓存 |
| 音频 | 最大时长、分片识别、静音裁剪 |
| 相册 | 分批索引、增量更新、优先最近数据 |
| 本地文件 | 文件类型白名单、大小限制、用户确认 |

输入规模控制应该发生在进入模型之前。如果等模型报错或 OOM 后再处理，用户已经经历了卡顿、等待或崩溃风险。

## 8.13 缓存与复用

缓存能显著改善端侧 AI 体验，但缓存也会带来内存和一致性问题。常见可缓存对象包括：

- runtime 实例。
- tokenizer 和词表。
- 图片预处理结果。
- embedding 向量。
- 检索索引。
- 最近一次推理结果。

缓存策略要回答三个问题：什么时候创建、什么时候复用、什么时候释放。高频轻量能力可以复用 runtime；低频大模型不应长期常驻。embedding 和索引可以持久化，但必须绑定模型版本；模型升级后旧缓存是否可用，需要由 schema 或 index version 判断。

缓存还要避免泄露隐私。用户退出账号、关闭功能或撤销权限后，相关本地缓存要清理或失效。性能优化不能绕过数据治理。

## 8.14 性能问题排查路径

当端侧 AI 功能变慢时，建议按下面顺序排查：

1. 确认是加载慢、预处理慢、推理慢，还是后处理慢。
2. 对比模型版本、runtime 版本和输入规模。
3. 查看是否发生主线程阻塞。
4. 检查图片、音频或文本是否超过预期上限。
5. 检查内存压力是否触发系统回收。
6. 检查设备是否进入热降频。
7. 对比低端设备和高端设备差异。
8. 查看是否频繁重复加载模型或重复计算缓存。

排查时不要急着换模型。很多问题来自输入解码、任务调度、缓存失效或数据库写入。只有把链路拆开记录，才能知道真正瓶颈在哪里。

性能问题的修复也要回到基线。一次优化如果只改善平均延迟，却让 P99 或内存峰值变差，仍可能不适合发布。

## 8.15 内存压力处理

端侧 AI 很容易触发内存压力，尤其是多模态、长文本和批处理任务。内存治理要从设计阶段开始：

| 来源 | 风险 | 处理 |
| --- | --- | --- |
| 模型权重 | 常驻内存过大 | 低频模型懒加载，内存压力时释放 |
| 图片解码 | 原图尺寸过大 | 按目标尺寸解码，分块处理 |
| KV cache | 长文本生成增长快 | 限制上下文长度，及时释放 |
| 批处理缓存 | 多任务叠加 | 分批执行，阶段性落盘 |
| 索引加载 | 启动时占用高 | 延迟加载、分页加载、按需查询 |

App 收到系统内存压力信号时，应暂停非关键任务，释放低频模型、图片缓冲和临时缓存。不要等到 OOM 后才把问题归因于“设备太低端”。移动端 AI 的目标是在目标设备范围内稳定运行，而不是只在开发机上跑通。

## 8.16 热状态与电量策略

发热和电量是端侧 AI 的长期体验。建议把系统热状态、电量状态和充电状态纳入调度：

| 状态 | 策略 |
| --- | --- |
| 正常 | 允许前台交互任务和小批量后台任务 |
| 轻微发热 | 降低后台任务频率 |
| 明显发热 | 暂停索引、批处理和重模型推理 |
| 低电量 | 只保留用户主动触发的关键任务 |
| 充电 + Wi-Fi | 可执行索引重建和模型下载 |

这些策略要能被用户理解。比如相册索引暂停时，可以提示“设备温度较高，将稍后继续整理”。不要让用户以为功能卡住。对后台任务来说，清楚的暂停和恢复，比强行跑完更符合移动端体验。

## 8.17 性能预算写入发布门槛

性能预算应进入发版门槛，而不是测试报告附件。建议每个端侧 AI 功能定义：

```text
load_time_p95 <= ...
inference_time_p95 <= ...
peak_memory_mb <= ...
background_batch_size <= ...
cancel_release_time_ms <= ...
low_end_device_supported = true/false
```

如果预算超标，要给出明确处理：优化、缩小输入、关闭低端设备、改为云端、延后发布。不要把“稍后优化”作为默认结论。端侧 AI 性能问题上线后很难靠服务端修复，必须在客户端发版前建立硬门槛。

预算还要按功能分层。输入联想的预算应非常严格；文档扫描可以接受较长处理时间；相册索引可以后台执行，但必须可暂停和可取消。统一延迟标准并不现实，关键是每个任务有自己的体验目标。

## 本章小结

端侧 AI 的性能优化必须从指标开始。没有预算、profiling、线上观测和降级策略，优化只是猜测。移动端工程师要把加载时间、推理延迟、峰值内存、电量和温度纳入发布门槛，让端侧 AI 成为稳定能力，而不是偶尔可用的实验功能。
