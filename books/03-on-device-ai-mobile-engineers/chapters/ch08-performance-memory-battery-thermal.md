# 第 8 章 性能、内存、电量与热管理

## 本章导读

端侧模型进入 App 后，性能问题会直接影响用户体验。延迟、内存、电量和发热必须在开发阶段被量化，而不是发版后再观察。对用户来说，“本地 AI 更隐私”不能成为卡顿、耗电和发热的理由。一个端侧模型如果让页面打开变慢、手机明显发热、后台被系统杀掉，即使准确率很高，也不是合格的移动端能力。

本章讨论如何建立端侧 AI 的性能预算、profiling 方法和线上观测。重点是把“感觉还行”变成可比较、可回归、可发布的指标。

## 学习目标

- 建立端侧 AI 性能指标。
- 设计本地 profiling 和线上观测。
- 理解低端设备和长时间运行风险。
- 能为端侧 AI 功能制定性能预算和降级策略。

## 8.1 性能预算

性能预算是上线前的约束。没有预算，优化就没有目标。端侧 AI 至少要定义：

| 指标 | 说明 |
| --- | --- |
| 首次加载时间 | 模型首次可用所需时间 |
| 单次推理延迟 | 用户等待时间 |
| 峰值内存 | 是否触发系统回收或崩溃 |
| 电量消耗 | 长任务和后台任务影响 |
| 温度 | 是否触发降频 |
| 包体 | 模型是否显著增加安装成本 |

不同任务的预算不同。输入法联想需要非常低延迟；文档扫描可以接受几秒处理；相册后台整理可以延迟完成，但不能持续发热；本地知识索引可以在充电和 Wi-Fi 条件下执行。

建议用表格记录：

| 任务 | 首次加载 | 单次延迟 | 峰值内存 | 可后台 | 可取消 |
| --- | --- | --- | --- | --- | --- |
| 语音命令 | 短 | 短 | 低 | 否 | 是 |
| 文档扫描 | 中 | 中 | 中 | 否 | 是 |
| 相册索引 | 长 | 长 | 可控 | 是 | 是 |
| 本地摘要 | 中 | 中 | 中 | 否 | 是 |

预算不必一开始完美，但必须存在。

## 8.2 冷启动与懒加载

模型加载常常比单次推理更影响体验。移动端工程师要决定：

- 模型是否随 App 启动加载。
- 是否在功能入口懒加载。
- 是否在用户可能使用前预热。
- 是否在内存紧张时释放。
- 是否缓存 tokenizer、特征提取器或 runtime。

不要把所有模型放到 App 启动路径。更好的策略是按功能懒加载，并在 UI 上明确 loading 状态。如果模型加载失败，功能应该降级，而不是让页面卡住。

对高频小模型，可以考虑预热；对低频大模型，应避免常驻内存。缓存策略要和系统内存压力结合，不能只从模型角度看。

## 8.3 内存峰值

端侧 AI 的内存不仅来自模型文件，还包括：

- runtime 初始化。
- 权重加载。
- tokenizer 或预处理资源。
- 输入张量。
- 中间激活。
- KV cache。
- 图片解码后的像素缓冲。
- 批处理临时数据。

图片和多模态任务尤其容易出现内存峰值。用户选择一张大图时，如果直接按原尺寸解码，再送入模型，峰值可能远高于模型本身。正确做法是按目标尺寸解码、分块处理或提前压缩。

移动端测试要覆盖低内存设备。高端机上没问题，不代表目标用户设备没问题。

## 8.4 电量与热管理

电量和发热是端侧 AI 的长期体验指标。短时间推理可能看不出问题，但连续语音、相册索引、长文本摘要、后台处理都可能触发降频或系统限制。

建议遵守：

- 长任务必须可取消。
- 后台任务要分批执行。
- 低电量模式下减少 AI 任务。
- 设备发热时暂停非关键任务。
- 充电和 Wi-Fi 条件下执行重索引。
- 避免在页面滚动或动画时执行重推理。

热管理不是等系统降频后再处理，而是产品策略的一部分。例如相册整理可以显示“将在设备空闲时继续处理”，而不是强制立即跑完。

## 8.5 Profiling 方法

本地 profiling 至少要记录：

```text
device_model
os_version
app_version
model_version
runtime
input_size
load_time_ms
inference_time_ms
peak_memory_mb
task_cancelled
error_type
```

测试时要覆盖：

- 首次安装后第一次使用。
- App 冷启动后使用。
- 连续多次使用。
- 前后台切换。
- 低端设备。
- 大输入。
- 权限拒绝和取消。

不要只记录平均值。P95、P99 和失败率更能反映真实体验。端侧 AI 的坏体验通常来自少数慢设备和大输入。

## 8.6 线上观测

端侧 AI 的线上观测要谨慎，不能上传敏感原文、图片或音频。可以上传聚合指标：

- 模型加载成功率。
- 推理成功率。
- 超时次数。
- 用户取消次数。
- 降级次数。
- 设备型号分布。
- 模型版本分布。
- 平均和分位延迟。

错误日志要避免包含原始输入。可以记录错误类型和输入大小，而不是输入内容。例如记录 `input_tokens_bucket=512-1024`，不要记录用户文本。

## 8.7 降级策略

性能降级应提前设计：

| 条件 | 降级 |
| --- | --- |
| 模型加载超时 | 使用传统规则或隐藏 AI 建议 |
| 内存不足 | 降低输入尺寸、减少 batch、释放缓存 |
| 设备发热 | 暂停后台索引或降低频率 |
| 低电量 | 延迟非必要任务 |
| 低端设备 | 默认关闭重模型功能 |
| 用户取消 | 立即停止任务并释放资源 |

降级不是失败，而是移动端产品质量的一部分。用户更能接受“稍后处理”或“当前设备暂不支持”，而不是卡死和崩溃。

## 8.8 案例：本地相册索引

相册索引是性能挑战集中的场景。它可能处理成千上万张图片，涉及解码、缩放、embedding、索引写入和增量更新。

合理策略：

1. 用户授权后不立即全量处理。
2. 先处理最近少量图片，让功能快速可用。
3. 后台分批索引，每批之间让出资源。
4. 只在设备空闲、充电或 Wi-Fi 条件下执行重任务。
5. 记录已处理资源和模型版本。
6. 用户关闭功能时停止并清理索引。

指标包括：每张图片处理时间、每批耗时、峰值内存、取消率、索引完成率、设备发热反馈。没有这些指标，相册索引很容易变成“测试机可用，真实用户不可用”。

## 8.9 性能评审清单

上线前建议检查：

| 项目 | 问题 |
| --- | --- |
| 预算 | 是否定义加载、推理、内存和电量目标？ |
| 设备 | 是否覆盖低端设备和主力机型？ |
| 输入 | 是否测试最大图片、最长文本、最长音频？ |
| 生命周期 | 前后台、取消、权限变化是否稳定？ |
| 观测 | 是否能区分加载失败、推理失败和超时？ |
| 隐私 | 性能日志是否不包含原始输入？ |
| 降级 | 超预算时是否有明确用户体验？ |
| 回归 | 模型更新后是否重复跑性能基准？ |

## 本章小结

端侧 AI 的性能优化必须从指标开始。没有预算、profiling、线上观测和降级策略，优化只是猜测。移动端工程师要把加载时间、推理延迟、峰值内存、电量和温度纳入发布门槛，让端侧 AI 成为稳定能力，而不是偶尔可用的实验功能。
