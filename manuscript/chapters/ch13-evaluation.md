# 第 13 章 大模型应用的评测方法

## 本章导读

大模型应用不能只靠“看起来回答得不错”判断质量。模型输出有随机性，知识库会更新，Prompt 会迭代，模型版本会变化，移动端状态也会影响体验。没有评测，团队很难知道一次修改是变好、变差还是只是在少数样本上看起来更顺。

配套脚本：`examples/mobile-knowledge-assistant/scripts/rag_eval.py`、`scripts/answer_eval.py`

## 学习目标

- 理解大模型评测和单元测试的区别。
- 从最小评测集开始覆盖高频问题和高风险问题。
- 区分检索评测、答案评测和移动端体验指标。
- 运行 RAG 检索评测和答案质量评测脚本。
- 知道什么时候需要人工复核，什么时候可以自动化门禁。

## 13.1 为什么必须评测

大模型应用的质量会被多种因素影响：Prompt、模型、检索、知识库、工具结果、采样参数、上下文长度和移动端状态。一次小改动可能让常见问题变好，也可能让边界问题变差。

评测要回答 4 个问题：

- 正确资料有没有被检索出来？
- 最终答案是否覆盖关键要点？
- 引用来源是否可信、可追溯？
- 移动端体验是否满足首 Token、错误恢复和取消要求？

单元测试验证确定性逻辑，例如 JSON 校验、路径限制、错误码。评测验证模型链路表现，例如召回率、答案要点、引用准确和风险表达。两者都需要，不能互相替代。

## 13.2 从最小评测集开始

早期不需要一次准备几千条样本。可以先从 20-50 条高价值问题开始：

| 类型 | 示例 |
| --- | --- |
| 高频问题 | 移动端为什么不能直接保存模型 API Key？ |
| 高风险问题 | 是否可以把用户手机号发给模型分析？ |
| 易混淆问题 | 弱网超时和服务端 500 如何区分？ |
| 无资料问题 | 知识库里没有的内部流程 |
| 权限问题 | 当前用户无权访问的资料 |
| 回归问题 | 曾经线上出错或用户反馈过的问题 |

每条样本至少包含问题、期望资料、关键要点、禁止表达和人工备注。评测集要随产品迭代持续更新。

## 13.3 检索评测

RAG 质量首先取决于检索。`scripts/rag_eval.py` 读取 `data/eval/rag_eval_cases.json`，检查期望文档章节是否出现在 Top-K 结果中。

运行：

```bash
cd examples/mobile-knowledge-assistant
python3 scripts/rag_eval.py --top-k 3
```

检索评测常用指标：

| 指标 | 含义 |
| --- | --- |
| Top-K 命中率 | 期望资料是否出现在前 K 条 |
| MRR | 正确结果排得越靠前分数越高 |
| 无结果率 | 召回为空的比例 |
| 低相关率 | 分数低于阈值的比例 |

检索评测不调用模型。它只能证明“资料有没有给对”，不能证明最终答案一定正确。修改切分、索引、Top-K、元数据过滤和重排后，应先跑它。

## 13.4 答案评测

`scripts/answer_eval.py` 检查最终回答是否覆盖要点、引用是否命中、是否出现禁词或高风险表达。

```bash
python3 scripts/answer_eval.py --min-score 0.8
```

答案评测可以从规则开始：

| 检查项 | 示例 |
| --- | --- |
| 要点覆盖 | 是否提到“移动端包可被反编译” |
| 引用命中 | 是否引用 `mobile_ai_api.md` |
| 禁止表达 | 不得说“可以把 API Key 放进 App” |
| 风险提示 | 涉及隐私或权限时是否提示人工确认 |
| 结构格式 | 是否返回期望字段 |

默认失败返回非零退出码，适合 CI 门禁。早期只想观察报告时，可使用 `--report-only`。

## 13.5 自动评分规则

自动评分要从明确规则开始，不要一开始就把所有判断交给模型评委。可自动化的内容包括关键词、引用 ID、JSON Schema、字段枚举、长度、禁词、错误码和工具调用状态。

适合人工复核的内容包括：答案是否真正解决问题、语气是否合适、风险提示是否足够、引用是否支持结论、业务建议是否符合团队流程。

如果使用模型评委，也应注意：

- 评委 Prompt 要版本化。
- 评委输入要脱敏。
- 高风险样本不能只依赖模型评分。
- 要抽样人工复核评委结果。
- 评委模型升级后要重新校准。

## 13.6 移动端体验指标

大模型答案正确，不代表移动端体验合格。还要评测：

| 指标 | 含义 |
| --- | --- |
| 首 Token 延迟 | 用户点击发送到第一段内容出现 |
| 完整回答耗时 | 收到 `done` 所需时间 |
| 取消响应时间 | 点击停止到页面停止更新 |
| 弱网恢复 | 断网、重连、超时后的状态 |
| 重复事件处理 | 旧 request_id 事件是否被丢弃 |
| 错误文案 | 限流、无资料、无权限、模型错误是否可区分 |

这些指标需要移动端埋点和服务端日志用同一个 `request_id` 串起来。否则很难定位问题发生在 App、服务端、检索还是模型网关。

## 13.7 评测接入迭代流程

建议流程：

1. 新增或修改 Prompt、检索、模型路由前，先补评测样本。
2. 本地运行单元测试、`rag_eval.py`、`answer_eval.py`。
3. PR 或 CI 中运行核心评测集。
4. 发布前查看成本、延迟、错误率和答案质量报告。
5. 上线后把用户反馈转成新样本。

不要把所有指标一开始都设为阻断。可以先让单元测试、Schema、隐私脱敏和少量关键评测阻断；其他指标进入报告。等口径稳定后，再把严重回归设为门禁。

## 13.8 读懂失败报告

评测失败时不要只看“分数低”。要定位链路：

| 现象 | 先查什么 |
| --- | --- |
| 没有引用 | 检索是否召回资料，Prompt 是否要求引用 |
| 引用错误 | Top-K 是否混入无关片段，文档元数据是否正确 |
| 答案缺要点 | Prompt 是否强调关键字段，资料是否完整 |
| 出现禁词 | 输出检查和拒答策略是否生效 |
| JSON 格式错 | 结构化输出和 Schema 是否约束足够 |

先分清是检索问题、Prompt 问题、模型问题、工具问题还是移动端状态问题，再决定修改点。

## 本章小结

评测是大模型应用从演示走向工程的关键。单元测试保证确定性代码，检索评测保证资料给对，答案评测保证输出合格，体验指标保证移动端可用。评测集不需要一开始很大，但必须覆盖高频、高风险、易混淆、无资料和权限边界样本，并随着线上反馈持续更新。

## 实践练习

1. 运行 `rag_eval.py --top-k 3`，记录命中率和失败样本。
2. 运行 `answer_eval.py --min-score 0.8`，观察失败时的退出码。
3. 新增一条“资料不足”的评测样本。
4. 为移动端流式输出设计首 Token、取消和错误恢复指标。
5. 把一次真实用户反馈改写成评测样本。
