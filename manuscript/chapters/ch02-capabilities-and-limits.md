# 第 2 章 大模型的核心能力与边界

## 本章导读

移动端开发者学习大模型，不能只停留在“它能聊天、能总结、能写代码”的表层能力上。真正进入工程实践后，开发者会遇到更具体的问题：为什么同一个问题有时回答不一样？为什么模型会编造不存在的接口？为什么上下文太长会变慢、变贵，甚至变差？为什么流式输出能改善体验，但不能改变模型本身的生成方式？

本章用移动端开发者熟悉的工程语言解释大模型的基本运行机制。你不需要先掌握深度学习数学，也不需要推导 Transformer 的公式，但需要理解几个关键事实：模型先把文本切成词元（Token），再把 Token 转成嵌入向量（Embedding）；Transformer 通过注意力机制（Attention）在上下文中寻找相关信息；模型每一步本质上是在预测下一个 Token 的概率分布；温度（Temperature）和核采样（Top-p / nucleus sampling）等参数会影响采样结果；上下文窗口（Context Window）决定模型一次能“看到”的信息范围。

理解这些机制后，很多工程现象就不再神秘：幻觉不是偶发故障，而是概率生成系统的天然风险；提示词（Prompt）不是魔法咒语，而是影响上下文和输出分布的输入协议；检索增强生成（RAG，Retrieval-Augmented Generation）不是让模型“学会”知识库，而是在生成前把可靠资料放进上下文；移动端不能把所有问题都推给模型，而要在客户端、服务端、检索系统和业务系统之间划清边界。

## 学习目标

- 理解大模型从输入文本到输出文本的基本运行链路。
- 理解词元（Token）、嵌入向量（Embedding）、上下文窗口、注意力机制（Attention）和概率生成的直觉含义。
- 知道温度参数为什么会影响输出稳定性。
- 能把大模型能力映射到移动端业务场景。
- 识别幻觉、知识时效性、不确定输出和权限边界带来的工程风险。
- 能运行配套实验，观察 temperature 对采样结果的影响。

## 2.1 先从一个移动端功能说起

假设你要在 App 中实现一个“崩溃日志分析助手”。用户上传崩溃堆栈、设备型号、系统版本和 App 版本后，系统返回：

- 崩溃摘要。
- 可能原因。
- 排查步骤。
- 相关文档引用。
- 是否建议创建工单。

如果不用大模型，这个功能通常需要大量规则：匹配异常类型、解析堆栈、维护关键词表、关联文档、生成说明文案。规则方案可以处理高频、稳定、格式明确的问题，但一旦日志格式复杂、上下文不足或用户描述很口语化，规则就会迅速膨胀。

大模型的价值在于，它能处理非结构化输入，理解自然语言和代码片段之间的关系，并生成更接近人工分析的说明。但它也有明显边界：模型不能天然知道你们团队最新的崩溃归因规则，不能保证每次输出完全一致，也不能直接决定是否关闭线上告警。

因此，移动端大模型功能的合理设计不是“让模型接管 App”，而是让模型承担语言理解、归纳和候选答案生成，再由服务端、检索系统、权限系统和客户端交互共同约束它。

## 2.2 大模型一次生成回答时发生了什么

从工程角度看，一次大模型调用可以简化为 6 个步骤：

1. 应用把用户输入、系统指令、历史消息和检索资料拼成上下文。
2. 模型服务把文本切成 Token。
3. Token 被转换成向量表示，也就是 Embedding。
4. Transformer 网络根据上下文计算每个位置和其他位置的关系。
5. 模型输出“下一个 Token”的概率分布。
6. 采样器根据概率和参数选择一个 Token，再继续预测下一个 Token。

这个过程会重复很多次，直到生成结束符、达到最大输出长度，或被服务端/客户端取消。

这条链路有两个重要结论。

第一，大模型不是一次性“想好整段答案”再返回。它是在一步一步生成 Token。流式输出正是利用了这一点：服务端可以把已经生成的片段尽早发给移动端，移动端再逐步追加到消息气泡中。

第二，模型的输出不是数据库查询结果。数据库查询是在明确数据集合中查找记录，而大模型是在上下文和模型参数共同影响下生成概率上合理的文本。它可能正确，也可能看起来正确但实际错误。

### 推理请求生命周期

在真实服务中，上面的 6 个步骤还可以进一步拆成一条更接近工程实现的时间线。理解这条时间线，有助于解释移动端最关心的几个问题：为什么首字等待时间有时很长？为什么上下文越长越慢？为什么流式输出能改善体验但不能让模型“瞬间完成”？为什么用户点击停止后，服务端还需要显式取消或丢弃后续输出？

第一步是上下文构造。服务端把系统指令、用户问题、历史摘要、检索资料、权限信息和输出格式约束拼成请求。这个阶段还没有进入模型，但它已经决定了成本和质量的上限。如果服务端把整份崩溃日志、整篇接口文档和十几轮历史都塞进去，后续模型阶段一定会更慢、更贵。

第二步是分词。模型服务把文本切成词元（Token）。从这一刻开始，输入长度不再按“字符数”或“行数”计算，而是按 Token 计算。移动端上传的日志、JSON、堆栈和多语言文案都可能比肉眼看到的字符更消耗 Token。

第三步是预填充（prefill）。模型会一次性处理已有上下文，计算每个位置的内部表示。这个阶段通常影响“首 Token 延迟”（time to first token）。如果上下文很长，即使最终只生成一句话，首 Token 也可能很慢。移动端看到的现象就是：用户点击发送后，消息气泡迟迟没有开始出现文字。

第四步是逐步解码（decode）。模型开始一个 Token 一个 Token 地生成输出。每生成一个 Token，模型都会基于已有上下文和刚生成的内容预测下一个 Token。这个阶段影响“输出速度”，也就是每秒能返回多少 Token。流式输出主要优化的是这一阶段的用户感知：模型生成一点，服务端就发一点，客户端就渲染一点。

第五步是缓存。实际推理系统通常会使用键值缓存（KV cache）保存已经计算过的注意力相关中间结果，避免每生成一个 Token 都从头计算全部上下文。你不需要掌握 KV cache 的公式，只需要知道它解释了一个工程事实：同一轮生成中，模型可以复用前面 Token 的计算；但如果请求被取消、上下文变化或会话重新组织，缓存复用就会受限。

第六步是停止。生成会在几种情况下结束：模型输出结束符；达到最大输出长度；服务端超时；用户点击停止；内容安全策略中断；工具调用需要等待外部结果。移动端的“停止生成”按钮不只是 UI 状态变化，它应该触发服务端取消、连接关闭或至少让客户端丢弃后续片段。

因此，移动端体验设计不能只看“模型总耗时”。更合理的指标包括：

| 指标 | 含义 | 移动端体验影响 |
| --- | --- | --- |
| 首 Token 延迟 | 从请求发出到收到第一个输出片段 | 决定用户是否觉得“没反应” |
| 生成速度 | 每秒输出多少 Token | 决定长回答是否顺滑 |
| 总耗时 | 完成整个回答所需时间 | 影响任务完成感 |
| 取消响应时间 | 用户点击停止到 UI 停止更新 | 影响可控感 |
| 错误恢复时间 | 超时、限流、断网后的恢复速度 | 影响弱网体验 |

如图 2-1 所示，上下文窗口是一块共享预算；如图 2-2 所示，模型输出是逐步采样出来的。把这两点结合起来看，就能理解大模型 App 的基本体验规律：先控制输入，再优化首 Token，再用流式输出改善长回答感知，最后用取消、重试和降级保证可控性。

## 2.3 Token：模型处理文本的基本单位

词元（Token）可以近似理解为模型处理文本时使用的文本片段。它不完全等同于中文汉字、英文单词或代码符号。中文句子、英文单词、标点、空格、代码里的驼峰命名，都可能被切成不同粒度的 Token。

例如，下面几类内容都会消耗 Token：

- 用户问题：`为什么登录页一直转圈？`
- 系统指令：`你是一个移动端故障分析助手。`
- 检索资料：接口文档、崩溃排查手册、FAQ。
- 历史对话：用户前几轮追问和模型回答。
- 输出内容：模型最终生成的答案。

对移动端工程来说，Token 直接影响三个指标。

第一是成本。大多数模型服务按输入 Token 和输出 Token 计费。把完整日志、完整文档和长历史都塞进请求，会让成本迅速上升。

第二是延迟。输入越长，模型处理上下文的时间通常越长；输出越长，生成时间也越长。移动端用户对等待更敏感，如果一个按钮点击后 8 秒都没有可见反馈，体验就会明显下降。

第三是稳定性。上下文越长，并不一定越准确。无关内容太多时，模型可能关注错误片段，或者把多个片段混在一起回答。一个常见错误是：为了“让模型知道更多”，把所有可能相关的资料都放进 Prompt。正确做法通常相反：只放当前任务真正需要的资料。

## 2.4 上下文窗口：模型一次能看到的范围

上下文窗口表示模型一次请求最多能处理多少 Token。它包括输入和输出，也包括系统指令、用户问题、历史消息和检索资料。

可以把上下文窗口想象成一次接口请求的“可用内存”。它不是模型的长期记忆，而是本次请求里模型能读取的工作区。请求结束后，模型不会自动记住这次对话；如果下一轮还需要历史信息，应用必须再次把必要历史传给模型，或者把历史摘要、用户偏好和检索结果重新组织进 Prompt。

移动端会话设计常见有 3 种方式：

- 短会话：只保留最近几轮消息，适合简单问答。
- 摘要会话：把较早历史压缩成摘要，适合长对话。
- 检索会话：把历史和资料存入服务端，按当前问题动态召回，适合知识助手。

不要把会话历史长期保存在客户端并原样回传。这样既浪费带宽和 Token，也可能泄露隐私。更合理的做法是：客户端保存会话 ID 和必要 UI 状态，服务端负责历史裁剪、摘要和检索。

## 2.5 Embedding：把文本放进语义空间

嵌入向量（Embedding）是把文本转换为向量的方法。向量可以理解为一组数字，用来表示文本的语义特征。语义相近的文本，在向量空间中的距离通常更近。它的工程意义是：系统可以用“距离”计算语义相似度，从而支持知识库检索、相似问题推荐和上下文召回。

例如：

- “忘记密码怎么办”
- “如何重置登录密码”
- “登录不上，提示密码错误”

这三句话字面不同，但语义接近。关键词搜索可能只命中包含“密码”的文档，而向量检索更容易找到“重置登录凭证”“账号恢复流程”等相关资料。

在本章中，Embedding 的作用是帮助你理解模型如何处理文本表示；在第 7 章和第 8 章中，它会进一步用于向量检索和 RAG。对移动端知识助手来说，Embedding 通常发生在服务端：文档进入服务端后被切分、向量化、写入向量库；用户提问时，服务端再把问题向量化并检索相关片段。

客户端不需要理解向量维度，也不需要保存向量库。客户端关心的是：服务端能否在合理延迟内返回答案，答案是否有引用来源，引用来源是否能在页面上清楚展示。

## 2.6 Attention：模型为什么能利用上下文

Transformer 是现代大模型的核心架构之一。它的关键机制是注意力机制（Attention）。不要把注意力理解成模型真的像人一样在思考，它更像一种计算方法：当模型处理某个位置的 Token 时，会计算这个 Token 和上下文中其他 Token 的相关程度，然后综合这些信息生成新的表示。它的工程意义是：模型可以在较长上下文中关联相隔很远的信息，但前提是这些信息被清楚地放进了上下文。

举一个移动端例子。用户输入：

```text
用户在 iOS 17 上打开扫码页后闪退，日志里出现 AVCaptureSession startRunning。
请判断可能原因。
```

模型要回答这个问题，就需要把多个片段关联起来：

- `iOS 17`
- `扫码页`
- `闪退`
- `AVCaptureSession`
- `startRunning`

这些词相隔较远，但 Attention 机制能让模型在生成“可能原因”时关注到相关 Token。它不会只看最后几个字，而是会根据训练得到的模式，在上下文中寻找可能相关的信息。

这也解释了为什么 Prompt 的组织方式很重要。同样的信息，如果结构混乱、顺序不清、边界不明确，模型更容易关注错误内容。工程化 Prompt 会把角色、任务、上下文、约束和输出格式分开，就是为了帮助模型更稳定地使用上下文。

## 2.7 概率生成：模型如何选择下一个 Token

大模型生成文本时，每一步都会给出一组候选 Token 的分数。经过 Softmax 等处理后，这些分数会变成概率分布。采样器再根据概率选择下一个 Token。

可以用一个简化例子理解。假设模型正在生成一句故障建议：

```text
建议先____
```

候选 Token 的概率可能类似：

| 候选 Token | 概率 |
| --- | ---: |
| 检查 | 0.62 |
| 查看 | 0.18 |
| 重启 | 0.10 |
| 删除 | 0.06 |
| 忽略 | 0.04 |

如果系统总是选择概率最高的 Token，输出会更稳定，但也可能更死板。如果允许按概率随机采样，输出会更丰富，但也更不稳定。

这就是为什么同一个问题可能得到不同回答。模型不是从固定答案库里取一条结果，而是在概率分布中生成文本。

## 2.8 Temperature、Top-p 和输出稳定性

温度（Temperature）通常用于控制概率分布的“尖锐程度”。温度低时，高概率 Token 会更占优势，输出更稳定、更保守；温度高时，概率分布会变平，低概率 Token 更容易被采到，输出更发散。它的工程意义是：同一个 Prompt 可以在“更稳定”和“更多样”之间取舍。

在移动端工程场景中，可以这样选择：

| 场景 | 建议温度 | 原因 |
| --- | ---: | --- |
| 意图分类 | 0-0.2 | 需要稳定分类 |
| JSON 信息抽取 | 0-0.3 | 需要格式稳定 |
| 崩溃日志归因 | 0.2-0.5 | 需要谨慎但允许补充推理 |
| 用户文案润色 | 0.5-0.8 | 需要一定表达多样性 |
| 营销创意生成 | 0.8 以上 | 需要更多变化 |

核采样（Top-p / nucleus sampling）是另一种控制采样范围的方法。它会选取累计概率达到某个阈值的一组候选 Token，再在这个集合中采样。比如 `top_p=0.9` 表示只在累计概率前 90% 的候选中采样，排除概率很低的长尾候选。它的工程意义是：限制模型从过长的低概率尾部候选中采样，降低明显跑偏的概率。

实际项目中，不建议同时大幅调高 temperature 和 top-p。对于移动端产品，稳定体验通常比“每次都很有创意”更重要。尤其是涉及故障分析、知识库问答、权限说明和用户隐私时，应该优先选择低温度、明确格式和可追溯引用。

## 2.9 幻觉：不是偶发 Bug，而是机制风险

幻觉指模型生成看似合理但实际错误的内容。它可能编造不存在的 API、错误引用文档、误解业务规则，甚至给出危险建议。

对移动端开发者来说，常见幻觉包括：

- 编造不存在的 iOS、Android 或 Flutter API。
- 把旧系统版本的行为当作新版本行为。
- 把通用网络错误归因到具体业务接口。
- 在没有足够资料时给出确定结论。
- 把用户上传的截图误解为另一个页面状态。
- 生成看似合法但不可编译的 Swift/Kotlin/Dart 代码。

幻觉的根本原因是：模型是在生成概率上合理的文本，而不是在查询事实数据库。它可以根据训练中见过的模式“补全”答案，但这种补全未必符合你当前项目的真实情况。

因此，工程上不能只要求模型“不要幻觉”。更可靠的方法是降低幻觉空间：

- 用 RAG 提供可追溯资料。
- 在 Prompt 中要求无法确定时明确说明。
- 对结构化输出做 Schema 校验。
- 对代码生成做编译或静态检查。
- 对工具调用做权限和参数校验。
- 对高风险动作加入人工确认。

更适合落地的做法，是把幻觉风险转化为工程门禁：

| 风险类型 | 移动端示例 | 服务端控制 | 客户端提示 | 是否需要人工确认 |
| --- | --- | --- | --- | --- |
| 编造 API | 生成不存在的 Swift/Android 方法 | 代码片段进入编译或静态检查 | 标注“建议代码，需验证” | 视影响范围而定 |
| 事实过期 | 把旧系统版本行为当作新版本行为 | RAG 召回最新文档并显示更新时间 | 展示引用来源和更新时间 | 高风险场景需要 |
| 错误归因 | 把弱网问题判断为服务端故障 | 要求输出置信度和证据片段 | 展示“可能原因”而非“确定原因” | 需要工程师确认 |
| 越权建议 | 建议查看无权限日志或用户数据 | 权限过滤在检索前执行 | 提示“无权限访问相关资料” | 必须 |
| 工具误调用 | 模型建议创建工单或发送通知 | 工具白名单、参数校验、审计日志 | 展示确认页和影响范围 | 必须 |
| 隐私泄露 | 把手机号、设备标识写入回答 | 输入脱敏、输出安全检查 | 隐私字段打码显示 | 必须 |

校验失败时不要把模型结论直接展示给用户。服务端应返回明确错误，例如 `OUTPUT_INVALID`，或者降级为人工复核、重新生成、只展示引用资料而不展示结论。客户端对应的文案也要避免暗示模型已经得出可靠判断。

## 2.10 知识时效性：模型不知道刚发生的事

模型的参数来自训练阶段。训练完成后，模型本身不会自动知道最新系统版本、新接口、新业务规则或团队内部文档。即使某些模型可以联网或使用工具，那也是应用层提供的外部能力，不是模型参数天然具备的能力。

移动端开发中，知识时效性尤其重要：

- iOS 和 Android 系统能力会持续变化。
- 应用商店审核规则可能更新。
- 设备型号和系统版本碎片化严重。
- 团队内部 SDK、埋点规范、接口协议经常调整。
- 线上事故复盘和故障手册会不断更新。

所以，涉及最新事实和团队内部知识时，应该优先通过检索、数据库查询、配置中心或工具调用提供上下文。模型负责理解和表达，事实来源应该来自可追溯系统。

## 2.11 大模型适合哪些移动端任务

大模型适合处理输入复杂、规则难以穷举、输出需要自然语言表达的任务。例如：

- 用户反馈分类与摘要。
- 崩溃日志初步归因。
- 接口文档摘要。
- App 内知识库问答。
- 多语言文案初稿。
- 截图问题分析。
- 测试用例生成。
- 需求说明整理。
- 代码 Review 辅助。

这些任务的共同点是：模型的输出可以作为“建议”“草稿”“候选结果”或“辅助解释”，再由程序校验、用户确认或人工复核。

| 任务 | 模型角色 | 确定性系统角色 | 人工确认 | 客户端提示 |
| --- | --- | --- | --- | --- |
| 用户反馈分类 | 生成分类候选和理由 | 校验枚举值、写入工单系统 | 通常不需要 | 显示可修改分类 |
| 崩溃日志分析 | 总结可能原因和排查步骤 | 脱敏、检索、关联版本信息 | 建议需要 | 标注“初步分析” |
| 文档问答 | 基于引用资料生成答案 | 权限过滤、检索和引用来源 | 通常不需要 | 展示来源片段 |
| 多语言文案 | 生成翻译或润色草稿 | 敏感词和长度校验 | 发布前需要 | 显示可编辑草稿 |
| 测试用例生成 | 生成用例草稿 | 测试平台执行与结果记录 | 需要审核 | 标注未执行状态 |

## 2.12 不适合直接交给模型的任务

以下任务不应直接由模型独立决定：

- 支付、退款、转账等资金动作。
- 删除账号、删除数据、修改权限等高风险操作。
- 自动通过审核、自动封禁用户等影响权益的决策。
- 生成并直接执行 SQL、Shell 或远程命令。
- 在没有验证的情况下修改线上配置。
- 无人工确认地发送正式通知或法律文本。

这并不是说模型完全不能参与这些流程。它可以帮助整理信息、生成草稿、解释原因、准备参数，但最终执行必须由确定性业务逻辑、权限系统和人工确认共同控制。

| 高风险任务 | 模型可以做什么 | 必须由系统控制什么 | 客户端应如何呈现 |
| --- | --- | --- | --- |
| 删除用户数据 | 总结用户请求和影响范围 | 身份验证、权限、删除事务 | 二次确认和不可逆提示 |
| 修改线上配置 | 生成变更说明和检查清单 | 配置校验、灰度、回滚 | 展示审批状态 |
| 发送正式通知 | 起草文案 | 收件人、发送时机、审计 | 预览并要求确认 |
| 自动封禁用户 | 汇总证据 | 风控规则、人工复核 | 显示“待审核”而非直接执行 |
| 执行命令 | 解释命令含义 | 命令白名单和沙箱 | 禁止直接执行未知命令 |

## 2.13 从模型边界反推产品设计

理解模型机制后，移动端产品设计会更清楚。

第一，客户端不直接调用模型供应商。API 密钥（API Key）、Prompt 模板、检索逻辑、审计日志和限流策略都应该放在服务端。客户端只调用自有 API。

第二，长任务必须有状态反馈。因为模型按 Token 生成，移动端应支持“生成中”“停止生成”“重新生成”“复制答案”“查看引用”等交互。

第三，答案要可追溯。知识库问答、故障分析和权限说明都应该展示引用来源，避免用户把模型输出当作不可质疑的事实。

第四，失败路径要明确。模型超时、限流、无资料、格式错误、权限不足，都应该有清晰的客户端提示，而不是统一显示“AI 出错了”。

第五，模型输出要进入校验流程。结构化结果要做 JSON Schema 校验；代码建议要编译或静态检查；工具调用要做权限校验；高风险动作要让用户确认。校验失败时，服务端不能“凑合展示”，而应该返回错误码、触发重新生成或进入人工复核。

下面以“崩溃日志分析助手”为例，把这些原则转成移动端可以直接对接的接口契约。

请求可以设计为：

```json
{
  "request_id": "req_20260621_001",
  "session_id": "s_123",
  "app_version": "6.8.0",
  "platform": "ios",
  "os_version": "17.5",
  "device_model": "iPhone15,3",
  "question": "请分析这段扫码页闪退日志",
  "attachments": [
    {
      "type": "crash_log",
      "object_key": "uploads/crash/req_20260621_001.log"
    }
  ]
}
```

响应可以设计为：

```json
{
  "request_id": "req_20260621_001",
  "status": "done",
  "answer": {
    "summary": "扫码页在启动相机采集会话时闪退。",
    "possible_causes": [
      "相机权限状态未处理",
      "AVCaptureSession 启动时机不在预期线程",
      "系统版本兼容性问题"
    ],
    "next_steps": [
      "检查相机权限授权回调",
      "确认 startRunning 调用线程",
      "对比 iOS 17 设备日志"
    ],
    "confidence": "medium"
  },
  "citations": [
    {
      "title": "扫码页故障排查手册",
      "section": "相机权限与采集会话",
      "snippet": "启动采集会话前必须确认权限状态和页面生命周期。"
    }
  ]
}
```

移动端状态机可以保持简单，但必须覆盖长任务和失败路径：

```text
idle -> submitting -> waiting_first_token -> streaming -> done
                         |                 |
                         v                 v
                      failed           cancelled
```

常见状态含义如下：

| 状态 | 触发条件 | 客户端行为 |
| --- | --- | --- |
| `idle` | 页面初始状态 | 输入框可编辑 |
| `submitting` | 已发送请求，尚未建立响应 | 禁用重复提交，显示加载 |
| `waiting_first_token` | 服务端已接收，等待首个片段 | 显示“正在分析” |
| `streaming` | 收到服务器发送事件（Server-Sent Events，SSE）片段 | 逐步追加到消息气泡 |
| `done` | 收到完成事件 | 展示引用和反馈入口 |
| `cancelled` | 用户点击停止 | 停止追加内容，保留已生成片段 |
| `failed` | 超时、限流、权限不足或格式错误 | 展示可理解错误和重试入口 |

取消生成接口可以设计为：

```http
POST /api/ask/{request_id}/cancel
```

即使底层模型供应商不支持真正取消，服务端也应该停止向客户端转发后续片段，并在日志中标记该请求已取消。这样移动端 UI 和用户感知是一致的。

错误码不应直接暴露模型供应商原始错误。建议转换为产品可理解的错误：

| 错误码 | 含义 | 客户端文案 |
| --- | --- | --- |
| `NO_CONTEXT` | 检索不到可用资料 | 当前资料不足，无法可靠判断 |
| `PERMISSION_DENIED` | 用户无权访问相关资料 | 你没有访问相关资料的权限 |
| `MODEL_TIMEOUT` | 模型响应超时 | 分析超时，请稍后重试 |
| `RATE_LIMITED` | 请求过于频繁 | 请求较多，请稍后再试 |
| `OUTPUT_INVALID` | 模型输出未通过格式校验 | 分析结果格式异常，请重新生成 |

如果使用 SSE 流式输出，可以约定最小事件格式：

```text
data: {"type":"token","content":"扫码页在启动"}

data: {"type":"token","content":"相机会话时闪退。"}

data: {"type":"done","citations":[{"title":"扫码页故障排查手册","section":"相机权限"}]}
```

错误和取消也应有明确事件：

```text
data: {"type":"error","code":"MODEL_TIMEOUT","message":"分析超时，请稍后重试"}

data: {"type":"cancelled","request_id":"req_20260621_001"}
```

客户端收到 `token` 事件时追加文本，收到 `done` 时展示引用来源和反馈入口，收到 `error` 时进入失败态，收到 `cancelled` 时停止追加并保留已生成内容。这样 UI 状态和服务端事件是一一对应的。

## 2.14 可运行实验：观察温度如何影响采样

配套工程提供了一个标准库脚本，用来观察温度（Temperature）如何改变候选 Token 的概率分布和采样结果：

```bash
cd examples/mobile-knowledge-assistant
python3 scripts/sampling_temperature_experiment.py --temperature 0.2 --rounds 50 --seed 7
python3 scripts/sampling_temperature_experiment.py --temperature 1.2 --rounds 50 --seed 7
```

这个脚本不会调用真实模型，而是用一个小型候选词表模拟“下一个 Token”的采样过程。低温度时，“检查”这类高分候选会更频繁出现；高温度时，“修复”“重启”等其他候选出现概率会增加。

在 `seed=7`、`rounds=50` 的条件下，低温度输出大致如下：

```text
temperature=0.2, rounds=50, seed=7
probabilities:
  检查: 0.971
  修复: 0.029
  删除: 0.000
  重启: 0.000
  忽略: 0.000
sampled counts:
  检查: 49
  修复: 1
  删除: 0
  重启: 0
  忽略: 0
```

高温度输出大致如下：

```text
temperature=1.2, rounds=50, seed=7
probabilities:
  检查: 0.561
  修复: 0.313
  删除: 0.064
  重启: 0.046
  忽略: 0.016
sampled counts:
  检查: 33
  修复: 14
  删除: 1
  重启: 2
  忽略: 0
```

可以看到，低温度几乎总是选择最高分候选，高温度则让其他候选获得更多机会。对应到移动端功能：如果任务是“故障原因分类”或“JSON 字段抽取”，应优先使用低温度；如果任务是“生成多个文案备选”，可以适当提高温度。

脚本对应测试位于：

```text
examples/mobile-knowledge-assistant/tests/test_sampling_temperature.py
```

运行全部测试：

```bash
PYTHONPATH=src python3 -m unittest discover -s tests
```

这个实验的价值不在于模拟真实模型的复杂度，而是让读者直观看到：temperature 不是“聪明程度”参数，而是采样分布控制参数。它影响输出稳定性，但不能替代 Prompt 设计、上下文质量和结果校验。

## 2.15 常见问题

**问题 1：大模型是不是越大越好？**

不一定。大模型通常能力更强，但成本、延迟和资源消耗也更高。移动端产品要看任务类型：简单分类、格式转换、短摘要可以用较小模型；复杂推理、多文档归纳和代码分析才需要更强模型。工程上常见做法是根据任务路由模型，而不是所有请求都使用最大模型。

**问题 2：上下文窗口越大，RAG 还重要吗？**

仍然重要。大窗口可以容纳更多内容，但不会自动判断哪些内容可靠、相关、最新。RAG 的价值在于检索、过滤、排序、权限控制和引用来源，而不仅仅是把资料塞进模型。

**问题 3：低温度能不能消除幻觉？**

不能。低温度可以让输出更稳定，但如果上下文缺少事实、Prompt 约束不清或模型本身不了解业务，稳定输出也可能稳定地错误。消除幻觉风险需要检索、校验和人工确认。

**问题 4：模型能不能作为 App 的规则引擎？**

不建议。规则引擎需要确定性、可解释、可审计。模型适合处理模糊输入和生成候选结果，不适合替代支付、权限、审核等核心规则。

**问题 5：移动端是否需要理解 Transformer 细节？**

不需要深入到训练和公式层面，但需要理解 Attention、上下文窗口和概率生成的直觉含义。这样才能解释延迟、成本、幻觉、流式输出和 Prompt 效果，而不是把所有问题都归因于“模型不够好”。

## 本章小结

大模型的核心能力来自对 Token 序列的建模。它把文本切成 Token，把 Token 转成向量，通过 Transformer 和 Attention 使用上下文，并按概率一步一步生成下一个 Token。Temperature、Top-p 等参数会影响采样稳定性；上下文窗口决定一次请求能处理的信息范围；Embedding 让语义检索成为可能。

对移动端开发者来说，理解这些机制的目的不是训练模型，而是设计可靠的大模型功能。模型可以帮助理解、归纳和生成，但事实来源要可追溯，业务动作要可控，输出结果要校验，密钥和权限要留在服务端。

本章最重要的工程结论是：大模型是能力组件，不是业务系统本身。只有把模型、服务端、检索系统、权限系统和移动端交互组合起来，才能构建可靠的大模型应用。
