# 第 7 章 Embedding 与向量检索

## 本章导读

Embedding 把文本转换成向量，使系统可以按语义相似度检索资料。对移动端开发者来说，它的价值不是理解每个维度的数学含义，而是知道如何把产品文档、故障手册、隐私规范和用户问题放进同一套检索链路，为 RAG 提供可靠上下文。

配套脚本：`examples/mobile-knowledge-assistant/scripts/tfidf_vector_search.py`

## 学习目标

- 理解 Embedding 和向量检索解决什么问题。
- 知道相似度、文档切分、元数据和 Top-K 的作用。
- 区分本地 TF-IDF 示例和生产 Embedding 检索。
- 运行本地检索脚本，观察查询、命中片段和分数。
- 能把检索结果映射到移动端引用卡片和反馈入口。

## 7.1 为什么要关心向量检索

移动端知识助手常见问题并不总是精确匹配文档关键词。例如用户问“登录一直转圈怎么办”，文档里可能写的是“OAuth 回调超时”“弱网重试”“Token 刷新失败”。关键词搜索可能漏掉这些相关内容，向量检索更适合处理字面不同但语义接近的问题。

Embedding 的作用是把文本放进语义空间。语义相近的文本距离更近，系统可以用距离排序找出最相关片段。它常用于知识库问答、相似问题推荐、历史工单召回、上下文补全和 RAG。

## 7.2 Embedding 的直觉

可以把 Embedding 理解为“文本的数字坐标”。同一类问题在空间里更接近，不相关内容更远。它不等于分类标签，也不等于关键词索引，而是一个可计算相似度的表示。

移动端知识库场景中，Embedding 通常发生在服务端：

1. 文档进入服务端。
2. 服务端切分片段。
3. 对片段生成向量。
4. 保存向量、正文、标题、章节、版本和权限标签。
5. 用户提问时，服务端对问题生成向量。
6. 向量库返回相似片段。
7. 服务端把片段放入 Prompt。

客户端不需要保存向量，也不需要理解向量维度。客户端关心的是答案是否有引用、引用是否清楚、检索失败时页面如何提示。

## 7.3 相似度和 Top-K

向量检索常用余弦相似度衡量方向相近程度。示例工程为了避免依赖外部服务，使用 TF-IDF 向量和余弦相似度模拟检索流程。

Top-K 表示返回前 K 个候选片段。K 太小可能漏召回，K 太大可能把无关资料塞进 Prompt，增加成本并干扰模型。移动端知识问答常见起点是 Top-3 到 Top-5，之后用评测集调整。

| 参数 | 过小风险 | 过大风险 |
| --- | --- | --- |
| Chunk 长度 | 信息不完整 | 片段太长、成本高 |
| Top-K | 正确资料漏召回 | 无关资料干扰 |
| 相似度阈值 | 可回答问题被拒绝 | 低相关资料进入 Prompt |

## 7.4 文档切分和元数据

切分决定模型看到的资料边界。移动端文档通常包含接口说明、权限规则、错误码、崩溃分析和版本差异。推荐优先按标题、段落和语义边界切分，而不是固定按字符数粗暴截断。

每个片段至少要保存：

| 字段 | 用途 |
| --- | --- |
| `doc_id` | 稳定定位文档 |
| `title` | 引用卡片标题 |
| `section` | 展示章节和定位 |
| `text` | 放入 Prompt 和引用展开 |
| `updated_at` | 判断资料时效 |
| `visibility` | 权限过滤 |
| `source_url` | 跳转原文 |

权限元数据必须在检索前生效。不能先把所有资料检索出来，再让模型判断用户能不能看。

## 7.5 本地 TF-IDF 检索

运行脚本：

```bash
cd examples/mobile-knowledge-assistant
python3 scripts/tfidf_vector_search.py \
  --query '移动端为什么不能保存模型 API Key' \
  --top-k 3
```

脚本会读取 `data/documents/`，切分文本，构建 TF-IDF 向量，计算相似度并输出命中片段。它不等同于生产 Embedding，但能说明检索链路：输入问题、向量化、相似度排序、返回引用。

读输出时重点看：

- 命中文档是否正确。
- 章节是否对应问题。
- 分数是否明显区分相关和不相关片段。
- 片段文本是否足够回答问题。

如果正确文档没有召回，先检查切分、关键词、同义词和元数据，不要急着改 Prompt。

## 7.6 替换为生产 Embedding

从本地脚本迁移到生产系统时，通常要补齐：

| 方向 | 要求 |
| --- | --- |
| Embedding 服务 | 选择模型、维度、批量生成、失败重试 |
| 向量库 | 写入、查询、过滤、删除、重建索引 |
| 混合检索 | 结合关键词和向量召回 |
| 重排 | 对候选片段再排序，提高前几条质量 |
| 元数据过滤 | 租户、角色、文档可见性和版本 |
| 更新机制 | 文档变更后的增量索引和失效处理 |
| 评测 | 召回率、MRR、Top-K 命中、人工复核 |

替换顺序建议：先保留接口结构，替换检索器；再加入权限过滤；然后加入评测集；最后优化模型、混合检索和重排。

## 7.7 移动端展示检索信号

移动端不需要展示向量距离，但需要展示可理解的引用来源。一个引用卡片可以包含标题、章节、更新时间、摘要和跳转入口。内部开发者工具可以额外显示分数、片段 ID、文档版本和权限标签。

用户反馈也应回流到检索优化。常见反馈包括“引用错误”“资料过期”“没有回答问题”“答案有用”。这些反馈可以变成评测样本，帮助调整切分、Top-K、重排和 Prompt。

## 本章小结

Embedding 与向量检索是 RAG 的基础。它们不负责生成答案，而负责把正确资料送到模型面前。移动端工程师需要理解切分、元数据、权限过滤、Top-K、引用展示和反馈闭环，而不是只关心向量库选型。能跑通本章 TF-IDF 示例后，再替换生产 Embedding 会更稳。

## 实践练习

2. 调整 `top_k`，观察返回片段数量和相关性变化。
3. 为文档片段设计 `updated_at`、`visibility`、`source_url` 字段。
4. 设计移动端引用卡片，包含标题、章节、摘要和反馈入口。
5. 写出从本地 TF-IDF 替换为生产向量库的步骤。
