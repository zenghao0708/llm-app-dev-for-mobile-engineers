# 写作与出版风格规范

## 读者画像

本书面向具有移动端开发基础的工程师。读者通常熟悉 iOS、Android、Flutter 或 React Native 中至少一种技术栈，理解 HTTP、JSON、异步请求、权限、缓存、日志和基础后端服务概念，但不要求具有机器学习或深度学习背景。

## 写作风格

1. 先解释移动端工程问题，再引入模型概念。
2. 每个术语第一次出现时给出中文解释、英文名称和工程意义。
3. 不把数学推导作为主线，只在必要处说明直觉。
4. 避免堆砌厂商名词，强调稳定的技术模式。
5. 代码以 Python 服务端示例为主，移动端用接口契约和关键调用片段说明，不在客户端硬编码真实密钥。
6. 每章必须包含“本章导读”“学习目标”“核心内容”“动手实践”“本章小结”“实践练习”。

## 篇幅控制

- 全书正文目标约 10 万字，合理浮动区间为 9 万-11 万字。
- 每章目标字数以 `publication-length-plan.md` 为准。
- 扩写时优先增加原理解释、移动端工程场景、可运行代码讲解和图注，不用泛泛背景材料凑字数。
- 单章不宜低于 5,000 字；核心章节如 RAG、API 调用、综合项目可达到 7,000-8,000 字。
- 每章至少包含 3-5 个能帮助移动端开发者落地的具体例子。

## 代码风格

- Python 版本建议使用 3.11 或更高版本。
- 示例优先使用标准库，再逐步引入 Web API、移动端调用和流式响应处理。
- API Key 使用 `LLM_API_KEY` 环境变量。
- 模型服务地址使用 `LLM_API_URL` 环境变量。
- 代码片段要短，完整项目放配套资源。移动端示例重点展示调用方式、状态管理、取消请求、重试、权限和 UI 渲染。
- 书中出现的核心示例必须能在配套工程中找到对应实现，不能只停留在不可运行的伪代码。
- 配套工程必须提供默认可运行的 mock provider，读者不申请模型 Key 也能完成启动、请求和测试。
- 配套工程必须包含测试命令；修改 Prompt、检索、服务端接口或流式响应后，应重新运行测试。
- 注释用于解释设计意图、工程边界和移动端注意事项，例如“为什么 API Key 只放服务端”“为什么 SSE 需要逐块 flush”。不要添加逐行复述代码的低价值注释。

## 图片规范

- 原理图优先采用 SVG 矢量图，便于印刷和高清缩放。
- 截图统一使用浅色主题，字号不低于 14 px。移动端截图同时准备 iOS 与 Android 风格示例，避免出现真实用户隐私数据。
- 每张图都要有图题、图注和正文引用。
- 不使用带真实密钥、真实用户数据、内部域名的截图。
- 对复杂流程图采用“少颜色、强对齐、清晰箭头”的出版风格。

## 重点高亮

- 使用 Markdown 引用块呈现关键判断，例如 `> **重点提示**：...`。
- 每章最多放 1-3 个高亮块，优先标出工程边界、上线门槛、安全风险和读者容易误解的结论。
- 高亮块应是可执行判断，不写泛泛口号；不要把整段代码或大表格放进高亮块。
